## 2. Deployment-readiness check

The Founder explicitly directed an audit for the "project cannot be deployed" class of blocker that hit `openclown-assistant` TKT-016. The audit confirms that **`developer-assistant` is at material risk of the same class of blocker**, but for a slightly different reason: openclown-assistant hit it because the boot path was under-specified; `developer-assistant` is at risk because its self-deployment story is *unspecified*.

**Specifically, `developer-assistant` has:**
- A clear contract for how *generated* projects are deployed (`docs/architecture/GENERATED-PROJECT-DEPLOYMENT-CONTRACT.md` v0.1.0).
- A clear contract for how the Hermes runtime relates to repository governance state (`docs/architecture/HERMES-RUNTIME-CONTRACT.md` v0.2.0).
- A gated live-smoke readiness harness for credentials + connectivity (TKT-017 `src/developer_assistant/smoke_readiness.py`).

**`developer-assistant` does NOT have:**
- A Dockerfile, `docker-compose.yml`, `Procfile`, `fly.toml`, `systemd` unit, or `Makefile` `deploy-self` / `install-self` target for the assistant ITSELF.
- A `SELF-DEPLOY.env.example` or equivalent env-var contract for the assistant's own deployment (Telegram bot token, GitHub PATs for project repos, OmniRoute API key, VPS-local paths, operational state store path).
- An ADR recording the choice between Docker Compose / systemd / bare-metal install for the assistant's own runtime.
- An architecture section describing how a fresh Ubuntu VPS goes from zero to a Telegram-reachable `developer-assistant` install.
- A health check command for the deployed assistant.
- A rollback / upgrade procedure for the deployed assistant.

ARCH-001 v0.2.0 § 2 v0.1 Scope item 7 explicitly says "A documented one-command VPS deployment contract for *generated* projects, **without automatic live deployment in v0.1**" — i.e., the architecture has consciously deferred the assistant's own deployment story. ARCH-001 § 3 Deferred Scope item "Fully autonomous production deployment to a founder VPS" reinforces this. The current `docs/orchestration/SESSION-STATE.md` § Next Recommended Action says the next step is "re-run a fresh `TKT-011@0.2.0` Ticket Orchestrator session with `TKT-018@0.1.0` as the selected trial vehicle" — but TKT-011 § 1 Scope requires the trial to begin "from an authenticated Telegram founder interaction through the TKT-015 Hermes Telegram transport boundary" which implicitly requires a deployed Hermes runtime. This is a sequencing trap.

**Architect-pass directive filed.** `docs/backlog/TKT-NEW-self-deployment-architect-pass.md` records the gap, the why-it-matters-now analysis, the proposed scope of the Architect pass (PRD revision → architecture section / new contract / ADR → at least three implementation tickets → Reviewer LLM RV-SPEC), the sequencing constraint (Architect pass MUST precede TKT-011), and the acceptance criteria for the eventual implementation tickets.
