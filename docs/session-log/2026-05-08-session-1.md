# Incident Record — 2026-05-08 Session 1

## 0. What this file is

This is **not** a routine cold/warm SO handoff. It is an **incident record** filed by the Strategic Orchestrator after the Founder ran the first live deployment of the system on a real VPS and the live behaviour did not match what the merged contracts mandate. The incident exposed an integration-verification gap that no prior ticket has closed.

This file captures: (1) what was observed live, (2) cross-reference to which contract/TKT each observation violates, (3) root cause, (4) the decision taken, and (5) the scope-stub the Architect must turn into a four-ticket family.

## 1. Trigger

The Founder deployed `developer-assistant` onto the production VPS using TKT-020 install scripts and started testing the deployed system through Telegram. The deployed Hermes runtime was nicknamed "CLAW" by the Founder for the duration of the test. Founder forwarded a complete Telegram log of the test session to the SO.

The test target was a separate forked project (`agents-office` at `https://github.com/errydayiwakingup/agents-office`) — used as a sample workload to drive the new runtime end-to-end. The runtime under test was the deployed `developer-assistant` Orchestrator runtime, **not** any specialist runtime.

Founder's verdict after the test: «продукт просто не рабочий ... нужно разобраться, где мы неправильно разрабатываем проект, почему он неработоспособен. и пофиксить это».

## 3. Root cause

The product is not "broken-by-design". The merged specifications are correct. What broke is the **integration verification gate**.

### 3.1 `runtime_check.check_runtime()` is not enforced at boot

`docs/tickets/TKT-021.md` § 1 specifies a `runtime_check.check_runtime(role, config_path, operational_db_path, env)` function that, on each runtime startup, must verify (a) `HERMES_DEVASSIST_ROLE` value, (b) loaded skills match the per-role expected set, (c) `operational.db` symlink correctness, (d) schema version match, (e) Telegram-token presence on Orchestrator only and Telegram-gateway absence elsewhere.

If this check were invoked blockingly from the systemd unit's `ExecStartPre=` and a startup failure caused the runtime to exit non-zero, the deployed runtime could not have reached steady state with `delegate_task` enabled, with `skill_manage` enabled, or with `hermes-agent` loaded, because each of those mismatches would have been caught by clause (b).

The fact that the deployed runtime steadily ran with all three mismatches is direct evidence that:

- either the systemd unit template (TKT-020 `etc/systemd/devassist@<role>.service.tmpl`) does not invoke `runtime_check.check_runtime()` in `ExecStartPre=`, or
- it does invoke it, but the function is not actually wired to abort on mismatch, or
- it aborts but the unit is configured with `RestartPolicy` that masks the abort.

This must be diagnosed and fixed; the diagnosis is part of the new ticket family in § 5.

### 3.2 `TKT-032` acceptance criteria are process-aliveness only

`docs/tickets/TKT-032.md` § 4 currently asserts: install-script exits zero, `verify-self.sh` exits zero, five units report `running` or `degraded`, one health endpoint returns valid JSON, no credential leaks, docs validation passes, unit tests pass.

This is sufficient to prove the system has booted. It is **not** sufficient to prove the system has booted *as composed*. None of the criteria checks:

- that the Orchestrator's loaded skill set is exactly the per-role allowlist (no extra `hermes-agent`, no missing `dev-assist-classifier`);
- that an attempted `delegate_task` invocation actually fails;
- that the runtime's `agent.system_prompt_path` resolves to `docs/prompts/<role>.md`;
- that a synthetic Telegram message round-trips through `dev-assist-classifier` and produces a row in `work_items`;
- that a specialist runtime claims that row and produces a result;
- that install-script provided `gh` and a committable git identity for the `devassist` user, and that no PAT is embedded in `remote.origin.url`.

Each of these is a behaviour-level invariant, not a process-level one. TKT-032 in its current form was always going to "pass" against a misconfigured runtime. That is the gap that the live test exposed.

### 3.3 TKT-011 (live trial) cannot rescue this

TKT-011 was specced as the integration trial that would catch behaviour-level mismatches. But TKT-011 is blocked on TKT-032, and TKT-032's pass condition does not require behaviour-level correctness. So the gate sequence as currently merged would have approved a non-functional deployment.

## 4. Decision

The Founder selected the strategic option (verbatim): «делаем правильно. так, чтобы после переустановки проекта он завелся так, как мы его разрабатывали, а не на дефолтном гермесе». This corresponds to the option recorded as "Variant A" in the SO's diagnostic message of 2026-05-08.

The decision:

1. **Halt TKT-032 in its current shape.** PR #118 is left open but the iter-1 NUDGE for the deployed Executor on the VPS is rescinded. TKT-032 is moved to status `blocked` in `SESSION-STATE.md` with a pointer to this incident record. The current AC of TKT-032 will eventually be superseded or absorbed by AUDIT-003 (see § 5).
2. **The Ticket Orchestrator session currently driving TKT-032 on the Founder's Windows PC is paused.** No further iter NUDGEs are issued for TKT-032 until the AUDIT family lands.
3. **A four-ticket Architect family is dispatched** with the scope stubs in § 5 below. The Architect runs in opencode on the Founder's Windows PC (since the on-VPS Architect runtime is itself part of what is being audited and cannot be trusted for spec work).
4. **No code on `main` is reverted.** The diagnosis is a verification gap, not a design defect. Specs in `main` are kept as-is; AUDIT-001..004 will add the missing enforcement layers, behaviour-level smoke, and operator-install fixes around them.
5. **No code is touched by the SO directly.** This file plus the `SESSION-STATE.md` update are the only artefacts the SO writes for this incident; the four ticket bodies are written by the Architect.

## 5. Scope-stub for the Architect ticket family

The four tickets below are scope stubs only. Section identifiers (1. Scope, 2. Non-scope, 3. Required Context, ..., 10. Execution Log) and full acceptance criteria are written by the Architect when these tickets land in `docs/tickets/`. The SO produces the scope intent only; it does not own ticket bodies.

Numbering is left to Architect discretion; the indicative names below are used in this incident record for cross-reference but the Architect may rename. The expected sequence number range is in the `TKT-NEW-INT-AUDIT-*` family or the next free `TKT-033..036` slot, whichever the Architect prefers per project convention.

### 5.1 AUDIT-001 — runtime_check enforcement at systemd boot

**Intent.** Make the runtime startup invariants in `docs/tickets/TKT-021.md` § 1 actually block boot. Diagnose why they currently do not block (most likely the `etc/systemd/devassist@<role>.service.tmpl` does not invoke `runtime_check.check_runtime()` from `ExecStartPre=`, or invokes it without abort-on-failure semantics). Fix the unit template, add a regression test, and extend `runtime_check` to additionally verify: (i) attempted `delegate_task` invocation must fail at runtime, (ii) `skill_manage` must be unreachable, (iii) `agent.system_prompt_path` resolves to `docs/prompts/<role>.md` and the file content matches the role's prompt SHA at install time.

**Out-of-scope.** Behaviour-level Telegram smoke (that is AUDIT-003). New ticket bodies. Modifying any role prompt body.

**Owner.** Architect spec, Executor implementation, Reviewer for both spec and code.

### 5.2 AUDIT-002 — install-script operator hygiene

**Intent.** Update `scripts/install-self.sh` (and `scripts/verify-self.sh`) so that a freshly bootstrapped VPS reaches a state where: (i) `gh` CLI is installed and authenticated against the runtime PAT environment variable, never via embedded tokens; (ii) git `user.name` and `user.email` are configured for the `devassist` system user; (iii) the runtime's `git remote get-url origin` returns a token-free HTTPS URL (PAT comes from the credential helper or env, not from the URL); (iv) `/srv/devassist/shared-skills/` is populated with all custom `dev-assist-*` skills at the pinned git commit declared in `MULTI-HERMES-CONTRACT.md` § 5.0, and the verify step asserts directory contents match the manifest.

**Out-of-scope.** Behaviour-level smoke. Touching the `delegate_task` enforcement (AUDIT-001's responsibility). New role prompts.

**Owner.** Architect spec, Executor implementation, Reviewer.

### 5.3 AUDIT-003 — behaviour-level smoke (replaces TKT-032 AC)

**Intent.** Replace the process-aliveness AC of `docs/tickets/TKT-032.md` § 4 with a behaviour-level smoke. Synthetic Telegram message → `dev-assist-classifier` produces a class label and persists it → Orchestrator writes a `work_items` row → a specialist runtime (Planner is the simplest target) claims the row within N seconds → produces a result that round-trips back to the Orchestrator → the run is observable in `errors`/`llm_calls` tables and in `/health` JSON. The smoke must be runnable without involving the Founder's actual Telegram account (use a scripted gateway client or a mock-injection path that is itself reviewed and gated).

**Out-of-scope.** TKT-011 full live trial (TKT-011 is rescoped in AUDIT-004). Founder-merge-policy changes. Adding new specialists.

**Owner.** Architect spec, Executor implementation, Reviewer. The Architect must explicitly decide whether AUDIT-003 supersedes TKT-032 entirely or co-exists alongside it; if the latter, TKT-032 must be reformulated as a process-aliveness pre-flight only and AUDIT-003 becomes the actual gate to TKT-011.

### 5.4 AUDIT-004 — TKT-011 reformulation

**Intent.** Rewrite `docs/tickets/TKT-011.md` so its dispatch precondition is "AUDIT-001 + AUDIT-002 + AUDIT-003 merged", not the current "TKT-020..030 + TKT-031 + TKT-032 merged". Update its AC so it is the *full pipeline integration trial* (PRD intake → PRD draft → architecture → tickets → execution → review → merge), with each pipeline edge asserted to flow through `work_items` rather than through any direct LLM-to-LLM channel. The Architect must also decide whether the historical iter-1/iter-2/iter-3 trial-vehicle records on TKT-011 are preserved as-is or amended.

**Out-of-scope.** Re-scoping merged tickets other than TKT-011 itself. Touching review-policy AC.

**Owner.** Architect spec, Executor implementation (limited; this ticket is mostly spec), Reviewer.

## 7. Multi-agent collaboration health (Founder-requested technical update)

In the Telegram log the Founder explicitly asked the deployed runtime to report on multi-agent collaboration quality. The runtime (CLAW) repeatedly reported "subagents responded usefully and synthesis was viable", but the same log later confirmed the runtime had **not** invoked any of the four specialist `developer-assistant` runtimes during the test. What CLAW described as "subagents" were generic Hermes `delegate_task` spawns — a tool that is BLOCKED in our composition (see § 2 row 1).

The honest read: **multi-agent collaboration in our composition was never exercised during the test.** No specialist claimed a `work_items` row; the queue was empty throughout. AUDIT-003 is what will actually exercise the collaboration path for the first time.

Until AUDIT-003 lands, no claim about multi-agent collaboration health on this deployment is verifiable.
