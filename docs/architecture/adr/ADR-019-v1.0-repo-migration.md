# ADR-019: Repo migration at v1.0 milestone for final PII residue elimination

## Context

The 2026-05-10 / 2026-05-11 F1-closure dual-phase privacy scrub
(Phase 1 = PR #165 file-content scrub; Phase 2 = git-history rewrite +
force-push) successfully eliminated personal-information (PII) from:

- All file contents on `main` (Phase 1, AC-1 verified).
- All current PR descriptions and issue/review comments (Phase 1, GitHub-side).
- All git commit metadata (author/committer fields, commit message bodies)
  across all 230+ commits and all branches (Phase 2, AC-2-1 through AC-2-5).
- All 706 GitHub Actions workflow runs and their `head_commit.message`
  snapshots (Phase 1.5 cleanup, 2026-05-11).
- The PR #165 audit-trail comment itself (re-edited to use pattern
  templates instead of verbatim PII, 2026-05-11).

After all of the above is applied, the following **residual PII sources**
remain accessible to anyone with read access to the public GitHub repository
and **cannot be eliminated through self-service GitHub APIs**:

1. **GitHub edit-history (F-CARRY-2 carry-over from Phase 1):**
   GitHub permanently stores prior versions of every edited PR body and
   issue comment, accessible via the "edited" link in the UI. Phase 1 edited
   11 PR bodies and 2 issue comments to remove PII; their pre-edit versions
   remain visible. Phase 1's own audit-trail comment was further edited
   2026-05-11 to remove verbatim PII; its pre-edit version is now part of
   this set as well. GitHub provides no public API to delete or scrub
   edit-history.
2. **Search engine caches (Google, Bing) and third-party archives
   (Wayback Machine, archive.today):** Any externally indexed snapshot of
   pre-scrub repository pages may still display PII until manually
   delisted, which requires per-URL privacy removal requests to each
   provider.
3. **GitHub's internal data warehouse:** All deleted artifacts (workflow
   runs, edited bodies) may persist in GitHub's internal logs/backups
   beyond user-facing visibility. Self-service removal is not possible;
   only a GitHub Support privacy ticket can request such scrub, and
   compliance is not guaranteed.

These three sources together constitute the "F-CARRY-2 + adjacent" residue.
Eliminating them fully requires either (a) a GitHub Support privacy ticket
(asynchronous, not guaranteed), or (b) **repo migration** — creating a new
repository at the same URL, pushing only the rewritten history into it, and
archiving the old repository under a renamed private slot. Repo migration
discards all GitHub-side metadata (PR threads, issue comments, workflow
runs, edit history, contributor graphs, stars/watchers) and provides a
guaranteed clean slate.

## Decision

We will perform a **repo migration at the v1.0 deployment milestone** —
the project's first successful production deployment — to eliminate all
residual PII from the public-facing repository surface. Migration is
deferred until v1.0 (rather than executed now) because:

1. **Disruption cost:** Migration requires re-configuring every GitHub
   integration (PR-Agent / Qodo, branch protection, repository secrets,
   Actions variables, deploy keys, webhooks), re-pinning issue and PR
   references in active development sessions, and notifying every local
   clone to re-clone fresh. During active pre-v1.0 development, this
   disruption is not justified given the residue level (≤ 3 carry-over
   classes, none verbatim in current-state artifacts).
2. **Preservation of in-flight forensic audit trail:** PR threads, review
   comments, and ticket cross-references provide forensic value during
   active development for debugging and root-causing. At v1.0, this
   provenance is fully captured in `docs/` (ticket § 10 Execution Log
   entries + ADR + session-log snapshots) and no longer needs to be
   public.
3. **Single-shot ROI:** Migrating once at v1.0 is more efficient than
   incremental partial migrations or repeated edit-history scrubs.

## Migration plan (executed at v1.0 milestone — informational, not yet
performed)

**Approach 2 — same URL preservation (recommended):**

1. **Phase 2-equivalent re-rewrite locally** of the (then-current) main
   tip, applying `git filter-repo --mailmap` + message-callback identical
   to 2026-05-11 Phase 2.
2. **Rename** `OpenClown-bot/developer-assistant` →
   `OpenClown-bot/developer-assistant-archive-<YYYY-MM-DD>` via GitHub
   API (`PATCH /repos/.../developer-assistant` with new name).
3. **Set archive private** via GitHub API (`PATCH /repos/.../...` with
   `"private": true`). Archive remains accessible only to repository
   admins for forensic preservation.
4. **Create new empty** `OpenClown-bot/developer-assistant` at the
   original URL. This preserves all external references to
   `github.com/OpenClown-bot/developer-assistant/...`.
5. **Push** the rewritten history (`git push origin --all && git push origin --tags`)
   into the new empty repo.
6. **Re-configure integrations:**
   - Re-install PR-Agent (Qodo) GitHub App via GitHub App settings.
   - Re-set repository secret `OMNIROUTE_API_KEY` (and any other secrets).
   - Re-create branch protection rules on `main` per project policy.
   - Re-create webhook endpoints (if any).
   - Re-pin issue / PR (if any).
7. **Notify clones.** Post an issue on the new repo and message any
   active contributors / Devin sessions instructing them to delete local
   clones and `git clone` fresh.
8. **(Optional, asynchronous):** Submit privacy removal requests to
   Google Search Console for any indexed snapshot URLs that previously
   carried PII; email `info@archive.org` for Wayback Machine snapshots if
   discovered.

**Approach 1 alternative (different URL):** Push to a new repo with a
distinct name and abandon the old URL. Rejected because external
documentation, Hermes Agent integration links, and any chat-log references
to old PR URLs would break, and the canonical URL would shift.

**Approach 3 alternative (transfer ownership):** Transfer the current
repository to a personal account, set it private, then create the new
empty repo at the org. Functionally equivalent to Approach 2 but requires
GitHub's repo-transfer flow which has additional rate-limit and quota
considerations. Approach 2 (rename) is simpler and recommended.

## Trigger and authority

Migration is triggered by:

- **Trigger event:** First successful deployment of the project to a
  production-grade environment, ratified by the Strategic Orchestrator as
  the v1.0 milestone closure. Concretely, this corresponds to the
  AUDIT/TKT cycle that closes the v1.0 readiness ticket family.
- **Authority:** Strategic Orchestrator dispatch under cross-zone authority
  (same precedent as F1 closure NUDGE 2026-05-10). The migration itself
  is a meta-operation, not a TKT cycle, and does NOT require a Reviewer
  artifact gate prior to execution. A retrospective Reviewer audit (RV-ARCH
  prefix per CONTRIBUTING.md § Review Gates § naming convention) is
  recommended for the post-migration cleanup PR that re-introduces baseline
  documentation/config on the new repository.

## Consequences

**Positive:**

- All residual PII (F-CARRY-2 edit-history, deleted workflow_runs cache,
  any GitHub-internal logs reachable via search/archive snapshots) is
  removed from public visibility in a single operation.
- Repository starts v1.0 with a clean GitHub-side surface (zero PR
  threads, zero comments, zero workflow run history).
- External URL `github.com/OpenClown-bot/developer-assistant` continues
  to resolve and serve the live (clean) repository.

**Negative:**

- Star/watcher counts reset to 0.
- Contributors graph resets (will show only `OpenClown-bot` as author).
- All historical PR thread context, code review conversations, and PR-Agent
  reviews are no longer publicly accessible (preserved in private archive).
- Integration re-configuration is required (~15-30 minutes of manual
  setup post-migration).
- Local clones break and require re-cloning.

**Neutral:**

- Git history (commit messages, file contents, tree SHAs) is preserved
  byte-identically with the post-Phase-2 state.
- All `docs/` content (ticket § 10 Execution Log entries, ADR, RV-*
  artifacts, session-log snapshots) is preserved verbatim and provides
  the forensic audit trail that PR thread context previously provided.

## Alternatives considered and rejected

1. **Repeated edit-history scrub via GitHub Support tickets** — Rejected:
   not self-service, not guaranteed to succeed, and would require one
   ticket per artifact (13+ artifacts).
2. **Make the entire repository private** — Rejected: project is intended
   to be open-source post-v1.0, and going private during pre-v1.0
   development conflicts with external visibility goals (Hermes Agent /
   OpenClaw integration documentation references this repo).
3. **Delete + recreate every edited comment** — Rejected: only addresses
   2 of 13 F-CARRY-2 artifacts (the comments; PR bodies cannot be
   deleted, only edited), and the destroy/recreate breaks PR thread
   structure and comment IDs.
4. **No-action / accept the residue permanently** — Rejected at user
   discretion: project's privacy posture for v1.0 launch warrants a
   single-shot clean slate.
