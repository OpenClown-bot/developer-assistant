# ADR-008: Escalation Classifier — deterministic rules + structured concept anchor

## Status

Draft, pending Founder approval. Supersedes none. Related: ADR-002 (repository state), ADR-006 (IPC), ADR-009 (model assignment).

v0.1.1 (RV-SPEC-012 F3 fix): the classifier no longer delegates the escalate / proceed decision to an opaque LLM call. The decision path is now fully deterministic, anchored on a structured project-concept artifact (`PROJECT-CONCEPT.md`). The LLM classifier survives only as an optional advisory-narrative generator on the escalation surface (`ESCALATION-POLICY.md` v0.1.1 § 5.5), out of the decision path.

## Context

`PRD-001.md` v0.2.1 § 13.1 defines the escalation rule:

> The assistant escalates to the Founder only when a candidate decision either (a) strongly deviates from the original concept captured at intake, or (b) risks breaking already-committed scope or operational state.

The PRD intentionally does not commit to a classifier mechanism; the Architect must produce one. `ESCALATION-POLICY.md` operationalizes the mechanism. This ADR records why the chosen mechanism is **deterministic rules first, LLM classifier second**, and why the alternatives were rejected.

## Decision

Operationalize the escalation rule with two layers, **both deterministic**:

1. **Deterministic rule set** (`ESCALATION-POLICY.md` v0.1.1 § 4): a curated set of patterns covering force-push, hard reset, file deletion under governance directories, schema-destructive SQL, credential rotation, public endpoint exposure, paid third-party introduction, write-zone violations, PRD/ADR status changes, and similar concrete risks. Match → escalate, no further checks.
2. **Deterministic concept-deviation classifier** (`ESCALATION-POLICY.md` v0.1.1 § 5): when no § 4 rule matches and the action is not read-only or within-catalog, the classifier compares the candidate action against a structured project-concept anchor (`PROJECT-CONCEPT.md` § 2) using a fully-specified pure-function predicate set (`PROJECT-CONCEPT.md` § 3). Match on a `deviation_rule` with `verdict: ESCALATE` → escalate. No LLM call inside the decision.

Both layers run inside the `dev-assist-escalation-policy` Hermes plugin, registered on the `pre_tool_call` hook in every specialist runtime. Fail-closed defaults apply when the rule engine errors, when `PROJECT-CONCEPT.md` is missing or malformed, or when the predicate evaluator raises.

The optional advisory-narrative call in `ESCALATION-POLICY.md` v0.1.1 § 5.5 — a Russian-language description of the escalation, generated by the runtime's catalog main model — is **NOT in the decision path**: it runs only after the deterministic classifier has already returned `ESCALATE`, and a deterministic English fallback narrative is used if the LLM call times out or fails. The advisory call exists for Founder UX, not for policy enforcement.

## Considered Options

### Option A — Deterministic rules + structured concept anchor (CHOSEN, v0.1.1)

How it works: as in the Decision section. § 4 catches enumerated high-stakes categories; § 5 catches concept-level deviations by structured comparison against `PROJECT-CONCEPT.md` § 2; both layers are pure deterministic functions of the candidate action.

Trade-offs:

- + Both decisions are pure functions: same input always produces same verdict; unit-testable end-to-end (TKT-023 § 1).
- + Auditable: every escalation records `trigger_kind` (`deterministic_rule:<id>` or `concept_deviation:<rule_id>`) and a `cite` back to the line in `ESCALATION-POLICY.md` § 4 or `PROJECT-CONCEPT.md` § 2.
- + Zero per-call LLM cost in the decision path; latency bounded by Python primitives (~ms, not seconds).
- + Tunable: adding/removing rules is small; modifying the structured anchor goes through the Founder loop the same way the rule set does.
- + No probabilistic component anywhere in the escalate / proceed decision — force-push cannot be silently classified as PROCEED on a bad day.
- − Concept-deviation rule set must be enumerated up front. Novel deviation categories require an Architect-led PR to extend `PROJECT-CONCEPT.md` § 2. Mitigated: `concept:edit_concept_anchor` (§ 4.10) makes the anchor itself an auditable governance artifact, so concept-rule additions are explicit and reviewed.
- − Every predicate type added to the `match` schema (e.g., `content_diff_touches`) is a small extension to the plugin's evaluator. Mitigated: predicate types are short, pure-Python, and the Reviewer can read the diff.

### Option B — Deterministic rules + LLM classifier (rejected by RV-SPEC-012 F3, was the v0.1.0 choice)

How it worked: § 4 caught enumerated categories; § 5 delegated the concept-deviation decision to an opaque auxiliary LLM call. Verdict came from `{"verdict":"ASK_FOUNDER"}` JSON parsing.

Trade-offs:

- + Adapts to novel concept-deviation categories without human rule updates.
- + Low to maintain: prompt is text.
- − Decision path is probabilistic. Same candidate action can be classified differently across runs (model temperature, retry timing). Reviewer (RV-SPEC-012 F3) flagged this as a non-specification: an Executor cannot implement the escalation policy as verifiable code if the verdict comes from an opaque LLM.
- − Auditability is weak: every concept-deviation escalation reads "the LLM said so" instead of "rule X matched on input Y."
- − LLM provider outage, rate-limit, or content-filter refusal can fail-open on the concept-deviation arm if not carefully handled. The v0.1.0 spec required fail-closed-on-timeout, which would block all work whenever the auxiliary LLM is unreachable — in practice an availability hazard.
- − Per-classification cost (≤0.001 USD) is small but non-zero; latency budget (≤10 s) is loose enough that decision-path latency is dominated by the LLM call.

Rejected by RV-SPEC-012 F3 ("the PRD requires the Architect to produce a specification that can be implemented as deterministic logic, not 'the LLM decides'"). Option A in v0.1.1 supersedes.

### Option C — Deterministic rules only (no concept-deviation classifier at all)

How it works: the § 4 rule set is the entire policy. Anything not matched proceeds; concept-level deviations are not separately checked.

Trade-offs:

- + Lowest implementation cost.
- + Smallest moving-parts surface.
- − Catches only what § 4 enumerates. The PRD § 13.1 first limb ("strongly deviates from the original concept") would be unenforced.
- − Every concept-level deviation case (e.g., “assistant adds a managed-Postgres dependency”, “assistant switches the deployment target to Kubernetes”) would have to be re-encoded into § 4 as a specific deterministic rule. § 4 already does this for the obvious cases (paid:*, deploy:*, scope:*, concept:*); but holding the concept-anchor structurally separate (§ 5 + `PROJECT-CONCEPT.md`) lets the deviation rules cite the PRD/ADR clauses directly, which is much more legible than packing them into § 4.

Rejected: § 5 + structured anchor is materially more legible and maintainable than packing every concept-deviation rule into § 4.

### Option D — Static type-system enforcement (compile-time)

How it works: encode the policy as type constraints in the runtime's code; tool calls that would violate are rejected at compile time.

Trade-offs:

- + Strongest guarantee: violation is impossible.
- − The runtime is an LLM-driven agent; "compile-time" doesn't really apply to runtime tool argument synthesis. Tool arguments are LLM output strings.
- − A static analyzer over LLM output is itself a runtime check, which collapses to Option A or C.

Rejected: category error.

### Option E — RAG over PRD/ADRs (retrieve-augmented classification)

How it works: every tool call is classified by an LLM that has the PRD/ADRs retrieved into its context. The retrieval grounds the classifier in the current scope.

Trade-offs:

- + Higher quality classifications (the LLM has the actual scope text to reason against).
- − Probabilistic, so it inherits Option B's RV-SPEC-012 F3 rejection on the decision-path side.
- − Higher latency (retrieval + LLM call) and higher cost.
- − Embedding model required; v0.1 is `MODEL-CATALOG.md` § 2 explicit about not running embeddings yet.
- − Adds a dependency (vector store) the v0.1 budget envelope rejects (`ESCALATION-POLICY.md` § 4.6 paid:* rules).

Rejected for v0.1; revisited for v0.2 only if the structured anchor in `PROJECT-CONCEPT.md` proves insufficient (which would surface as repeated false-negatives audited by `ESCALATION-POLICY.md` § 8) and only as input to the *advisory narrative* in § 5.5, never to the decision.

### Option F — Human-in-the-loop on every tool call

How it works: every tool call goes to the Founder for approval (essentially Hermes' approval mode set to the strictest level for everything).

Trade-offs:

- + Maximum safety.
- − Defeats the purpose of high autonomy. Founder is woken up every few minutes.
- − Fails the PRD § 13.1 mandate.

Rejected outright.

## Decision Criteria And Mapping

| Criterion | Option A (rules + structured anchor, CHOSEN) | Option B (rules + LLM, rejected by F3) | Option C (rules only) | Option D (types) | Option E (RAG) | Option F (HITL all) |
| --- | --- | --- | --- | --- | --- | --- |
| Catches deterministic categories | Yes | Yes | Yes | N/A | Probabilistic | Yes |
| Catches concept drift | Yes (deterministic via structured anchor) | Yes (probabilistic) | Partial (only what § 4 enumerates) | N/A | Yes (probabilistic) | Yes |
| Decision path is deterministic | Yes | No | Yes | N/A | No | Yes |
| Cost-bounded | Yes (zero per-call cost in decision) | Yes (≤0.001 USD) | Yes | N/A | No | N/A (Founder time) |
| Latency-bounded | Yes (~ms) | Yes (cache, ≤10 s) | Yes | N/A | No | No (Founder lag) |
| Auditable | Yes (rule_id + cite) | Weak ("the LLM said so") | Yes | N/A | Weak | Yes |
| Compatible with v0.1 budget | Yes | Yes | Yes | N/A | No | N/A |
| Compatible with PRD § 13.1 | Yes (autonomy + bounded) | Yes (probabilistic) | Partial | N/A | Yes (probabilistic) | Yes (but defeats autonomy) |
| Compatible with RV-SPEC-012 F3 | Yes | No (was the v0.1.0 design) | Yes | N/A | No | Yes |
| Implementation cost | Low (config + small evaluator) | Low | Low | N/A | High | Trivial |

Option A wins on every relevant axis; the F3-rejected Option B (the v0.1.0 design) loses on "decision path is deterministic" and on "compatible with RV-SPEC-012 F3."

## Consequences

- Implementation in TKT-023 produces the `dev-assist-escalation-policy` plugin with both deterministic layers (§ 4 rule set + § 5 concept-deviation classifier) and the `pre_tool_call` hook integration.
- The deterministic rule set is enumerated in `ESCALATION-POLICY.md` § 4 and is the authoritative source for the rule layer. Rule changes go through the Architect → RV-SPEC → Founder loop.
- The structured project-concept anchor is enumerated in `PROJECT-CONCEPT.md` § 2. Anchor changes go through the same loop and are caught by the deterministic rule `concept:edit_concept_anchor` (§ 4.10) so they cannot land without Founder approval.
- The plugin loads `PROJECT-CONCEPT.md` at startup. Missing or malformed anchor = fatal startup error (`ExecStart` exits non-zero, `Restart=on-failure`); install verify catches this before the start gate (`SELF-DEPLOYMENT-CONTRACT.md` § 8).
- The plugin re-reads the anchor on `SIGHUP` so an in-place anchor update can roll out without a full systemd restart of every runtime.
- The deterministic classifier and the rule engine both run on a redacted version of the action's arguments (§ 5.4). Redaction list = the deterministic `secret:*` rule set in § 4.4; adding a new secret regex strengthens both surfaces simultaneously.
- The optional advisory-narrative call (§ 5.5) uses the runtime's catalog main model from `MODEL-CATALOG.md` v0.1.1 § 4.1. There is no separate "auxiliary classifier" model entry in the catalog. Bounds: ≤10 s, ≤0.001 USD per call. If the call fails, the plugin attaches the deterministic English fallback narrative built from `rule_id` + `cite`.
- Every escalation is durable in the `escalations` table; aggregate audit queries (`ESCALATION-POLICY.md` § 8) inform tuning.
- Fail-closed defaults: if the rule engine errors, the YAML parser raises, or the predicate evaluator raises, ESCALATE rather than PROCEED.
