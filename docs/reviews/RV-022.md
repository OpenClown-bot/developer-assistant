## 5. Findings

### HIGH ŌĆö `write_work_item` dedup check is racy (SELECT-then-INSERT)

- **Location**: `src/developer_assistant/work_queue.py:44-52`
- **Description**: The function checks for an existing dedup key with a `SELECT`, then `INSERT`s if none was found. Between the two statements another connection could insert the same dedup key, causing a unique-constraint violation or a duplicate row (depending on isolation). SQLite in WAL mode still allows this race.
- **Recommendation**: Wrap the SELECT+INSERT in an explicit transaction, or (better) use `INSERT ... ON CONFLICT DO NOTHING RETURNING id` and fall back to a SELECT only when `RETURNING` yields no row.

### MODERATE ŌĆö `write_work_item` returns `-1` on duplicate instead of the existing row id

- **Location**: `src/developer_assistant/work_queue.py:52`
- **Description**: The AC states: "refuses to insert if a row with the same dedup_key already exists ... returns the existing id instead." The implementation returns `-1`.
- **Recommendation**: Return the existing row's `id` (from the SELECT result) rather than `-1`.

### MODERATE ŌĆö `claim_work_item` missing `lease_minutes` parameter

- **Location**: `src/developer_assistant/work_queue.py:68-73`
- **Description**: AC specifies `claim_work_item(db, claiming_role, lease_minutes=30)`. The implementation hard-codes a 30-minute lease via `_default_lease_until()` and does not expose `lease_minutes`.
- **Recommendation**: Add `lease_minutes: int = 30` parameter and compute `lease_until` from it.

### MODERATE ŌĆö `release_work_item` missing `increment_attempts` parameter

- **Location**: `src/developer_assistant/work_queue.py:138-142`
- **Description**: AC specifies `release_work_item(db, work_item_id, increment_attempts=False)`. The implementation never increments `attempt_count`.
- **Recommendation**: Add `increment_attempts: bool = False` and, when `True`, increment `attempt_count` in the UPDATE statement.

### MODERATE ŌĆö `revoke_founder_binding` signature mismatches AC

- **Location**: `src/developer_assistant/founder_identity.py:73-78`
- **Description**: AC specifies `revoke_founder_binding(db, binding_id)`. The implementation takes `(adapter_id, upstream_user_id)` instead.
- **Recommendation**: Either update the helper signature to accept `binding_id: int`, or update the AC text if the composite-key lookup is the intended interface. Given the table's `UNIQUE (adapter_id, upstream_user_id)` constraint, the composite lookup is pragmatic, but the ticket AC should be updated to match.

### MODERATE ŌĆö `read_pending_escalations` missing 5-minute re-surface cooldown

- **Location**: `src/developer_assistant/escalations.py:60-82`
- **Description**: AC requires filtering with `surfaced_at IS NULL OR surfaced_at < now-5min` so that re-surfacing is retried only after a 5-minute cooldown. The implementation returns all `pending` and `surfaced` rows without age filtering.
- **Recommendation**: Add a `surfaced_at` cooldown clause to the WHERE condition.

### MODERATE ŌĆö `write_escalation` builds `options_json` via unsafe string concatenation

- **Location**: `src/developer_assistant/escalations.py:45`
- **Description**: The helper concatenates `options` strings with manual escaping to form a JSON array. This violates the AC "no string concatenation of user-supplied values" (even though it is concatenated into a JSON payload, not directly into SQL, the pattern is brittle and can produce malformed JSON for edge-case strings).
- **Recommendation**: Use `json.dumps(options)` and pass the result as a bound parameter.

### MODERATE ŌĆö Missing threading-based concurrent-claim test

- **Location**: `tests/test_work_queue.py`
- **Description**: TKT-022 ┬¦6 Test/Validation Requirements explicitly mandates: "Tests must verify atomicity of claim_work_item by simulating two concurrent callers (use threading or sequenced subprocess invocations against the same WAL-mode connection-set) and asserting exactly one wins." No such test exists.
- **Recommendation**: Add a `threading` test that shares a single file-backed WAL database between two threads and asserts mutual exclusion.

### MODERATE ŌĆö Missing dedup-key reuse-after-completed test

- **Location**: `tests/test_work_queue.py`
- **Description**: TKT-022 ┬¦6 requires verifying "dedup key behavior with the same key in pending (rejected), claimed (rejected), failed (rejected), and completed (allowed-because-status-is-terminal-and-key-is-reusable) states." The existing `test_dedup_key_prevents_duplicate` only tests `pending` reuse. There is no test for `claimed`, `failed`, or `completed` transitions.
- **Recommendation**: Add tests that transition a dedup-key item through `claimed` ŌåÆ `completed` and verify re-insertion is allowed after `completed`, and still rejected after `failed`.

### LOW ŌĆö `read_pending_escalations` default `limit=100` vs AC `limit=10`

- **Location**: `src/developer_assistant/escalations.py:63`
- **Description**: AC specifies `limit=10`. Implementation uses `limit=100`.
- **Recommendation**: Change default to `10` to match AC.

### LOW ŌĆö `expire_old_escalations` missing `max_age_days` parameter

- **Location**: `src/developer_assistant/escalations.py:144`
- **Description**: AC specifies `expire_old_escalations(db, max_age_days=7)`. The implementation hard-codes 7 days.
- **Recommendation**: Add `max_age_days: int = 7` parameter.

## 7. Security / Process Notes

- **Secrets exposure**: None. No tokens, PATs, raw chat IDs, or hostnames appear in source or tests.
- **Write zone compliance**: Confirmed. PR touches only files listed in TKT-022 ┬¦5 Allowed Files.
- **Other**: The `dedup_key` is implemented as a top-level column with a SQLite `UNIQUE` constraint, whereas `OPERATIONAL-STATE-STORE.md` ┬¦3.5 and `ADR-006` describe it as living inside `payload_json.dedup_key`. This deviation is pragmatic because it enables native `UNIQUE` enforcement at the database level. Recommendation: update architecture docs to reflect the top-level column, or add a note explaining the design choice.
