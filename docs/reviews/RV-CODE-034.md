## 9. Residual Risks

1. **Stale token counts in footers** (MEDIUM Finding above). Fixable in minutes; does not affect runtime behavior.
2. **Case sensitivity of MCP exclusion**. `is_mcp_excluded` is case-sensitive (`MCP:foo` is accepted). Hermes skill names are conventionally lowercase, so this is acceptable for v0.1. If the project ever uses mixed-case skill naming, the exclusion should be lower-cased before matching.
3. **Tokenizer variance**. The stdlib fallback (`ceil(len(text)/4)`) is documented as ±15% for English markdown. If Hermes adopts a different tokenizer, the numbers should be refreshed. The parenthetical `(tokenizer: cl100k_base_chars_per_token_fallback)` in every footer makes this visible.
4. **ARCH-002 § 5.0.1 stale reference**. ARCH-002 line 527 references `§ 5.0.1-5.6` (per-role tables); after renumbering this should read `§ 5.0.2-5.6`. ARCH-002 is a forbidden zone for TKT-040; surfaced by the SO for a follow-up Architect cycle.
5. **MULTI-HERMES § 2 stale prompt path**. Line 27 references `docs/prompts/orchestrator.md` which does not exist; the actual file is `docs/prompts/runtime-hermes-orchestrator.md`. Out-of-zone for TKT-040; surfaced by the SO for follow-up.
6. **Custom skills not yet on disk**. All 15 `dev-assist-*` skills contribute 0 tokens because their `SKILL.md` files do not exist yet (TKT-021 / TKT-025 will populate them). The measurement script auto-discovers them when they land, but `role-context-budgets.md` will need a refresh after each population pass.
