## 5. Findings

### Medium ŌĆö Resource leak: `urllib.request.urlopen` response never closed in success path

- **Location**: `src/developer_assistant/model_catalog.py:232` (`_probe_identifier`)
- **Description**: The success path assigns `resp = urllib.request.urlopen(...)` and reads it, but never closes the response object. `HTTPError` exception objects are also left unclosed. This produces `ResourceWarning` in test output (observed on 401 and 404 stub cases) and is a real resource leak in production use.
- **Recommendation**: Wrap the `urlopen` call in a `with` statement, or explicitly call `resp.close()` / `exc.close()` in each branch.

### Medium ŌĆö `probe_omniroute` public API does not support exhaustive probing

- **Location**: `src/developer_assistant/model_catalog.py:305-313`
- **Description**: The ticket spec states `probe_omniroute` should support probing fallbacks "(in `--exhaustive` mode)". The function signature has no `exhaustive` parameter and only probes the main model. The CLI implements exhaustive mode by directly calling the private `_probe_identifier`, forcing non-CLI consumers to use a private symbol.
- **Recommendation**: Add `exhaustive: bool = False` to `probe_omniroute` and return a list of `ProbeResult`, or make `_probe_identifier` a public/documented API.

### Low ŌĆö CLI imports private module symbols `_CATALOG` and `_probe_identifier`

- **Location**: `src/developer_assistant/cli/model_catalog_cli.py:8-16`
- **Description**: The CLI imports `_CATALOG` (to list all roles) and `_probe_identifier` (to run exhaustive probes). This creates a refactor hazard: renaming either private symbol will break the CLI without a compile-time signal.
- **Recommendation**: Either expose the symbols publicly (e.g. `CATALOG`, `probe_identifier`) or shift the exhaustive-loop logic into a public library function so the CLI only imports public names.

### Low ŌĆö Timeout test stub sleeps 5s, unnecessarily slowing the suite

- **Location**: `tests/test_probe_omniroute.py:96` (`_TimeoutHandler.sleep_seconds = 5.0`)
- **Description**: The client timeout is only 0.5s. A server sleep of 1.0s is sufficient to guarantee the timeout fires. 5.0s is wasteful.
- **Recommendation**: Reduce `sleep_seconds` to `1.0`.

### Low ŌĆö Test flakiness observed in `TestProbeMismatchedModel`

- **Location**: `tests/test_probe_omniroute.py`
- **Description**: On the first full-suite run, `test_mismatched_model_returns_not_resolved` failed (no assertion output captured because the run was truncated by shell buffering). On re-run in isolation and full-suite, it passed. This suggests a possible race between daemon server thread teardown and the next test's setup, or transient port-reuse interaction.
- **Recommendation**: Monitor in CI. If it recurs, add a small `time.sleep(0.05)` after `server.shutdown()` in `tearDownClass`, or use `threading.Event` to ensure the serve thread has exited.

## 8. Verdict

**pass_with_changes**

The PR correctly implements every functional acceptance criterion. The catalog parser, assignment lookup, runtime config verification, OmniRoute probe, CLI, and round-trip test are all present and behave as specified. The implementation aligns with `MODEL-CATALOG.md` v0.1.1 ┬¦ 4.1, ADR-009, and `ESCALATION-POLICY.md` ┬¦ 4.6.

Two medium-severity code-quality items should be addressed before ratification:
1. Close `urllib.request.urlopen` response objects in `_probe_identifier` to eliminate `ResourceWarning` and resource leaks.
2. Either add `exhaustive` to the public `probe_omniroute` API or promote `_probe_identifier` to public status, so the CLI does not need to import private symbols.

Both are non-breaking refactors and can be resolved in a follow-up commit without altering architecture or test semantics.

## 9. Residual Risks

- The catalog parser binds to the exact `### 4.1` heading string. If `MODEL-CATALOG.md` is restructured in a future version, the parser will break. This is an accepted risk because catalog changes are Founder-escalated per `ESCALATION-POLICY.md` ┬¦ 4.6.
- `MODEL-CATALOG.md` frontmatter says `version: 0.2.0`, while the code and ticket reference `v0.1.1`. The parser does not validate the document version header; it only validates the table contents. This is acceptable because the table is the authoritative assignment data, but a version-mismatch guard would add defensive depth.
- The `except CatalogViolation: pass` pattern inside `_cmd_probe_omniroute` is unusual (raising an exception only to swallow it). It does not propagate the escalation to the shell caller; only the non-zero exit code does. Future refactor should make the escalation helper return a structured message instead of raising, or let the exception propagate naturally.
