# TKT-016: Bind GitHub Executors To Real Runtime HTTP And Git

## 1. Scope

Implement concrete runtime bindings for the TKT-008 / TKT-014 GitHub execution protocols so the existing GitHub PR integration can use real HTTP and constrained `git` execution instead of mocked executors.

This ticket must bind:

- TKT-008 `RESTExecutor` protocol calls to real execution of TKT-014 `GitHubRESTRequest` objects.
- TKT-008 `GitExecutor` protocol calls to real execution of TKT-014 `GitCommand` objects.
- Runtime error/log/progress handling to the existing TKT-014 and TKT-008 token-redaction guarantees.

The implementation should remain the smallest practical runtime adapter around the already-reviewed request and command builders. It must not reimplement the GitHub workflow layer or replace TKT-008's orchestration logic.

## 2. Non-scope

- Do not enable Hermes bundled `github-pr-workflow`, `github-issues`, or `github-auth` with production `GITHUB_TOKEN` / `GH_TOKEN` credentials.
- Do not add `GITHUB_TOKEN` or `GH_TOKEN` fallback behavior; `PROJECT_GITHUB_PAT` remains the approved runtime credential path.
- Do not store tokens in repository files, `.env`, credential files, token-bearing remotes, command-line arguments, logs, exceptions, telemetry, tests, or documentation.
- Do not implement autonomous merge or bypass founder acknowledgement before merge in v0.1.
- Do not bypass ticket, CI, PR, Reviewer, or founder gates.
- Do not implement idempotent branch/PR retry behavior from `TKT-NEW-008-A`; document any runtime retry limitation instead.
- Do not implement a live smoke-test harness from `TKT-NEW-008-B` unless it is an optional, explicitly gated path with sanitized output and no mandatory live credentials.
- Do not implement GitHub state persistence from `TKT-NEW-008-C`; keep persistence as a separate follow-up unless a minimal non-secret runtime field is strictly required by this adapter.
- Do not rework Telegram transport from TKT-015 except to preserve existing founder-facing progress/error text boundaries.
- Do not run the full Telegram-to-PR orchestration trial; `TKT-011` remains the later end-to-end trial after live GitHub executor binding and live Telegram smoke readiness are confirmed.
- Do not implement VPS deployment, OpenClaw, marketplace/community skills, project-local plugins, GitHub App provisioning, or credential rotation.

## 4. Acceptance Criteria

- [ ] A concrete `RESTExecutor` implementation executes TKT-014 `GitHubRESTRequest` objects against GitHub's REST API using the approved `PROJECT_GITHUB_PAT` credential path through `load_credential()` or the existing TKT-008 integration entry point.
- [ ] The REST executor injects authorization only at send time, never stores authorization headers in durable request objects, logs, returned metadata, exception text, progress text, or repository artifacts.
- [ ] REST execution supports the request methods and response shapes already used by TKT-008/TKT-014 for repository create/register, PR open/update, check-status read, PR metadata read, and any already-existing merge-gate path.
- [ ] REST errors, HTTP error bodies, URLs, headers, response metadata, and raised exceptions are sanitized with existing token-redaction helpers before they can reach logs, test output, Telegram progress text, or caller-visible errors.
- [ ] A concrete `GitExecutor` implementation executes TKT-014 `GitCommand` objects through subprocess or an equivalent runtime command boundary without using a shell, without accepting raw command strings, and without allowing token-bearing remotes or command-line token arguments.
- [ ] The git executor only executes commands that pass TKT-014 constrained command validation and continues blocking force push, `--force-with-lease`, hard reset, branch deletion, token-bearing remotes, shell metacharacter hazards, and other prohibited command forms covered by existing tests.
- [ ] Git stdout, stderr, command rendering, return-code errors, and exception text are sanitized before storage or display; no token value can appear in output, logs, Telegram text, or test failure messages.
- [ ] The adapter can be injected into the existing TKT-008 GitHub PR integration without replacing the high-level integration API or bypassing founder merge acknowledgement.
- [ ] Runtime configuration is explicit and non-secret: environment variable names may be documented, but secret values and private runtime config are not committed.
- [ ] Unit tests cover successful REST execution, REST failure redaction, successful git execution, git failure redaction, command-validation enforcement, subprocess shell avoidance, credential-source rejection, and no use of Hermes bundled GitHub skills.
- [ ] Integration-style tests use mocked HTTP and subprocess boundaries by default. Any live GitHub smoke path is optional, disabled by default, requires an explicit non-secret gate, uses `PROJECT_GITHUB_PAT` only, prints only sanitized repository/PR URLs, and documents cleanup expectations.
- [ ] `python scripts/validate_docs.py` passes.
- [ ] `python -m unittest discover -s tests -p "test_*.py" -v` passes.

## 8. Risks

- Real HTTP execution can leak tokens through headers, URLs, error bodies, repr output, debug logs, or exception messages if redaction is incomplete.
- Real subprocess execution can leak tokens through command rendering, stdout/stderr, git credential helpers, remotes, or shell expansion if command boundaries are too permissive.
- GitHub live API behavior can differ from mocked request construction, especially around rate limits, branch conflicts, PR uniqueness, check-suite timing, and permission errors.
- Runtime retries remain vulnerable to the TKT-008 idempotency gap until `TKT-NEW-008-A` is implemented or a clear operator-visible fail-fast policy is adopted.
- Active GitHub integration state remains in memory until `TKT-NEW-008-C` persists it through SQLite or another approved runtime persistence mechanism.
- Optional live smoke tests can create branches or PRs that require manual cleanup if interrupted.
- Windows and Linux subprocess behavior may differ; tests should keep shell avoidance and argument-list execution portable.
