# TKT-015: Wire Telegram Adapter To Hermes Gateway Transport

## 1. Scope

Wire the TKT-006 Telegram founder interaction adapter to live Hermes Telegram gateway transport for one trusted founder and one active project.

This ticket should bind inbound Hermes/Telegram messages into `TelegramEvent` inputs, bind outbound `TelegramSender.send()` calls to the reviewed Hermes Telegram gateway path, and preserve the repository governance boundary defined by `HERMES-RUNTIME-CONTRACT.md`.

## 2. Non-scope

- Do not implement GitHub REST or `git` executor binding; TKT-NEW-008-D remains the follow-up for real GitHub HTTP/subprocess execution.
- Do not implement a full Telegram-to-PR trial; TKT-011 remains the end-to-end trial ticket after live Telegram and GitHub runtime bindings are ready.
- Do not change Telegram command semantics beyond the TKT-006 adapter contract unless a narrowly scoped bug is found and documented in the Execution Log.
- Do not implement OpenClaw, a web dashboard, autonomous merges, live VPS deployment, or marketplace/community skills.
- Do not commit `.env`, `config.yaml` containing secrets, Telegram bot tokens, raw private chat IDs, raw private user IDs, PATs, API keys, or VPS credentials.

## 4. Acceptance Criteria

- [ ] Inbound Hermes Telegram gateway messages are converted into TKT-006 `TelegramEvent` instances without persisting raw private chat IDs or raw private user IDs in repository artifacts, committed fixtures, logs, or errors.
- [ ] The inbound transport enforces the reviewed founder allowlist or DM pairing requirement before adapter handling; allow-all modes remain disabled.
- [ ] Outbound `TelegramSender.send()` calls are delivered through the reviewed Hermes Telegram gateway path and preserve Russian founder-facing text from the TKT-006 adapter.
- [ ] Runtime configuration validation enforces TKT-012 / `HERMES-SKILL-ALLOWLIST.md` constraints: `TELEGRAM_BOT_TOKEN` may be used only from approved runtime secret storage, founder allowlist or DM pairing is configured, `GATEWAY_ALLOW_ALL_USERS` and `TELEGRAM_ALLOW_ALL_USERS` remain unset/false, polling is preferred for v0.1, and webhook mode requires `TELEGRAM_WEBHOOK_SECRET`.
- [ ] Manual or automated smoke coverage demonstrates sanitized `/new_project`, `/status`, `/decisions`, `/pause`, `/resume`, free-form classification, specialist-question delivery, founder-answer capture, and progress-report delivery through the transport boundary.
- [ ] Smoke-test documentation records commands, sanitized chat/user references, and outcomes without token values, raw private chat IDs, raw private user IDs, or private runtime config.
- [ ] The implementation does not enable Hermes bundled GitHub credential-bearing skills, marketplace skills, project-local plugins, OpenClaw plugins, autonomous merge, or live deployment.
- [ ] Existing TKT-006 adapter behavior remains covered by unit tests, and new transport-boundary tests use mocked Hermes/Telegram gateway inputs and send calls where live credentials are unavailable.
- [ ] `python scripts/validate_docs.py` passes.
- [ ] `python -m unittest discover -s tests -p "test_*.py" -v` passes.

## 8. Risks

- Hermes gateway APIs or message payload shape may differ from current assumptions, requiring a small adapter shim or a documented follow-up.
- Telegram transport logs can accidentally expose bot tokens, raw chat IDs, or raw user IDs if sanitization is incomplete.
- Webhook mode introduces public endpoint exposure and secret-handling risk; polling is preferred for v0.1 unless webhook mode is explicitly required and protected by `TELEGRAM_WEBHOOK_SECRET`.
- TKT-006 currently uses in-memory progress scheduling; this ticket may deliver progress messages through live transport but does not persist scheduling through SQLite.
- TKT-006's filesystem artifact writer remains a separate concern; this ticket should not broaden into durable file-path generation unless needed for transport wiring.
- End-to-end Telegram-to-PR behavior still depends on GitHub runtime executor binding from TKT-NEW-008-D and the later TKT-011 trial.
