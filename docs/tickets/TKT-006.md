# TKT-006: Implement Telegram Founder Interaction Through Hermes

## 1. Scope

Implement the minimal Telegram founder interaction path through Hermes Agent for one trusted founder and one active project.

## 2. Non-scope

- Do not implement OpenClaw.
- Do not build a public web dashboard.
- Do not implement autonomous merge or live VPS deployment.
- Do not enable unreviewed marketplace skills/plugins.

## 4. Acceptance Criteria

- [ ] Hermes can receive Telegram messages from an allowlisted founder chat.
- [ ] `/new_project`, `/status`, `/decisions`, `/pause`, and `/resume` behavior is implemented or explicitly documented if Hermes provides equivalent commands.
- [ ] Free-form founder messages are classified as intake, answer, clarification, approval, rejection, or general question.
- [ ] Specialist-agent questions can be routed to Telegram with context, options, recommendation, impact, and urgency.
- [ ] Founder answers that affect durable decisions are written to repository artifacts.
- [ ] Progress reports can be sent after ticket/review milestones and on a 30 to 60 minute schedule.
- [ ] Telegram token and chat identifiers are not committed to the repository.
- [ ] Production `TELEGRAM_BOT_TOKEN` use follows `HERMES-SKILL-ALLOWLIST.md` constraints: founder allowlist or DM pairing configured, allow-all flags disabled, polling preferred for v0.1, webhook mode only with `TELEGRAM_WEBHOOK_SECRET`, and no token value in git-tracked config.
- [ ] `python scripts/validate_docs.py` passes.

## 8. Risks

- Telegram session state may diverge from repository decisions if capture is incomplete.
- Hermes setup details may differ by deployment mode.
- Manual Telegram tests can accidentally expose chat IDs or tokens if logs are not sanitized.
- Telegram gateway source review passed only with the constraints recorded in `HERMES-SKILL-ALLOWLIST.md`; implementation must not broaden the reviewed credential-bearing path.
- State-store FK enforcement and partial-update semantics from TKT-013 are now available, but the runtime adapter must preserve project-binding integrity when updating chat/project mappings.

## 9. Dependencies

- TKT-005 must be done and must define the Hermes runtime integration contract.
- TKT-009 must be done before enabling credential-bearing Telegram runtime capabilities.
- TKT-007 must be done before relying on durable Telegram chat bindings, project registry, or scheduled progress reports.
- TKT-012 is done; production Telegram bot credentials may be used only under the reviewed constraints documented in `HERMES-SKILL-ALLOWLIST.md`.
- TKT-013 is done; rely on the documented SQLite FK enforcement, project-binding upsert semantics, and single-thread/WAL guidance in `OPERATIONAL-STATE-STORE.md`.
- User-approved baseline: Telegram uses hybrid commands plus free-form classification.
- User-approved baseline: operational state defaults to SQLite on VPS unless Hermes native persistence is proven sufficient.
