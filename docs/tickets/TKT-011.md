# TKT-011: Run First Telegram-to-PR Orchestration Trial

## 1. Scope

Run the first end-to-end trial of the revised v0.1 workflow after the TKT-017 readiness harness is available on `main`.

The trial must begin from an authenticated Telegram founder interaction through the TKT-015 Hermes Telegram transport boundary, use the TKT-006 founder interaction logic to select or confirm one already-ready implementation ticket, dispatch the normal role-separated Ticket Orchestrator / Executor / Reviewer flow, create a branch and PR through the TKT-008/TKT-014/TKT-016 GitHub path, observe CI/docs validation, collect Reviewer and PR-Agent gate results, send sanitized Telegram progress updates, and request explicit founder acknowledgement before merge.

Before any live end-to-end operation begins, the Executor/Orchestrator must run or invoke the TKT-017 gated readiness harness semantics for both lanes:

- GitHub lane: explicit non-secret gate enabled, `PROJECT_GITHUB_PAT` available through the approved credential path, unique smoke branch behavior, sanitized evidence, and `pass`/`blocked`/`fail` outcome.
- Telegram lane: explicit non-secret gate enabled, reviewed Telegram runtime constraints satisfied, sanitized chat/user labels, and a live gateway proof callback or equivalent live proof that can produce `pass`.

If either lane is `blocked`, `fail`, or unavailable, stop before the full trial and document the sanitized blocker in this ticket's Execution Log and the PR body. Do not treat TKT-017's existence alone as live readiness.

## 2. Non-scope

- Do not perform live VPS deployment.
- Do not bypass ticket status, CI, PR-Agent, Reviewer LLM, or founder acknowledgement gates.
- Do not merge without explicit founder acknowledgement.
- Do not introduce OpenClaw.
- Do not enable Hermes bundled `github-pr-workflow`, `github-issues`, or `github-auth` with production `GITHUB_TOKEN` / `GH_TOKEN` credentials.
- Do not use `GITHUB_TOKEN`, `GH_TOKEN`, `~/.git-credentials`, token-bearing remotes, committed config, or command-line token arguments for production GitHub operations; `PROJECT_GITHUB_PAT` is the approved v0.1 credential path.
- Do not commit `.env`, `config.yaml` containing secrets, Telegram bot tokens, GitHub PATs, LLM API keys, VPS SSH keys, raw Telegram chat IDs, raw Telegram user IDs, private runtime config, token-bearing remotes, or sensitive VPS details.
- Do not store or print raw Telegram chat IDs or raw Telegram user IDs in repository artifacts, PR bodies, review artifacts, logs, screenshots, copied terminal output, or smoke evidence; use sanitized labels such as `chat:founder` and `user:founder`.
- Do not implement idempotent branch/PR retry behavior from `TKT-NEW-008-A`; if a partial live GitHub operation blocks retry safety, fail closed with an operator-visible blocker.
- Do not implement GitHub integration state persistence from `TKT-NEW-008-C`; do not imply runtime continuity across restart beyond the existing operational state.
- Do not open public endpoints, switch Telegram to webhook mode, change credential scopes, rotate credentials, spend money, enable marketplace/community skills, enable project-local plugins, or perform live deployment.
- Do not execute the trial directly from this architecture PR; this ticket only authorizes a later implementation/orchestration PR after normal review gates.

## 8. Risks

- GitHub remote, local clone state, `git` executable, or `PROJECT_GITHUB_PAT` may be unavailable or insufficiently scoped; this must block readiness/trial execution rather than falling back to unsafe credential paths.
- Telegram/Hermes live gateway credentials, founder allowlist/DM pairing, or live proof callback may be unavailable; config-only checks must not produce a pass.
- Live smoke or trial evidence can leak secrets, raw Telegram identifiers, private runtime config, HTTP errors, git remotes, command output, or screenshots if sanitization is incomplete.
- GitHub smoke/trial operations can leave branches or PRs behind if interrupted; cleanup must be documented and destructive cleanup must require explicit approval where policy requires it.
- TKT-008 branch/PR creation is not retry-idempotent until `TKT-NEW-008-A`; unattended retries are not allowed, and partial failure should fail closed with an operator-visible blocker.
- GitHub integration state remains in memory until `TKT-NEW-008-C`; the trial must not imply production continuity across process restart.
- Hermes Telegram gateway payload shape may differ from mocked TKT-015 assumptions and may require a small follow-up shim.
- PR-Agent or Reviewer can still find blockers after the trial PR opens; founder acknowledgement must not be requested until those gates are resolved.
- The trial may reveal missing validation rules, prompt ambiguity, role-boundary confusion, or orchestration-state update gaps.
