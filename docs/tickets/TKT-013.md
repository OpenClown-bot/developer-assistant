# TKT-013: Harden Operational State Store Semantics

## 1. Scope

Address the low-severity operational state store follow-ups from RV-CODE-010 before Telegram and GitHub runtime work relies heavily on the store.

## 2. Non-scope

- Do not implement Telegram command handling.
- Do not implement GitHub repository or PR automation.
- Do not change the selected SQLite backend.
- Do not add a migration framework unless strictly required for the minimal schema hardening.
- Do not store secrets or private identifiers in repository artifacts.

## 3. Required Context

- `AGENTS.md`
- `CONTRIBUTING.md`
- `docs/architecture/ARCH-001.md`
- `docs/architecture/HERMES-RUNTIME-CONTRACT.md`
- `docs/architecture/OPERATIONAL-STATE-STORE.md`
- `docs/architecture/adr/ADR-002-repository-state.md`
- `docs/reviews/RV-CODE-010.md`
- `docs/tickets/TKT-007.md`

## 4. Acceptance Criteria

- [ ] SQLite foreign-key behavior is either enforced with documented schema/API changes or explicitly documented as intentionally unenforced for v0.1 with a rationale.
- [ ] `upsert_project_binding` optional-field behavior is either hardened to preserve omitted fields or documented and tested as full-replacement semantics.
- [ ] Tests cover the selected `upsert_project_binding` semantics.
- [ ] `docs/architecture/OPERATIONAL-STATE-STORE.md` is updated with any schema/API behavior changes and single-thread/WAL guidance if relevant.
- [ ] No secret values or private Telegram identifiers are introduced.
- [ ] `python scripts/validate_docs.py` passes.
- [ ] Relevant state-store tests pass.

## 5. Allowed Files

- `src/developer_assistant/state_store.py`
- `tests/test_state_store.py`
- `docs/architecture/OPERATIONAL-STATE-STORE.md`
- `docs/tickets/TKT-013.md` Execution Log only

## 6. Test/Validation Requirements

- Run `python scripts/validate_docs.py`.
- Run the relevant state-store test command, currently `python -m unittest discover -s tests -p "test_*.py" -v`.

## 7. PR Requirements

- Link this ticket.
- Include validation and test output.
- State whether the implementation chose enforced foreign keys or documented unenforced relationships.
- State whether `upsert_project_binding` uses partial-update or full-replacement semantics.

## 8. Risks

- Schema changes may affect existing local operational databases if any exist.
- Changing upsert semantics without documenting caller expectations can create runtime confusion.
- Foreign-key enforcement may require careful reset/delete ordering.

## 9. Dependencies

- TKT-007 must be done.
- RV-CODE-010 findings should be treated as guidance for this hardening scope.
