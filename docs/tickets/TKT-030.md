# TKT-030: Recovery Playbook Execution Discipline

## 1. Scope

Make the `RECOVERY-PLAYBOOK.md` runbook executable and verifiable so that every command path it documents continues to work as the system evolves, per `OBSERVABILITY-CONTRACT.md` v0.1.0 § 13 (FR-OBS-10):

- A test harness `tests/test_recovery_playbook_invariants.py` that:
  - Parses `docs/operations/RECOVERY-PLAYBOOK.md` and extracts every fenced code block whose first line is a shell command (e.g., `dev-assist-cli ...`, `journalctl ...`, `systemctl ...`, `curl ...`, `sqlite3 ...`).
  - For each command, validates structural correctness against a mapping `tests/fixtures/recovery_playbook/command_validators.py` that knows how to evaluate each command shape against the actual codebase (e.g., `dev-assist-cli` subcommand exists, the systemd unit name exists in the install template, the SQL referenced exists in the schema).
  - Reports a structured failure when the playbook references a command, subcommand, unit name, port, table, or file path that is not actually present in the codebase.
- A small CI invariant that runs the harness as part of `python -m unittest discover` and fails the build if the playbook drifts from the codebase.
- A documentation note in `RECOVERY-PLAYBOOK.md` § 2 (added to the existing § 2 Quick Reference Card via a follow-up edit included in this ticket's PR — yes, this is allowed because the playbook is in the Architect's write zone) that explicitly references this ticket's harness.
- A pre-merge convention added to `CONTRIBUTING.md` (or an equivalent contributor doc) that says "if you change a command surface (CLI subcommand, systemd unit name, port, table, schema), update RECOVERY-PLAYBOOK.md in the same PR".
- Tests in `tests/test_recovery_playbook_invariants.py` covering: every documented `dev-assist-cli` subcommand resolves to a real argparse subparser; every `systemctl` unit name resolves to a real install-template unit; every `curl http://127.0.0.1:818<n>` references a real health-endpoint port; every `sqlite3` query references a real column in the rendered schema; every `scripts/<file>.sh` reference resolves to a real script (when the script lands; for v0.1 the harness is permissive and emits a warning rather than a failure when scripts have not yet landed in the repo).

## 8. Risks

- The harness is parser-driven; if the playbook adopts a non-fenced command convention (e.g., inline backtick commands inside prose), the parser misses them. Mitigated: the parser also walks inline single-backtick spans and applies a lightweight classifier; ambiguous cases emit warnings, not failures.
- The harness is brittle to argparse refactors in TKT-027; the test would fail when `dev-assist-cli`'s internal argparse structure changes even if the public CLI surface is unchanged. Mitigated: the validator uses `argparse.parse_known_args(['<subcommand>', '--help'])` rather than introspecting the internal parser, so only the public surface matters.
- Adding a script reference to the playbook before the script lands triggers warnings until the script lands. Documented in the PR body so reviewers don't mistake a warning for a regression.
- The harness adds a small CI cost (parsing + validating ~200 commands). Mitigated: harness runs in-process and completes well under one second on the existing CI runner.
- The contributor convention is a documentation expectation, not a CI-enforced rule for ALL changes; only changes that touch the surfaces the harness validates are caught. Documented in the convention itself.
