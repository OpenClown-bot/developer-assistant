# TKT-020: Self-Deployment Bootstrap (install / verify / rollback / upgrade scripts and systemd units)

## 1. Scope

Implement the four shell-script entry points and six systemd unit files that operationalize `SELF-DEPLOYMENT-CONTRACT.md` v0.1.0:

- `scripts/install-self.sh` — idempotent installer that lays down the `/srv/devassist/` filesystem layout, installs Hermes Agent at `/usr/local/lib/hermes-agent/`, creates five per-runtime `HERMES_HOME` directories, renders six systemd unit files, runs `systemctl daemon-reload`, runs `scripts/verify-self.sh`, and exits without starting any runtime.
- `scripts/verify-self.sh` — connectivity-only health check that exercises the seven invariants in `SELF-DEPLOYMENT-CONTRACT.md` § 8. Among the seven, the model-probe invariant is delegated to TKT-026's `src/developer_assistant/cli/model_catalog_cli.py probe-omniroute` subcommand: `verify-self.sh` calls the CLI for each of the five role identifiers; the CLI issues a 1-token completion against `http://127.0.0.1:<omniroute_port>/v1/chat/completions`. Failure raises `paid:third_party_external_service_not_yet_supported` with no silent fallback to direct-Fireworks.
- `scripts/rollback-self.sh` — restores the most recent `operational.db` backup (the shared operational store at `/srv/devassist/state/operational.db`, NOT the per-runtime `~/.hermes/state.db` Hermes native sessions database) and corresponding runtime-state tarball, flips the `releases/current` symlink, runs verify, restarts units only if verify passes.
- `scripts/upgrade-self.sh` — fetches a new release into `/srv/devassist/releases/<id>/`, takes an `operational.db` backup, runs the in-place install against the new release, runs verify, and stops at the activation gate awaiting `--activate`. On `--activate`: stops the umbrella target, flips the `releases/current` symlink, restarts the umbrella target, runs verify again.
- `etc/systemd/system/devassist.target` — umbrella target.
- `etc/systemd/system/devassist-<role>.service` — five per-runtime services for `orchestrator`, `planner`, `architect`, `executor`, `reviewer` per `SELF-DEPLOYMENT-CONTRACT.md` § 5.2.

The scripts must use only standard Ubuntu 22.04 LTS tooling (`bash`, `systemctl`, `journalctl`, `sqlite3`, `tar`, `curl`, `git`) plus the already-required Hermes Agent, Docker, Python 3.11, Node.js 22.

## 8. Risks

- systemd unit semantics differ subtly across systemd versions; `systemd-analyze verify` may be more strict on the Founder's VPS than in CI. Mitigated by pinning the Hermes deployment to Ubuntu 22.04 LTS where systemd 249 is the standard.
- The dry-run prefix abstraction is easy to mis-implement; a script that ignores `INSTALL_DRY_RUN=1` and writes to the real `/srv/` would be a serious regression. Tests must verify dry-run containment.
- Idempotency is easy to break in subtle ways (e.g., a `chown` that doesn't notice the directory already has the right owner). Tests cover the second-run path explicitly.
- The seven verify invariants are connectivity-only; they do not exercise behavior. A unit that is `active` but unable to do work is not detected by verify. This is by design per `PRD-001.md` § 10 Q12 recommendation; behavior verification belongs to the upcoming TKT-011 trial.
- Symlink flipping (`releases/current`) under load could race if the umbrella target is started concurrently. The scripts serialize: stop, flip, start.
