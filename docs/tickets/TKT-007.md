# TKT-007: Implement Operational State Store for Telegram and Hermes Runs

## 1. Scope

Implement the minimal operational state store required by Telegram-first Hermes orchestration.

## 2. Non-scope

- Do not store secrets in the operational database.
- Do not replace repository artifacts as governance state.
- Do not implement multi-tenant isolation.
- Do not build a web dashboard.

## 3. Required Context

- `AGENTS.md`
- `CONTRIBUTING.md`
- `docs/architecture/ARCH-001.md`
- `docs/architecture/HERMES-RUNTIME-CONTRACT.md`
- `docs/architecture/adr/ADR-002-repository-state.md`
- `docs/tickets/TKT-005.md`

## 4. Acceptance Criteria

- [ ] The implementation stores Telegram chat/user allowlist or bindings without committing private identifiers unless explicitly sanitized.
- [ ] The implementation stores project registry entries mapping Telegram conversations to GitHub repositories and local workspaces.
- [ ] The implementation stores scheduled progress report timestamps.
- [ ] The implementation stores Hermes run IDs, retry/idempotency keys, and in-flight task metadata.
- [ ] The implementation does not store canonical product, architecture, security, merge, or deployment decisions without writing repository summaries.
- [ ] The implementation documents backup and reset behavior for v0.1.
- [ ] `python scripts/validate_docs.py` passes.

## 5. Allowed Files

- Runtime source/config files explicitly established by the approved implementation plan.
- `tests/`
- `docs/tickets/TKT-007.md` Execution Log only

## 6. Test/Validation Requirements

- Run `python scripts/validate_docs.py`.
- Add or run tests for project binding, scheduled-report metadata, and run-id persistence where feasible.
- Verify no secret values are written to repository files.

## 7. PR Requirements

- Link this ticket.
- Include validation and test output.
- Document selected storage backend and migration/reset behavior.
- Document which fields are non-secret metadata.

## 8. Risks

- Operational state can become hidden source of truth if durable summaries are skipped.
- Local database loss may interrupt Telegram scheduling unless reset behavior is clear.
- Storing chat identifiers may require privacy handling.
- Hermes native persistence may prove sufficient for some fields; if so, the implementation must document why SQLite is not needed for those fields.

## 9. Dependencies

- TKT-005 must be done and must define operational-state responsibilities in the Hermes runtime integration contract.
- User-approved baseline: operational state defaults to SQLite on VPS unless Hermes native persistence is proven sufficient.

**Selected storage backend:** SQLite

- Per ARCH-001 Section 8 and ADR-002: SQLite on VPS is the default unless Hermes native persistence is proven sufficient.
- Hermes native persistence is designed for conversation continuity, not structured relational queries on project bindings, scheduled timers, or idempotency-key deduplication.
- SQLite provides transactional integrity, schema enforcement, and index-backed lookups needed for idempotency and scheduling.
- Implementation is stdlib-only (sqlite3, json, datetime).
