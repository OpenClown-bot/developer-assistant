# TKT-027: `dev-assist-cli` Operator CLI

## 1. Scope

Implement the operator CLI that the Founder uses to inspect the running system, per `OBSERVABILITY-CONTRACT.md` v0.1.0 Â§ 6 (FR-OBS-03):

- A single-file Python script `src/developer_assistant/cli/dev_assist_cli.py` that:
  - Reads recent log entries from systemd journald via `journalctl --output=json --no-pager` (subprocess; the script does not depend on the `systemd` Python binding so it remains importable for offline tests).
  - Reads the operational store at `/srv/devassist/state/operational.db` directly with the stdlib `sqlite3` module (read-only, `mode=ro`, `nolock=1` query string parameters).
  - Reads localhost-only health endpoints (`http://127.0.0.1:818{1..5}/health`) for the per-runtime `/health` summary.
  - Does NOT call any specialist runtime; it works even when all runtimes are down.
- Subcommands per `OBSERVABILITY-CONTRACT.md` Â§ 6:

- An installer step (added to TKT-020's `install-self.sh`) places the script at `/opt/dev-assist/bin/dev-assist-cli` mode 0755 and creates the directory.

## 8. Risks

- The CLI parses `journalctl --output=json` output; if a future systemd version changes the JSON shape, the parser breaks. Mitigated: the parser only requires a small subset of fields (`MESSAGE`, `_SYSTEMD_UNIT`, `__REALTIME_TIMESTAMP`) which are stable across systemd versions.
- The `--format human` output is consumed by humans; future formatting changes invalidate the snapshot test. Documented: snapshot updates are an explicit PR-time decision, not silent.
- The CLI reads a possibly-large `operational.db`; for `costs --since today` the query is filtered with an indexed `ts_iso` predicate and aggregated server-side. Documented in the implementation as a query plan comment.
- Out-of-band file rotation by the journald drop-in or cron entries (TKT-020) may shorten the visible window for `errors --since 1h` if the rotation runs concurrently. Documented; the CLI prints the actual window covered.
- The CLI is intentionally not protected with privilege checks beyond what `journalctl` and the file ownership of `operational.db` enforce. Running as the `devassist` user (or root via sudo) is sufficient.
