# TKT-003: Define Hermes-Aligned Role Prompt Set

## 1. Scope

Create or update role prompts for Orchestrator, Business Planner, Architect, Executor, and Reviewer so each role has explicit mission, required reading, allowed write zone, outputs, completion criteria, and Hermes/Telegram handoff behavior aligned with ARCH-001.

## 2. Non-scope

- Do not implement Hermes runtime execution.
- Do not implement Telegram bot commands.
- Do not change production code.
- Do not approve tickets or change ticket statuses outside this ticket's Execution Log.

## 3. Required Context

- `AGENTS.md`
- `CONTRIBUTING.md`
- `docs/architecture/ARCH-001.md`
- Relevant ADRs in `docs/architecture/adr/`
- Existing prompts under `docs/prompts/`

## 4. Acceptance Criteria

- [ ] Orchestrator prompt explains Hermes runtime responsibilities, Telegram question routing, progress reports, durable state, role handoffs, and user decisions.
- [ ] Business Planner prompt explains PRD scope and how founder questions are sent through Hermes/Telegram.
- [ ] Architect prompt aligns with Hermes-first architecture and documents when to escalate blockers.
- [ ] Executor prompt requires ticket, architecture, ADRs, `CONTRIBUTING.md`, and `AGENTS.md` before implementation.
- [ ] Reviewer prompt requires PR diff, ticket, architecture, ADRs, CI results, and repository rules.
- [ ] Each prompt states allowed write zone and stop conditions for out-of-zone work.
- [ ] `python scripts/validate_docs.py` passes.

## 5. Allowed Files

- `docs/prompts/`
- `docs/tickets/TKT-003.md` Execution Log only

## 6. Test/Validation Requirements

- Run `python scripts/validate_docs.py`.
- Manually inspect prompts for consistency with `CONTRIBUTING.md` write zones and ARCH-001 Telegram/Hermes flow.

## 7. PR Requirements

- Link this ticket.
- Summarize which prompts changed.
- Note any intentional differences from ARCH-001.

## 8. Risks

- Prompt ambiguity may allow scope creep.
- Overly broad prompts may cause agents to write outside role zones.
- Telegram question routing can lose decisions if prompts do not require repository capture.

## 9. Dependencies

- ARCH-001 and ADR-001 should be approved or explicitly accepted as implementation baseline.
