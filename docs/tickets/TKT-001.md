# TKT-001: Implement Docs Artifact Validation Baseline

## 1. Scope

Create or update deterministic docs validation for the repository governance artifacts required by the Telegram-first Hermes-centered v0.1 architecture.

## 2. Non-scope

- Do not implement Hermes runtime integration.
- Do not implement Telegram bot behavior.
- Do not validate live GitHub PR state.
- Do not add secrets or credential configuration.

## 3. Required Context

- `AGENTS.md`
- `CONTRIBUTING.md`
- `docs/architecture/ARCH-001.md`
- `docs/architecture/adr/ADR-001-platform-foundation.md`
- `docs/architecture/adr/ADR-002-repository-state.md`
- Existing `scripts/validate_docs.py`, if present

## 4. Acceptance Criteria

- [ ] `python scripts/validate_docs.py` exits non-zero on missing required frontmatter keys for core docs artifacts.
- [ ] Ticket files under `docs/tickets/` are checked for sections 1 through 10 required by the ticket template.
- [ ] ADR files under `docs/architecture/adr/` are checked for `id`, `version`, and `status` frontmatter.
- [ ] Architecture docs under `docs/architecture/ARCH-*.md` are checked for `id`, `version`, and `status` frontmatter.
- [ ] Validation output names each failing file and reason.
- [ ] Existing valid repository docs pass validation.

## 5. Allowed Files

- `scripts/validate_docs.py`
- `tests/` if tests are added for the validator
- `.github/workflows/` only if needed to keep CI calling the validator
- `docs/tickets/TKT-001.md` Execution Log only

## 6. Test/Validation Requirements

- Run `python scripts/validate_docs.py`.
- If validator tests are added, run the relevant test command and document it in the PR.

## 7. PR Requirements

- Link this ticket.
- Include validation command output.
- Explain any schema decisions or intentionally deferred checks.
- Do not include unrelated documentation rewrites.

## 8. Risks

- Overly strict validation may block useful draft docs.
- Under-specified section matching may create false confidence.
- YAML parsing should remain simple or use an existing dependency already present in the project.

## 9. Dependencies

- Revised ARCH-001 and ADRs must be approved before this ticket is moved to `ready`.

## 10. Execution Log

- **Branch:** `feat/TKT-001-docs-validation-baseline`
- **Summary:** Refactored `scripts/validate_docs.py` from fail-fast to collect-all-errors model. Added ticket section validation (sections 1–10), specific frontmatter key checks per artifact type (ADR, ARCH, ticket, PRD, reviews, backlog, questions, orchestration, prompts all require `id`, `version`, `status`), and error reporting that names each failing file and reason. Added 15 unit tests in `tests/test_validate_docs.py` using `unittest` (standard library only). All existing repository docs pass validation.
- **Validation commands run:**
  - `python scripts/validate_docs.py` — passed (0 errors)
  - `python -m unittest discover -s tests -p "test_*.py" -v` — 15 tests passed
- **Known limitations:**
  - Frontmatter parsing uses regex rather than a YAML parser; complex YAML (nested keys, multiline values) may not be validated correctly.
  - Ticket section matching is based on `## N.` heading pattern; alternative heading formats are not recognized.
  - Cross-link validation (e.g., ticket references ADR IDs that exist) is intentionally deferred per TKT-001 scope.
