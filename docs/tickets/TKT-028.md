# TKT-028: Structured Per-Runtime Logging And `work_item_id` Propagation

## 1. Scope

Implement structured JSON-line logging on every specialist runtime and propagate `work_item_id` end-to-end across the five-runtime topology, per `OBSERVABILITY-CONTRACT.md` v0.1.0 § 4 (FR-OBS-01) and § 5 (FR-OBS-02):

- A logging helper module `src/developer_assistant/observability/structured_logger.py` that:
  - Wraps the stdlib `logging` module with a JSON formatter emitting one log line per event with the mandatory fields from `OBSERVABILITY-CONTRACT.md` § 4 (`ts_iso`, `level`, `runtime_role`, `work_item_id`, `model`, `tokens_in`, `tokens_out`, `latency_ms`, `event`, `message`, plus any extras passed by the caller).
  - Reads `runtime_role` from the env var `DEVASSIST_RUNTIME_ROLE` (set by the systemd unit per `SELF-DEPLOYMENT-CONTRACT.md` § 5.2).
  - Provides a context-managed correlation primitive (`with structured_logger.work_item(work_item_id=...): ...`) that pushes `work_item_id` onto a contextvar; all log lines emitted inside the block carry it.
  - Provides decorators for instrumenting LLM calls (`@instrument_llm_call(model_id="<id>")`) that emit `llm.call.start` and `llm.call.complete` (or `.fail`) events with `tokens_in`, `tokens_out`, `latency_ms`, `cost_usd`.
- A small adapter that hooks into the Hermes runtime startup so structured logging is initialized before any user code runs. The adapter is loaded as a Hermes plugin at runtime startup; see `HERMES-RUNTIME-CONTRACT.md` § 4 for the plugin loader contract.
- A propagation rule: when a runtime claims a `work_items` row, it reads the `id` and pushes it onto the contextvar for the lifetime of the work-item processing; child work items propagate via the `parent_id` field already present in the schema.
- The Orchestrator is updated to bind `work_item_id` to every Telegram-message-driven request from the moment a work item is created.
- A small middleware injected into the OmniRoute integration (TKT-031) that reads `work_item_id` from a request header (`X-DEVASSIST-Work-Item-Id`) set by the Hermes adapter and forwards it through to the structured log line emitted by the OmniRoute service.
- Tests in `tests/test_structured_logger.py` covering: JSON shape, mandatory field presence, contextvar propagation, decorator instrumentation, log-line equivalence across log levels, behavior when `runtime_role` env var is missing.
- Tests in `tests/test_work_item_id_propagation.py` covering: a synthetic work-item lifecycle (Orchestrator → Architect → Executor → Reviewer) emits log lines that all carry the same `work_item_id` value.

## 4. Acceptance Criteria

- [ ] `src/developer_assistant/observability/structured_logger.py` exists, imports cleanly, and exposes `get_logger(name: str) -> logging.Logger`, `work_item(work_item_id: str)` (context manager), `instrument_llm_call(model_id: str)` (decorator), and `init_runtime_logger()` (called once at runtime startup).
- [ ] Calling `get_logger(__name__).info("event", extra={"event": "...", "additional_field": ...})` emits a JSON-line to stderr with all mandatory fields per `OBSERVABILITY-CONTRACT.md` § 4 plus the extras.
- [ ] `runtime_role` is read from env var `DEVASSIST_RUNTIME_ROLE`; if unset, the logger uses `unknown` and emits a one-time `logger.warning` event `runtime_role_missing`.
- [ ] `with structured_logger.work_item(work_item_id="abc123"):` causes every `logger.X(...)` call inside the block (including from awaited coroutines and from spawned threads via the `contextvars`-aware copy) to emit `work_item_id: "abc123"`. Logs outside the block emit `work_item_id: null`.
- [ ] `@instrument_llm_call(model_id="deepseek-v4-pro")` wraps an LLM-call function so it emits `llm.call.start` before the call and `llm.call.complete` (or `.fail`) after, with `tokens_in`, `tokens_out`, `latency_ms`, `cost_usd` populated from the LLM response. The decorator is async-aware and works for both sync and async call sites.
- [ ] The decorator never logs prompt content or completion content; only metadata. (The body of the LLM call's prompt is potentially sensitive; FR-OBS-09 retention does not protect against prompt leakage.)
- [ ] An Orchestrator code path that creates a work-item-from-Telegram-message wraps the dispatch in `with structured_logger.work_item(work_item_id=...):` so every downstream log carries the id.
- [ ] When a specialist runtime claims a `work_items` row (via the queue lease in `MULTI-HERMES-CONTRACT.md` § 6.2), it pushes the row's `id` onto the contextvar for the lifetime of processing. The propagation is preserved across the runtime's Hermes session boundaries (i.e., across messages within a session).
- [ ] When a specialist runtime spawns a child work item (via `MULTI-HERMES-CONTRACT.md` § 5.1 parent_id semantics), the child's processing on its target runtime carries the child `work_item_id`, NOT the parent. The parent is recoverable through `parent_id`; this matches `dev-assist-cli logs --work-item <id> --recursive` (TKT-027).
- [ ] An adapter shim in `src/developer_assistant/observability/omniroute_header.py` injects `X-DEVASSIST-Work-Item-Id: <id>` into outbound LLM calls when the contextvar is set; OmniRoute middleware (TKT-031) reads this header.
- [ ] Tests in `tests/test_structured_logger.py` cover: JSON shape on each log level, contextvar propagation across `asyncio.create_task`, contextvar propagation across `threading.Thread` (using `contextvars.copy_context()`), decorator on sync function, decorator on async function, decorator emits `.fail` on raised exception with the exception class name (not the full traceback) in the log line, no prompt/completion content in the log.
- [ ] Tests in `tests/test_work_item_id_propagation.py` cover: a synthetic four-step pipeline (Orchestrator-create → Architect-claim → Executor-claim → Reviewer-claim) where a single `work_item_id` appears in all four log streams; a parent-child case where parent and child carry distinct `work_item_id`s but child carries the correct `parent_id`.
- [ ] The structured-logger adapter integrates as a Hermes plugin per `HERMES-RUNTIME-CONTRACT.md` § 4 and is enabled by default in each runtime's `agent.json` rendered by TKT-021.
- [ ] No real `OPENCLOWN_DEV_ASSISTANT_PAT`, `TELEGRAM_BOT_TOKEN`, `OMNIROUTE_API_KEY`, `FIREWORKS_API_KEY`, or `OPENROUTER_API_KEY` values appear in tests.
- [ ] `python scripts/validate_docs.py` passes.
- [ ] `python -m unittest discover -s tests -p "test_*.py" -v` passes.

## 8. Risks

- Contextvar propagation across spawned threads requires the caller to use `contextvars.copy_context()`; library code that uses raw `threading.Thread` without this pattern will lose `work_item_id`. Mitigated: the structured logger ships a thin `dispatch_in_thread(fn)` helper that does the right thing, and the runtime adapter uses it for any thread spawn.
- Logging the `model_id` in every line increases log volume; for a high-throughput runtime this is non-trivial. Mitigated: `OBSERVABILITY-CONTRACT.md` FR-OBS-09 retention (`SystemMaxUse=1G, MaxRetentionSec=30d`) bounds disk usage.
- The `instrument_llm_call` decorator depends on the LLM call returning a response object with `tokens_in`/`tokens_out`/`cost_usd` accessible. The OmniRoute SDK shape determines this; the decorator includes a small adapter layer to pull the fields from any of three shapes (OmniRoute response, OpenAI-compat dict, plain dict). Documented in the implementation.
- If a future Hermes version changes the plugin loader contract, the structured-logging plugin breaks. Mitigated: the plugin entry point is small and the structured logger works without the plugin (the plugin only ensures one-time init at startup; without it, the first `get_logger` call performs init lazily).
