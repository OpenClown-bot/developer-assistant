# TKT-025: Shared Custom Skills (`dev-assist-classifier`, `dev-assist-progress-report`, `dev-assist-escalation-surface`) and Allowlist Additions

## 1. Scope

Implement the three custom Hermes skills loaded by the Orchestrator runtime, and update `HERMES-SKILL-ALLOWLIST.md` to reflect them as project-local custom skills.

**Bundle-size disposition (RV-SPEC-013 § 3.1):** This ticket bundles three skills + one allowlist update. The Reviewer flagged this as potentially too large for one Executor session. The Architect's disposition: keep as one ticket, with explicit checkpoint discipline for the Executor (see § 6 Test/Validation Requirements). Rationale: (a) the three skills share the Russian-localized message scaffolding and locale loader; splitting would duplicate the loader plumbing across three PRs. (b) Each skill is small (one tool surface, one cron driver, ~50-100 LOC). (c) The Executor may, at its discretion, split implementation into three sequential commits within one PR (one per skill + one for the allowlist update). The AC remain identical regardless of commit shape. (d) If during execution the Executor finds the work cannot fit one session, it surfaces an escalation and the Architect will issue TKT-025a/b/c follow-up tickets. v0.1 default is one ticket.

1. **`dev-assist-classifier`** (per `MULTI-HERMES-CONTRACT.md` § 5.1, `UPSTREAM-ADAPTER-CONTRACT.md` § 9):
   - Classifies an inbound Telegram message into one of: `intake`, `progress_query`, `command`, `freeform_chat`, `escalation_response`.
   - Returns the classification plus an extracted intent payload (e.g., a parsed `/approve <id>`, a project key, a free-form question).
   - Calls the Orchestrator runtime's main LLM (per `MODEL-CATALOG.md` v0.1.1 § 4.1: `accounts/fireworks/models/minimax-m2p7` via OmniRoute).

2. **`dev-assist-progress-report`** (per `MULTI-HERMES-CONTRACT.md` § 5.1, `MULTI-HERMES-CONTRACT.md` § 6 work_items kind `report_progress`):
   - Driven by the Orchestrator's `cronjob` skill firing every 30 minutes.
   - Iterates `project_bindings` rows where the binding is active.
   - For each project, queries `scheduled_progress` (TKT-019 helpers) to determine if a report is due.
   - For each due project, gathers PR status from GitHub (via the existing `github-pr-workflow` skill, configured for read-only operations only), formats a Russian-localized progress report, and dispatches an outbound message via the upstream router (TKT-024).
   - Marks the report sent via `mark_report_sent` (TKT-019).

3. **`dev-assist-escalation-surface`** (per `MULTI-HERMES-CONTRACT.md` § 5.1, `ESCALATION-POLICY.md` § 7):
   - Driven by the Orchestrator's `cronjob` skill firing every 5 seconds.
   - Calls `read_pending_escalations` (TKT-022) to retrieve up to N pending escalations.
   - For each, formats the Russian-localized prompt (per the format already specified in TKT-024's adapter implementation), calls `outbound_approval_prompt` via the upstream router, and on success calls `mark_escalation_surfaced` with the returned upstream message id.
   - Periodically (every 60 seconds) calls `expire_old_escalations` to advance escalations past the 7-day cutoff to `expired` status.

The three skills are NOT loaded by the four specialist runtimes; they are Orchestrator-only.

## 4. Acceptance Criteria

### 4.1 `dev-assist-classifier`

- [ ] Skill source under `src/developer_assistant/hermes_skills/dev_assist_classifier/`.
- [ ] Skill manifest declares Hermes skill contract version, exposes a `classify(text: str, founder_id: str, current_session_state: dict) -> dict` tool.
- [ ] The output is a structured dict with at least `kind` (one of `intake`, `progress_query`, `command`, `freeform_chat`, `escalation_response`) and `intent` (a kind-specific payload).
- [ ] For `escalation_response` kind, the intent payload extracts the escalation id from `/approve <id>`, `/deny <id>`, or a free-text reply tied to the most recent surfaced escalation.
- [ ] For `command` kind, the intent payload identifies the command (`/start`, `/help`, `/status`, `/projects`, `/pause`, `/resume`).
- [ ] Skill calls the Orchestrator's main LLM via the runtime's tool dispatcher (mocked in tests); does NOT call provider SDKs directly. The runtime's tool dispatcher routes via OmniRoute on the harmonized port (PR-E ADR-011); the skill does NOT hardcode `accounts/fireworks/models/minimax-m2p7` and does NOT bypass OmniRoute.
- [ ] Skill is loaded only by the Orchestrator runtime config (`MULTI-HERMES-CONTRACT.md` § 5.1).
- [ ] Tests in `tests/test_classifier_skill.py` cover: each `kind` with a representative input, fallback to `freeform_chat` on ambiguous input, schema validation of the LLM response (rejects malformed responses), `escalation_response` with the most-recent-surfaced escalation lookup against `read_pending_escalations`.

### 4.2 `dev-assist-progress-report`

- [ ] Skill source under `src/developer_assistant/hermes_skills/dev_assist_progress_report/`.
- [ ] Skill manifest declares a tool `tick(now_iso: str) -> dict` invoked by the cronjob.
- [ ] `tick` iterates active `project_bindings`, calls `is_report_due` (TKT-019) for each, gathers PR status via the bundled `github-pr-workflow` skill (mocked in tests), formats a Russian-localized progress report, dispatches via the upstream router (TKT-024), and on success calls `mark_report_sent`.
- [ ] The progress report includes: project name, current PR list with statuses, last completed work item summary, any blocked work items, optional "no progress" affirmation if the project is idle.
- [ ] If GitHub is unreachable, the skill skips that project's report this tick (rather than failing or partial-reporting); structured log entry recorded.
- [ ] Cronjob frequency: every 30 minutes; default interval per project is 60 minutes per `ARCH-001.md` § 7 and TKT-019.
- [ ] Tests in `tests/test_progress_report_skill.py` cover: report due → message dispatched → mark_report_sent called; report not due → skipped; GitHub unreachable → skipped with log entry; first-time-due (no prior row) → message dispatched; multiple projects → independent dispatch per project.

## 8. Risks

- The classifier's accuracy on Russian-language messages is empirical; the LLM is `accounts/fireworks/models/minimax-m2p7` (Fireworks-hosted, multilingual; the Orchestrator's catalog main model per `MODEL-CATALOG.md` v0.1.1 § 4.1). If accuracy is unacceptable, a follow-up ticket adds a Russian-tuned classifier prompt or asks the Founder to reassign the Orchestrator role. Out of v0.1 scope.
- The progress-report cadence (30-minute cron, 60-minute per-project default) may produce too many or too few messages depending on project velocity. Tunable per project via `scheduled_progress.interval_minutes`; v0.1 default is acceptable.
- The escalation surface 5-second cadence is aggressive but necessary so the Founder sees prompts promptly. SQLite read load is negligible at this rate (single-row read with index).
- The `github-pr-workflow` skill is BLOCKED in production per `HERMES-SKILL-ALLOWLIST.md` § 4 with `production GITHUB_TOKEN`; for the progress-report use case, this skill is configured for read-only PR queries and does NOT use a token capable of write operations. The runtime-check helper (TKT-021) verifies the configured GitHub credential is read-only-scoped at startup; if a write-capable credential is provided, the runtime fails fast.
- The Russian-localized templates may need wording iteration based on Founder feedback. The locale-file separation makes this a low-risk change.
- Bundling three skills + allowlist amendment in one ticket carries a Reviewer-flagged risk of work-item over-sizing. Mitigated by the per-skill checkpoint discipline in § 6 and by the explicit fallback to TKT-025a/b/c if the Executor cannot fit the work in one session.
