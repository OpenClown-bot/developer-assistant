# TKT-017: Verify Live Telegram And GitHub Smoke Readiness For TKT-011

## 1. Scope

Create and run a sanitized live-smoke readiness check that proves the completed Telegram and GitHub runtime bindings are safe enough to unblock a later `TKT-011` end-to-end Telegram-to-PR orchestration trial.

This ticket covers two live evidence lanes:

- Live Telegram/Hermes gateway smoke readiness for the TKT-015 transport boundary.
- Live GitHub REST/`git` smoke readiness for the TKT-008/TKT-014/TKT-016 GitHub integration path, promoted from `TKT-NEW-008-B`.

The intended outcome is a small, sanitized evidence artifact or ticket Execution Log entry that records whether the runtime environment can safely perform the prerequisite live interactions without leaking secrets or private identifiers. If live credentials or runtime access are unavailable, the ticket must fail closed by documenting the blocker rather than running `TKT-011`.

## 2. Non-scope

- Do not execute `TKT-011` or any full Telegram-to-PR orchestration trial.
- Do not implement new GitHub workflow behavior beyond a minimal smoke path over existing TKT-008/TKT-014/TKT-016 APIs.
- Do not implement idempotent branch/PR retry behavior from `TKT-NEW-008-A`; use a uniquely named smoke branch/PR and document cleanup instead.
- Do not implement GitHub integration state persistence from `TKT-NEW-008-C`.
- Do not rework Telegram adapter or transport semantics from TKT-006/TKT-015 except for a narrowly scoped smoke harness or manual-smoke instructions.
- Do not enable Hermes bundled `github-pr-workflow`, `github-issues`, or `github-auth` with production `GITHUB_TOKEN` / `GH_TOKEN` credentials.
- Do not use `GITHUB_TOKEN`, `GH_TOKEN`, `~/.git-credentials`, token-bearing remotes, committed config, or command-line token arguments for GitHub authentication.
- Do not commit `.env`, `config.yaml` containing secrets, Telegram bot tokens, GitHub PATs, LLM API keys, VPS SSH keys, raw Telegram chat IDs, raw Telegram user IDs, private runtime config, token-bearing remotes, or sensitive VPS details.
- Do not open public endpoints, switch to webhook mode, change credential scopes, rotate credentials, perform live VPS deployment, or spend money.
- Do not merge anything automatically; founder acknowledgement remains required before every merge in v0.1.
- Do not introduce OpenClaw, marketplace/community skills, project-local plugins, or new credential-bearing Hermes skills.

## 4. Acceptance Criteria

- [ ] A smoke readiness entry point or documented manual procedure exists for GitHub live smoke testing and is disabled by default unless an explicit non-secret gate is set.
- [ ] GitHub smoke uses `PROJECT_GITHUB_PAT` only through the existing TKT-014/TKT-016 credential path and rejects or ignores `GITHUB_TOKEN`, `GH_TOKEN`, `~/.git-credentials`, token-bearing remotes, committed config, and CLI token arguments.
- [ ] GitHub smoke performs the smallest safe live sequence needed to prove runtime behavior: repository registration or explicit use of the current repository, unique smoke branch creation, sanitized commit/push or equivalent constrained `git` transport step if needed, PR open or draft PR open linked to this ticket, check-status read, PR metadata read, and documented cleanup expectations.
- [ ] GitHub smoke uses TKT-014 constrained `GitCommand` builders and TKT-016 runtime executors; no raw shell command strings, force push, hard reset, branch deletion, token-bearing remotes, or bypass of command validation is allowed.
- [ ] Telegram smoke verifies live Hermes Telegram gateway readiness through the TKT-015 transport boundary using sanitized command names and sanitized chat/user labels for `/status`, `/decisions`, `/pause`, `/resume`, progress delivery, and at least one free-form classification path.
- [ ] Telegram smoke enforces TKT-012/TKT-015 constraints: founder allowlist or DM pairing is configured, allow-all modes remain disabled, polling is preferred, webhook mode is not introduced unless already safely configured with `TELEGRAM_WEBHOOK_SECRET`, and `TELEGRAM_BOT_TOKEN` is read only from approved runtime secret storage.
- [ ] The smoke procedure records only sanitized evidence: command names, sanitized chat/user labels, sanitized branch name, sanitized repository/PR URLs, check status conclusion, PR metadata fields that contain no secrets, timestamps, and pass/block/fail outcome.
- [ ] Evidence explicitly states whether each lane was `pass`, `blocked`, or `fail`; `blocked` is required when credentials, Hermes gateway access, GitHub permissions, or runtime environment are unavailable.
- [ ] No token value, `.env` content, private runtime config, raw Telegram chat ID, raw Telegram user ID, PAT/API key, credential file path contents, token-bearing remote, or sensitive VPS detail appears in committed artifacts, logs, test fixtures, exceptions, or smoke evidence.
- [ ] The smoke path does not enable Hermes bundled GitHub credential-bearing skills, marketplace skills, project-local plugins, OpenClaw plugins, autonomous merge, or live deployment.
- [ ] The smoke path does not merge the smoke PR. If a PR is opened, it remains unmerged until normal CI, Reviewer, and founder gates apply, or it is closed manually with sanitized evidence.
- [ ] If live smoke passes, the resulting evidence clearly states that `TKT-011` may be considered for a separate readiness promotion/revision or execution attempt; this ticket itself does not change `TKT-011` status.
- [ ] If live smoke blocks or fails, the resulting evidence records the blocker and recommended follow-up before any `TKT-011` attempt.
- [ ] `python scripts/validate_docs.py` passes.
- [ ] `python -m unittest discover -s tests -p "test_*.py" -v` passes if any code or tests are changed.

## 8. Risks

- Live smoke can leak secrets through HTTP errors, Telegram payloads, command output, git remotes, logs, exceptions, screenshots, or copied terminal output if sanitization is incomplete.
- GitHub smoke can leave behind branches or PRs if interrupted; cleanup must be documented and manual deletion must avoid destructive commands unless explicitly approved.
- TKT-008 branch/PR creation is not retry-idempotent until `TKT-NEW-008-A`; smoke must use a unique branch name and avoid unattended retries.
- GitHub integration state remains in memory until `TKT-NEW-008-C`; smoke evidence must not imply production continuity after process restart.
- Hermes Telegram gateway payload shape may differ from the mocked TKT-015 payload assumptions and may require a later adapter shim.
- Telegram live smoke requires real runtime credentials and founder allowlist/pairing; if unavailable, the safe result is `blocked`, not a bypass.
- GitHub PAT scopes may be insufficient or too broad; insufficient scope blocks smoke, while overbroad scope should be treated as a risk that requires founder/operator correction outside repository artifacts.
- Opening a live PR can trigger CI and PR-Agent; this is expected, but merge remains prohibited without normal gates and founder acknowledgement.
