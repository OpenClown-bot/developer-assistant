# TKT-014: Implement Project-Specific GitHub Workflow Capability

## 1. Scope

Implement the project-specific GitHub workflow capability required before TKT-008 can build the full Hermes-orchestrated repository and PR integration.

The capability should provide a minimal, reviewed path for repository-scoped GitHub operations using REST API calls plus a constrained `git` orchestration wrapper. It replaces production use of Hermes bundled `github-pr-workflow`, `github-issues`, and `github-auth`, which failed TKT-012 source review for credential-bearing use.

## 2. Non-scope

- Do not implement the full TKT-008 end-to-end project creation and PR orchestration flow.
- Do not enable Hermes bundled GitHub skills with production `GITHUB_TOKEN` or `GH_TOKEN`.
- Do not implement autonomous merges.
- Do not store GitHub tokens in repository files, git remotes, command logs, or plaintext credential-store examples.
- Do not implement GitHub App installation flow unless it is strictly smaller than fine-grained PAT support; fine-grained PAT remains the v0.1 baseline.
- Do not modify production GitHub Actions workflows unless a test fixture or template is explicitly required.

## 4. Acceptance Criteria

- [ ] A project-specific GitHub workflow module or wrapper exists for the minimal operations TKT-008 needs: repository create/register, branch create, commit/push, PR open/update, check-status read, and PR metadata read.
- [ ] The capability uses GitHub REST API calls for API operations and constrained `git` commands for repository transport; it does not depend on Hermes bundled GitHub skills.
- [ ] The wrapper accepts credentials only from approved runtime secret mechanisms such as `GITHUB_TOKEN` or `GH_TOKEN` environment variables, never from `~/.git-credentials`, token-in-remote URLs, committed config, or command-line arguments.
- [ ] The capability documents required token scopes: repository-scoped `contents:write`, `pull_requests:write`, `checks:read`, `statuses:write`, `actions:read`, and `issues:write` only if issue support is implemented.
- [ ] Merge operations require an explicit founder-acknowledgement input and are disabled by default for v0.1.
- [ ] Dangerous or prohibited Git operations are blocked or omitted, including force push, hard reset, branch deletion, and token-bearing remote URLs.
- [ ] Tests cover credential-source rejection, token redaction in logs/errors, REST API request construction with mocked GitHub responses, constrained git command construction, and merge-gate behavior.
- [ ] `python scripts/validate_docs.py` passes.
- [ ] `python -m unittest discover -s tests -p "test_*.py" -v` passes.

## 7. PR Requirements

- Link this ticket.
- Include validation and test output.
- Document required GitHub token scopes and the approved credential source.
- State that Hermes bundled GitHub skills remain blocked for production credential-bearing use.
- Include risk notes for any live-test limitations or unimplemented GitHub operations deferred to TKT-008.

## 8. Risks

- A custom wrapper can still mishandle credentials if token redaction and credential-source checks are incomplete.
- Git command construction can become dangerous if untrusted branch names, remotes, or file paths are not constrained.
- Mocked GitHub tests may miss live API edge cases; TKT-008 should still include an optional sanitized smoke test if credentials and repository access are available.
- Merge support is high risk and must remain gated by founder acknowledgement in v0.1.

## 9. Dependencies

- TKT-009 must be done before credential-bearing GitHub automation is implemented.
- TKT-012 must be done and is the reason this project-specific fallback exists.
- TKT-008 must remain draft until this ticket is done.
- User-approved baseline: GitHub credentials use a fine-grained PAT first, with GitHub App deferred until later.
