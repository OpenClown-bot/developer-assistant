# TKT-008: Implement GitHub Repository and PR Integration

## 1. Scope

Implement the GitHub integration required for Hermes-orchestrated project creation, branch/PR workflow, CI observation, and review gate tracking by wiring the project-specific GitHub workflow capability delivered by TKT-014 into the runtime/orchestration path.

TKT-012 rejected Hermes bundled GitHub credential-bearing skills (`github-pr-workflow`, `github-issues`, `github-auth`) for production `GITHUB_TOKEN` / `GH_TOKEN` use. TKT-014 is complete and provides the reviewed project-specific REST API + constrained `git` workflow capability that this ticket must use.

## 2. Non-scope

- Do not implement autonomous merges.
- Do not store GitHub tokens in repository files.
- Do not implement deployment to VPS.
- Do not bypass ticket, CI, or Reviewer gates.
- Do not use Hermes bundled `github-pr-workflow`, `github-issues`, or `github-auth` with production GitHub credentials.
- Do not reimplement the low-level GitHub REST request and constrained `git` command builders already delivered by TKT-014 unless a narrowly scoped gap is discovered and documented in the Execution Log.

## 4. Acceptance Criteria

- [ ] The integration can create or register a GitHub repository for a project.
- [ ] The integration can create branches and open PRs linked to one ticket.
- [ ] The integration can read CI/check status for a PR.
- [ ] The integration can attach or reference Reviewer artifacts under `docs/reviews/`.
- [ ] The integration requires founder acknowledgement before merge in v0.1.
- [ ] GitHub credentials are scoped and supplied through the TKT-014-approved `PROJECT_GITHUB_PAT` runtime secret, not committed files, `GITHUB_TOKEN` / `GH_TOKEN` fallback, `~/.git-credentials`, token-bearing remotes, or CLI arguments.
- [ ] The integration uses the project-specific GitHub workflow capability from TKT-014 as the primary GitHub operation path and does not enable Hermes bundled GitHub skills rejected by TKT-012.
- [ ] The integration composes GitHub PR state with the TKT-006 Telegram founder interaction logic so founder-facing status/progress/approval messages can reference repository, PR, CI, and review-gate state without treating Telegram chat history as authoritative.
- [ ] `python scripts/validate_docs.py` passes.

## 8. Risks

- Overbroad GitHub tokens can expose all assistant-owned repositories.
- PR automation can bypass governance if ticket links and review gates are not enforced.
- Live API tests may be flaky or unavailable in local environments.
- Hermes bundled GitHub skills failed TKT-012 source review for production credential-bearing use; enabling them would violate ADR-003 controls and v0.1 founder-acknowledgement gates.
- The project-specific GitHub workflow from TKT-014 is reviewed and tested, but TKT-008 can still introduce policy bypasses if it does not use `load_credential()` as the primary credential entry point, does not preserve token redaction, or circumvents constrained command builders.
- TKT-006 currently provides a logic-layer Telegram adapter, not live Hermes/Telegram transport; TKT-008 should integrate with that logic contract without assuming live gateway wiring is already complete.

## 9. Dependencies

- TKT-002 and TKT-004 should be complete.
- TKT-005 must be done and must define how Hermes runtime work references repository governance artifacts.
- TKT-009 must be done before enabling credential-bearing GitHub automation.
- TKT-012 is done and blocks Hermes bundled GitHub credential-bearing skills from production use.
- TKT-014 is done and provides the reviewed project-specific REST API + constrained `git` workflow capability used by this ticket.
- TKT-006 is done and provides the Telegram founder interaction logic layer used for founder-facing status, progress, and approval messages.
- User-approved baseline: GitHub credentials use a fine-grained PAT first, with GitHub App deferred until later.
