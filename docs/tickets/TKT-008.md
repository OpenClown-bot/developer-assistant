# TKT-008: Implement GitHub Repository and PR Integration

## 1. Scope

Implement the GitHub integration required for Hermes-orchestrated project creation, branch/PR workflow, CI observation, and review gate tracking.

This ticket is not ready until a project-specific GitHub workflow capability exists. TKT-012 rejected Hermes bundled GitHub credential-bearing skills (`github-pr-workflow`, `github-issues`, `github-auth`) for production `GITHUB_TOKEN` / `GH_TOKEN` use, so this ticket must not implement production GitHub automation by enabling those bundled skills.

## 2. Non-scope

- Do not implement autonomous merges.
- Do not store GitHub tokens in repository files.
- Do not implement deployment to VPS.
- Do not bypass ticket, CI, or Reviewer gates.
- Do not use Hermes bundled `github-pr-workflow`, `github-issues`, or `github-auth` with production GitHub credentials.

## 4. Acceptance Criteria

- [ ] The integration can create or register a GitHub repository for a project.
- [ ] The integration can create branches and open PRs linked to one ticket.
- [ ] The integration can read CI/check status for a PR.
- [ ] The integration can attach or reference Reviewer artifacts under `docs/reviews/`.
- [ ] The integration requires founder acknowledgement before merge in v0.1.
- [ ] GitHub credentials are scoped and supplied through approved secret mechanisms, not committed files.
- [ ] The integration uses the project-specific GitHub workflow capability from TKT-014, not Hermes bundled GitHub skills rejected by TKT-012.
- [ ] `python scripts/validate_docs.py` passes.

## 5. Allowed Files

## 6. Test/Validation Requirements

- Run `python scripts/validate_docs.py`.
- Add or run tests with mocked GitHub API behavior where feasible.
- If a live GitHub smoke test is run, document repository/PR URLs without printing secrets.

## 7. PR Requirements

- Link this ticket.
- Include validation and test output.
- Document required GitHub token scopes.
- Document any live-test limitations.

## 8. Risks

- Overbroad GitHub tokens can expose all assistant-owned repositories.
- PR automation can bypass governance if ticket links and review gates are not enforced.
- Live API tests may be flaky or unavailable in local environments.
- Hermes bundled GitHub skills failed TKT-012 source review for production credential-bearing use; enabling them would violate ADR-003 controls and v0.1 founder-acknowledgement gates.
- A custom GitHub wrapper can still mishandle tokens or bypass policy if its command allowlist, REST calls, and merge gates are not tested before this ticket.

## 9. Dependencies

- TKT-002 and TKT-004 should be complete.
- TKT-005 must be done and must define how Hermes runtime work references repository governance artifacts.
- TKT-009 must be done before enabling credential-bearing GitHub automation.
- TKT-012 is done and blocks Hermes bundled GitHub credential-bearing skills from production use.
- TKT-014 must be done before this ticket can be promoted to ready; it must provide the reviewed project-specific REST API + `git` orchestration wrapper used by this ticket.
- User-approved baseline: GitHub credentials use a fine-grained PAT first, with GitHub App deferred until later.
