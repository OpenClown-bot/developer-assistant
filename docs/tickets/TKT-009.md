# TKT-009: Implement Hermes Skill and Plugin Security Allowlist

## 1. Scope

Implement and document the v0.1 allowlist and enforcement approach for Hermes skills/plugins used by Telegram, GitHub, coding-agent delegation, and sandboxed execution.

## 2. Non-scope

- Do not approve broad marketplace auto-installation.
- Do not enable OpenClaw plugins.
- Do not commit secrets.
- Do not perform a full third-party security audit.

## 3. Required Context

- `AGENTS.md`
- `CONTRIBUTING.md`
- `docs/architecture/ARCH-001.md`
- `docs/architecture/HERMES-RUNTIME-CONTRACT.md`
- `docs/architecture/adr/ADR-003-plugin-supply-chain.md`
- Hermes Agent runtime documentation used by the implementation team

## 4. Acceptance Criteria

- [x] An allowlist artifact or runtime config lists every enabled Hermes skill/plugin.
- [x] Each entry includes name, source URL, version/commit, purpose, maintenance category, required credentials, permission scope, source review result, sandbox mode, dangerous operations, and rollback procedure.
- [x] Marketplace auto-installation and unreviewed project-local plugins are disabled or documented as unavailable in the selected Hermes deployment mode.
- [x] Credential-bearing skills/plugins use least-privilege credentials.
- [x] Rollback steps are tested or documented for disabling a skill/plugin and revoking credentials.
- [x] `python scripts/validate_docs.py` passes.

## 5. Allowed Files

- Runtime source/config files explicitly established by the approved implementation plan.
- `docs/architecture/` for allowlist documentation if treated as architecture/security policy.
- `tests/`
- `docs/tickets/TKT-009.md` Execution Log only

## 6. Test/Validation Requirements

- Run `python scripts/validate_docs.py`.
- Validate the allowlist/config syntax if a tool is available.
- Manually review that no secret values are included.

## 7. PR Requirements

- Link this ticket.
- Include validation output.
- Include the initial allowlist and risk notes.
- Document any skills/plugins deferred due to review or sandboxing risk.

## 8. Risks

- Hermes capabilities may require enabling more skills than initially expected.
- Source review may be shallow if dependencies are large.
- Rollback may interrupt active agent runs.
- Incomplete allowlist decisions can block TKT-006 and TKT-008 because Telegram and GitHub automation require credential-bearing capabilities.

## 9. Dependencies

- TKT-005 must be done so the allowlist can match the selected Hermes runtime integration contract.
- User-approved baseline: minimal Telegram, GitHub, coding-agent delegation, and sandbox/runtime capabilities only.
- ADR-003 must remain accepted as the supply-chain policy.
