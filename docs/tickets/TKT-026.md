# TKT-026: Model-Catalog Enforcement Helper

## 1. Scope

Implement a small enforcement helper that the install script (TKT-020) and the runtime-check helper (TKT-021) call to verify that each runtime's effective model assignment matches the Founder-pre-approved catalog (`MODEL-CATALOG.md` v0.1.0):

- Module `src/developer_assistant/model_catalog.py` exposing:
  - A typed catalog dataclass populated at import time from `docs/architecture/MODEL-CATALOG.md` (parsed from the YAML frontmatter and the per-role tables) so the catalog has exactly one source of truth.
  - `get_role_assignment(role: str) -> RoleAssignment` returning main + fallback chain.
  - `get_auxiliary_classifier_set() -> list[str]` returning the permitted auxiliary classifier models.
  - `verify_runtime_config(role: str, config: dict) -> None` that raises a structured `CatalogViolation` when the runtime's `agent.model` or `agent.fallback_models` deviates from the catalog.
  - `verify_classifier_model(env: Mapping[str, str]) -> None` that raises when `DEV_ASSIST_ESCALATION_CLASSIFIER_MODEL` is set to a value not in the auxiliary classifier set.
- A small CLI `src/developer_assistant/cli/model_catalog_cli.py` invoked by the install script's preflight (`scripts/install-self.sh` per TKT-020).
- A test that round-trips: parse `MODEL-CATALOG.md` → render runtime configs (TKT-021) → verify each rendered config passes catalog enforcement.

## 2. Non-scope

- Do not implement live LLM calls or live provider connectivity checks (the verify-self script's connectivity check, TKT-020 invariant 5, covers connectivity).
- Do not modify `MODEL-CATALOG.md` content; this ticket only consumes it.
- Do not implement the routing layer (OmniRoute / OpenRouter); routing is out of scope and uses Hermes' built-in mechanism.
- Do not implement model-fallback orchestration; that is Hermes' built-in retry behavior.
- Do not introduce paid third-party dependencies.
- Do not loosen secret handling, credential policy, founder acknowledgement, CI, PR-Agent, Reviewer LLM, or merge gates.

## 4. Acceptance Criteria

- [ ] `src/developer_assistant/model_catalog.py` exists and parses `docs/architecture/MODEL-CATALOG.md` at import time.
- [ ] The parser resolves the per-role assignment table (`MODEL-CATALOG.md` § 4.1) into a `RoleAssignment(main: str, fallbacks: list[str])` per role, with all five roles populated.
- [ ] The parser resolves the auxiliary classifier set (`MODEL-CATALOG.md` § 4.2) into `auxiliary_classifier_default: str` and `auxiliary_classifier_alternatives: list[str]`.
- [ ] The parser is robust to whitespace and surrounding markdown; it rejects gracefully (with a structured error) if the catalog markdown structure is malformed.
- [ ] `get_role_assignment("orchestrator")` returns `RoleAssignment(main="gpt-5.1", fallbacks=["claude-sonnet-4.5", "gemini-2.5-pro", "deepseek-v3.5"])`.
- [ ] `get_role_assignment("architect")` returns `RoleAssignment(main="claude-opus-4.5", fallbacks=["gpt-5.1", "claude-sonnet-4.5", "deepseek-v3.5"])`.
- [ ] `get_role_assignment("planner")`, `get_role_assignment("executor")`, `get_role_assignment("reviewer")` return values matching `MODEL-CATALOG.md` § 4.1.
- [ ] `get_role_assignment("nonexistent_role")` raises a structured `UnknownRole` error with a list of valid role ids in the message.
- [ ] `get_auxiliary_classifier_set()` returns `{"default": "gpt-5.1-mini", "alternatives": ["claude-haiku-4.5", "gemini-2.5-flash"]}`.
- [ ] `verify_runtime_config(role, config)`:
  - raises `CatalogViolation` if `config['agent']['model']` does not equal the role's `main`;
  - raises `CatalogViolation` if `config['agent']['fallback_models']` is not a prefix of (or equal to) the role's `fallbacks` list (a runtime may declare a shorter fallback chain but not a different one);
  - raises `CatalogViolation` if `config['agent']['fallback_models']` contains any model not in the role's `fallbacks` list;
  - succeeds quietly otherwise.
- [ ] `verify_classifier_model(env)`:
  - succeeds if `DEV_ASSIST_ESCALATION_CLASSIFIER_MODEL` is unset (default applies);
  - succeeds if it is set to the default or any alternative;
  - raises `CatalogViolation` if it is set to any other value.
- [ ] CLI `model_catalog_cli.py` provides `verify-runtime --role <role> --config <path>` and `verify-classifier --env-file <path>` subcommands; non-zero exit on violation.
- [ ] Tests in `tests/test_model_catalog.py` cover: parse correctness for all five roles, parse correctness for the auxiliary classifier set, malformed-catalog detection, `get_role_assignment` happy path and unknown role, `verify_runtime_config` happy path and each violation case (wrong main, extra fallback, reordered fallback, fallback that's a prefix of the canonical list (allowed), shorter chain (allowed)), `verify_classifier_model` happy path and rejection of unknown classifier model.
- [ ] Tests in `tests/test_runtime_layout_catalog_round_trip.py` cover: render runtime config for each role (using TKT-021's renderer), call `verify_runtime_config` against the rendered output, assert it passes.
- [ ] No real `OMNIROUTE_API_KEY`, `OPENROUTER_API_KEY`, or LLM API key values appear in tests.
- [ ] `python scripts/validate_docs.py` passes.
- [ ] `python -m unittest discover -s tests -p "test_*.py" -v` passes.

## 8. Risks

- The catalog parser has a hard dependency on the markdown table format used in `MODEL-CATALOG.md` § 4.1 and § 4.2. If a future Architect refactors the catalog markdown shape (e.g., switches to YAML frontmatter for the catalog data), the parser must be updated. Mitigated: the parser tests include synthetic malformed inputs to catch refactor regressions early.
- The "prefix-of-canonical-fallbacks is allowed" rule is a deliberate softening: it allows a runtime to declare a shorter chain (e.g., for cost-sensitive contexts) without escalating. If the Founder prefers strict equality, this rule can be tightened in a follow-up.
- The CLI's `verify-runtime` subcommand parses the runtime config from a path passed by the install script; if the install script renders the config to an unexpected location, the CLI fails clearly. Documented.
- Provider model identifiers ("gpt-5.1", "claude-opus-4.5", etc.) are placeholders; the routing layer (OmniRoute / OpenRouter) resolves them to current provider slugs. The catalog enforcement helper does NOT verify the slugs exist on a provider; that is the job of `verify-self.sh` connectivity invariant 5 (TKT-020).
