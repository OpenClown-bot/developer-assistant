# TKT-026: Model-Catalog Enforcement Helper And OmniRoute Resolution Probe

## 1. Scope

Implement a small enforcement helper that the install script (TKT-020) and the runtime-check helper (TKT-021) call to verify that each runtime's effective model assignment matches the Founder-pre-approved catalog (`MODEL-CATALOG.md` v0.1.1):

- Module `src/developer_assistant/model_catalog.py` exposing:
  - A typed catalog dataclass populated at import time from `docs/architecture/MODEL-CATALOG.md` (parsed from the YAML frontmatter and the per-role tables) so the catalog has exactly one source of truth.
  - `get_role_assignment(role: str) -> RoleAssignment` returning main + fallback chain (real OmniRoute Fireworks identifiers per `MODEL-CATALOG.md` v0.1.1 § 4.1).
  - `verify_runtime_config(role: str, config: dict) -> None` that raises a structured `CatalogViolation` when the runtime's `agent.model` or `agent.fallback_models` deviates from the catalog.
  - `probe_omniroute(omniroute_base_url: str, role: str, timeout_seconds: float = 10.0) -> ProbeResult` that issues a 1-token chat-completion request against `<omniroute_base_url>/v1/chat/completions` for the role's main identifier and (in `--exhaustive` mode) for each fallback. On HTTP non-2xx, on timeout, on connection refused, or on response with `model` field mismatching the requested identifier: returns a `ProbeResult` with `ok=False` and a structured failure reason (one of: `unreachable`, `auth_failure`, `not_resolved`, `timeout`, `unexpected_response`).
  - `paid_third_party_external_service_not_yet_supported_error(role, identifier, reason)` raises the structured escalation rule id from `ESCALATION-POLICY.md` v0.1.1 § 4.6 with the failing role + identifier + reason.
- A small CLI `src/developer_assistant/cli/model_catalog_cli.py` invoked by the install script's preflight (`scripts/install-self.sh` per TKT-020) with two subcommands:
  - `verify-runtime --role <role> --config <path>` (catalog conformance)
  - `probe-omniroute --omniroute-port <port> [--role <role>] [--exhaustive]` (OmniRoute resolution gate; this is the implementation behind the verify-self.sh model-probe invariant per TKT-020 § 4 AC and per ADR-011 binding precondition).
- A test that round-trips: parse `MODEL-CATALOG.md` → render runtime configs (TKT-021) → verify each rendered config passes catalog enforcement.
- A test that exercises `probe_omniroute` against a stub HTTP server returning the five canned responses (success, 401, 404, timeout, mismatched model field).

**Auxiliary classifier removal (RV-SPEC-012 F3):** `MODEL-CATALOG.md` v0.1.1 has NO auxiliary classifier set; the deterministic concept-deviation classifier in `ESCALATION-POLICY.md` v0.1.1 § 5 has no LLM in the decision path. The v0.1.0 functions `get_auxiliary_classifier_set()` and `verify_classifier_model()` are removed in v0.1.1. The advisory-narrative call (TKT-023) uses the runtime's catalog main model and is verified by `verify_runtime_config`.

## 2. Non-scope

- Do not implement the routing layer itself; OmniRoute is a third-party service installed by `install-self.sh` per ADR-011 and TKT-031 (PR-E). This ticket consumes OmniRoute's local OpenAI-compatible HTTP endpoint only.
- Do not modify `MODEL-CATALOG.md` content; this ticket only consumes it.
- Do not implement model-fallback orchestration; that is Hermes' built-in retry behavior over the OmniRoute base URL.
- Do not issue more than 1 token per probe (the probe is a binary reachability gate, not a capability check; `MODEL-CATALOG.md` v0.1.1 cost-posture override per ADDENDUM-001 still applies but the probe is still bounded for hygiene).
- Do not call `api.fireworks.ai` or any non-OmniRoute provider endpoint directly.
- Do not introduce paid third-party dependencies beyond OmniRoute (already approved per ADDENDUM-001 patch; uses Founder's existing Fireworks API key as backend).
- Do not loosen secret handling, credential policy, founder acknowledgement, CI, PR-Agent, Reviewer LLM, or merge gates.

## 4. Acceptance Criteria

- [ ] `src/developer_assistant/model_catalog.py` exists and parses `docs/architecture/MODEL-CATALOG.md` v0.1.1 at import time.
- [ ] The parser resolves the per-role assignment table (`MODEL-CATALOG.md` v0.1.1 § 4.1) into a `RoleAssignment(main: str, fallbacks: list[str])` per role, with all five roles populated.
- [ ] The parser does NOT look for an auxiliary classifier table; in v0.1.1 the catalog has no auxiliary classifier set and `get_auxiliary_classifier_set()` is not exposed. Tests verify that the v0.1.0 function name `get_auxiliary_classifier_set` is NOT present in the public API.
- [ ] The parser is robust to whitespace and surrounding markdown; it rejects gracefully (with a structured error) if the catalog markdown structure is malformed.
- [ ] `get_role_assignment("orchestrator")` returns `RoleAssignment(main="accounts/fireworks/models/minimax-m2p7", fallbacks=["accounts/fireworks/models/kimi-k2p6", "accounts/fireworks/models/qwen3p6-plus", "accounts/fireworks/models/deepseek-v4-pro"])`.
- [ ] `get_role_assignment("planner")` returns `RoleAssignment(main="accounts/fireworks/models/qwen3p6-plus", fallbacks=["accounts/fireworks/models/kimi-k2p6", "accounts/fireworks/models/minimax-m2p7", "accounts/fireworks/models/deepseek-v4-pro"])`.
- [ ] `get_role_assignment("architect")` returns `RoleAssignment(main="accounts/fireworks/models/deepseek-v4-pro", fallbacks=["accounts/fireworks/models/kimi-k2p6", "accounts/fireworks/models/glm-5p1", "accounts/fireworks/models/qwen3p6-plus"])`.
- [ ] `get_role_assignment("executor")` returns `RoleAssignment(main="accounts/fireworks/models/glm-5p1", fallbacks=["accounts/fireworks/models/deepseek-v4-pro", "accounts/fireworks/models/kimi-k2p6", "accounts/fireworks/models/qwen3p6-plus"])`.
- [ ] `get_role_assignment("reviewer")` returns `RoleAssignment(main="accounts/fireworks/models/kimi-k2p6", fallbacks=["accounts/fireworks/models/deepseek-v4-pro", "accounts/fireworks/models/glm-5p1", "accounts/fireworks/models/qwen3p6-plus"])`.
- [ ] `get_role_assignment("nonexistent_role")` raises a structured `UnknownRole` error with a list of valid role ids in the message.
- [ ] `verify_runtime_config(role, config)`:
  - raises `CatalogViolation` if `config['agent']['model']` does not equal the role's `main`;
  - raises `CatalogViolation` if `config['agent']['fallback_models']` is not a prefix of (or equal to) the role's `fallbacks` list (a runtime may declare a shorter fallback chain but not a different one);
  - raises `CatalogViolation` if `config['agent']['fallback_models']` contains any model not in the role's `fallbacks` list;
  - succeeds quietly otherwise.
- [ ] `probe_omniroute(omniroute_base_url, role, timeout_seconds=10.0)`:
  - issues a POST to `<omniroute_base_url>/v1/chat/completions` with `{"model": <role.main>, "messages": [{"role": "user", "content": "ping"}], "max_tokens": 1}`;
  - on 2xx response with `data.model` matching the requested identifier: returns `ProbeResult(ok=True, role=role, identifier=role.main, latency_ms=...)`;
  - on 401/403: returns `ProbeResult(ok=False, reason="auth_failure", ...)`;
  - on 404 or response with `model` mismatch: returns `ProbeResult(ok=False, reason="not_resolved", ...)`;
  - on connection refused: returns `ProbeResult(ok=False, reason="unreachable", ...)`;
  - on timeout: returns `ProbeResult(ok=False, reason="timeout", ...)`;
  - on any other error: returns `ProbeResult(ok=False, reason="unexpected_response", ...)`.
- [ ] CLI `model_catalog_cli.py` provides `verify-runtime --role <role> --config <path>` and `probe-omniroute --omniroute-port <port> [--role <role>] [--exhaustive]` subcommands; non-zero exit on violation. The exit code on `probe-omniroute` failure is `2` and the failure summary reports `role`, `identifier`, and `reason` with no secret values.
- [ ] CLI `probe-omniroute` without `--role` probes all five roles' main identifiers; with `--exhaustive` it probes every (role, identifier) pair across mains and fallbacks (5 roles × 4 entries = 20 probes total).
- [ ] CLI `probe-omniroute` failure raises `paid:third_party_external_service_not_yet_supported` per `ESCALATION-POLICY.md` v0.1.1 § 4.6 with no silent fallback to direct-Fireworks; the install script (TKT-020) propagates the non-zero exit and refuses to complete install.
- [ ] Tests in `tests/test_model_catalog.py` cover: parse correctness for all five roles, malformed-catalog detection, `get_role_assignment` happy path and unknown role, `verify_runtime_config` happy path and each violation case (wrong main, extra fallback, reordered fallback, fallback that's a prefix of the canonical list (allowed), shorter chain (allowed)), assertion that `get_auxiliary_classifier_set` is absent from the public API.
- [ ] Tests in `tests/test_runtime_layout_catalog_round_trip.py` cover: render runtime config for each role (using TKT-021's renderer), call `verify_runtime_config` against the rendered output, assert it passes for all five roles with the real Fireworks identifiers.
- [ ] Tests in `tests/test_probe_omniroute.py` cover: success path (stub returning 200 + matching model), `auth_failure` (401), `not_resolved` (404), `not_resolved` (200 with mismatched model field), `unreachable` (connection refused on closed port), `timeout` (stub sleeps past timeout), `unexpected_response` (200 with malformed JSON).
- [ ] No real `OMNIROUTE_API_KEY`, `OPENROUTER_API_KEY`, `FIREWORKS_API_KEY`, or LLM API key values appear in tests; the probe stub uses a placeholder OmniRoute base URL like `http://127.0.0.1:<test-port>/`.
- [ ] `python scripts/validate_docs.py` passes.

## 8. Risks

- The catalog parser has a hard dependency on the markdown table format used in `MODEL-CATALOG.md` v0.1.1 § 4.1. If a future Architect refactors the catalog markdown shape (e.g., switches to YAML frontmatter for the catalog data), the parser must be updated. Mitigated: the parser tests include synthetic malformed inputs to catch refactor regressions early.
- The "prefix-of-canonical-fallbacks is allowed" rule is a deliberate softening: it allows a runtime to declare a shorter chain without escalating. If the Founder prefers strict equality, this rule can be tightened in a follow-up.
- The CLI's `verify-runtime` subcommand parses the runtime config from a path passed by the install script; if the install script renders the config to an unexpected location, the CLI fails clearly. Documented.
- Provider model identifiers in v0.1.1 are real OmniRoute Fireworks paths (e.g., `accounts/fireworks/models/deepseek-v4-pro`); the routing layer (OmniRoute) resolves them to live Fireworks endpoints. `probe-omniroute` is the live-resolution gate; the binding precondition is recorded in ADR-011. Failure raises `paid:third_party_external_service_not_yet_supported` and the install script halts (no silent fallback to direct-Fireworks).
- The 1-token probe consumes minimal billable usage per OmniRoute call (~5 calls per install at ~1 token each). Per ADDENDUM-001 cost-posture override, this is well within Founder-approved spend.
- A future OmniRoute version may change the `data.model` echo field. Tests cover the `not_resolved` reason for that case; recovery path is updating the parser, not silently passing.
