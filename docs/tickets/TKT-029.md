# TKT-029: Daily Digest And Telegram `/status` Command

## 1. Scope

Implement the daily digest and the Telegram `/status` command, per `OBSERVABILITY-CONTRACT.md` v0.1.0 § 7 (FR-OBS-04) and § 8 (FR-OBS-05):

- A digest renderer module `src/developer_assistant/observability/daily_digest.py` that:
  - Queries `operational.db` for the previous 24 hours of activity (per-runtime work-item counts, escalation counts, error counts, cost aggregation from `llm_calls`).
  - Renders a Markdown summary matching the format in `OBSERVABILITY-CONTRACT.md` § 8.
  - Writes the rendered digest to `/var/log/dev-assist/daily-digest-YYYYMMDD.md` (with `YYYYMMDD` in VPS local time).
  - Sends the digest to the Founder via the Telegram bot (the existing `telegram-gateway` skill on the Orchestrator).
- A Hermes cron entry on the Orchestrator runtime, registered at install time (TKT-020 update — but the cron entry definition itself lives here for the Executor to insert), that runs the digest renderer at 08:00 in the VPS timezone.
- A Telegram `/status` command handler in the Orchestrator's `telegram-gateway` skill that:
  - Receives a `/status` message from the Founder (allowlist-gated).
  - Invokes the equivalent of `dev-assist-cli status --format human` (TKT-027) by importing the CLI's status function and capturing its output (do NOT subprocess-shell out to the CLI binary; share code paths to avoid divergence).
  - Replies with the resulting summary as a Telegram message, paginated if it exceeds 4096 chars.
  - Logs the request and reply per the structured logger (TKT-028).
- Tests in `tests/test_daily_digest.py` covering: digest content for an empty 24-hour window, digest content for a synthetic populated window, digest content when one or more runtimes have errors, idempotent file write (writing the digest twice for the same date overwrites cleanly), Telegram delivery failure path (file is written but delivery is queued for retry).
- Tests in `tests/test_telegram_status_command.py` covering: `/status` handler dispatch, allowlist enforcement (non-allowlisted user gets a generic rejection), pagination boundary, behavior when operational.db is unreachable.

## 8. Risks

- Telegram message length limit (4096 chars) means that for a heavy-activity day the digest can be split into many parts. Mitigated: pagination is at line boundaries; the digest renderer can compress (drop low-signal rows) when it predicts more than 5 parts. Documented as a follow-up in `OBSERVABILITY-CONTRACT.md` § 8 if compression is added later.
- Hermes cron entries run with the runtime's user identity; if the runtime is down at 08:00, the cron does not fire. Mitigated: `RECOVERY-PLAYBOOK.md` § 9 documents the recovery, and a missed digest for one day is acceptable.
- VPS timezone choice affects digest semantics ("yesterday's activity"). Mitigated: the digest header explicitly states the window `[start_iso, end_iso)`. Founder can `timedatectl` to see the VPS timezone.
- `/status` handler shares code with `dev-assist-cli` via `status_query.py`; if a future PR breaks the shared module, both surfaces break together. Documented; the unit tests cover both call sites.
- The `telegram-gateway` skill is on the allowlist as enabled by default for the Orchestrator (`HERMES-SKILL-ALLOWLIST.md` § 4.1). Adding the `/status` handler does not change the skill's allowlist status.
