# TKT-031: `errors` + `llm_calls` + `llm_calls_daily` Tables, Client-Side LLM Instrumentation, Per-Runtime Health Endpoints, ObservabilityManager

## 1. Scope

Implement the data-store and runtime-surface scaffolding for v0.1 observability, per `OBSERVABILITY-CONTRACT.md` v0.1.1 § 9 (FR-OBS-06, errors), § 10 (FR-OBS-07, llm_calls), § 11 (FR-OBS-08, health endpoints), § 12 (FR-OBS-09a/b/c log-retention split), § 14 (ObservabilityManager class spec), and the **client-side LLM instrumentation** (primary) that writes `llm_calls` rows:

- A migration in `src/developer_assistant/state/migrations/` that adds three tables to `operational.db` (in addition to the work_items and escalations tables introduced in TKT-022):
  - `errors` per `OPERATIONAL-STATE-STORE.md` v0.3.0 § 3.5.
  - `llm_calls` per `OPERATIONAL-STATE-STORE.md` v0.3.0 § 3.6.
  - `llm_calls_daily` per `OPERATIONAL-STATE-STORE.md` v0.3.0 § 3.7.
- A small data-access module `src/developer_assistant/state/observability_store.py` exposing typed helpers:
  - `record_error(role, kind, message, work_item_id=None, stack=None, context=None)`.
  - `record_llm_call(role, work_item_id, model_id, routing_path, tokens_in, tokens_out, latency_ms, cost_usd, status, error_class=None)`.
  - `aggregate_llm_calls_daily(target_date)` performing the previous-day aggregation idempotently and returning the row count inserted.
  - `query_errors(since: datetime, until: datetime | None)`, `query_llm_calls(...)`, `query_llm_calls_daily(...)` for the CLI and digest consumers.
- An `ObservabilityManager` class (`src/developer_assistant/observability/observability_manager.py`) per `OBSERVABILITY-CONTRACT.md` v0.1.1 § 14: owns the `observability_store` write path, owns the per-runtime health-endpoint lifecycle, owns the structured-logging context (work_item_id propagation), and exposes the `record_llm_call` / `record_error` calls used by the client-side LLM instrumentation. Lifecycle: instantiated once per Hermes runtime at startup (TKT-028 plugin); shut down gracefully on SIGTERM (drains pending writes, releases health-endpoint port).
- **Client-side LLM instrumentation** (`src/developer_assistant/observability/llm_client_instrumentation.py`) — PRIMARY approach per RV-SPEC-014 M-001 fix. The Python code that issues HTTP requests to OmniRoute (the runtime-side model client) is wrapped with logging/timing instrumentation that calls `ObservabilityManager.record_llm_call(...)` before and after each API call. Implementation: a thin async wrapper around the OmniRoute / OpenRouter HTTP client (httpx-based) that:
  - Reads the `work_item_id` from `ObservabilityManager`'s logger context (set by TKT-028 at work-item dequeue).
  - Reads the `runtime_role` from the `ObservabilityManager` instance (set at startup).
  - Computes `tokens_in` / `tokens_out` / `latency_ms` from the OmniRoute response payload (OmniRoute returns OpenAI-compatible `usage` field) and the wall-clock timing.
  - Computes `cost_usd` from the rate snapshot in `MODEL-CATALOG.md` § 4.1 using the catalog parser from TKT-026.
  - Calls `ObservabilityManager.record_llm_call(...)` with the appropriate fields. On HTTP transport error, calls `record_llm_call(..., status="fail", error_class=...)` with a canonical error class.
  - Requires NO OmniRoute cooperation: the instrumentation is inside the runtime's own process, not inside OmniRoute. This is the implementation guaranteed to work for v0.1 because it does not depend on any OmniRoute extension API.
- **OmniRoute server-side middleware** (`src/developer_assistant/observability/omniroute_middleware.py`) — SECONDARY / OPTIONAL approach. Only implemented if the OmniRoute v3.7.x middleware extension API is verified by reading the OmniRoute source code or documentation; the verified API version/commit hash is recorded in `ADR-011-routing-layer.md` Consequences section before this module is implemented. If the extension API exists, the middleware adds parallel per-request rows from OmniRoute's perspective (useful for cross-checking client-side accounting). If the extension API does NOT exist, this module is omitted from v0.1 entirely; client-side instrumentation is the single source of truth. Decision recorded in this ticket's § 10 Execution Log.
- A localhost-only HTTP listener per Hermes runtime that serves `GET /health` returning the JSON shape in `OBSERVABILITY-CONTRACT.md` § 11. Implementation as a small `aiohttp` (or `http.server` if the runtime runtime already pulls aiohttp via Hermes; coordinate with TKT-021's runtime template) listener bound to `127.0.0.1:HEALTH_PORT` where `HEALTH_PORT` is read from the env var set by the systemd unit (TKT-020 / `SELF-DEPLOYMENT-CONTRACT.md` § 5.2). The listener is owned by `ObservabilityManager` (§ 14 of OBSERVABILITY-CONTRACT) and its lifecycle is bound to the manager's start/stop.
- The cron entries in `/etc/cron.d/dev-assist-observability` (laid down by TKT-020's install script) that invoke `aggregate_llm_calls_daily` and the `errors` retention DELETE; the SQL the cron invokes lives in this ticket. Per FR-OBS-09b (RV-SPEC-014 M-002 split), the cron is unit-testable against a synthetic `operational.db`.
- The journald retention configuration (FR-OBS-09a per RV-SPEC-014 M-002 split): `SystemMaxUse=` and `MaxRetentionSec=` are written to `/etc/systemd/journald.conf.d/dev-assist.conf` by TKT-020's install script. THIS ticket does NOT unit-test journald retention (it is system-level config); the install script's verify step (TKT-020) checks the file exists with the expected key=value pairs (FR-OBS-09c).
- Tests in `tests/test_observability_store.py` covering: schema migration applies idempotently; `record_error` writes correct rows; `record_llm_call` writes correct rows; `aggregate_llm_calls_daily` is idempotent (re-running for the same date does not duplicate rows); query helpers return correctly filtered results.
- Tests in `tests/test_observability_manager.py` covering: manager start/stop lifecycle; logger-context propagation of `work_item_id`; `record_llm_call` / `record_error` proxy to `observability_store`; health-endpoint port acquisition and release; SIGTERM drain semantics.
- Tests in `tests/test_llm_client_instrumentation.py` (PRIMARY observability path) covering: wrapper records the expected llm_call row on success against a stub OmniRoute HTTP server; wrapper records `status=fail` row with `error_class` on transport error; wrapper records `status=fail` with `error_class=provider_5xx` on upstream HTTP 5xx; rate-snapshot lookup via the `MODEL-CATALOG.md` parser from TKT-026; `cost_usd` math.
- Tests in `tests/test_omniroute_middleware.py` (SECONDARY path; runs ONLY if `OBSERVABILITY-CONTRACT.md` v0.1.1 § 14 declares the OmniRoute extension API verified via ADR-011 update) covering: middleware reads the work_item_id and role headers; middleware writes the expected llm_call row on success; middleware writes the expected row with `status=fail` and `error_class` set on failure; rate-snapshot lookup uses the `MODEL-CATALOG.md` parser from TKT-026. If the extension API is not verified, this test file is not created and `omniroute_middleware.py` is not implemented.
- Tests in `tests/test_health_endpoint.py` covering: endpoint binds to localhost and returns `200 OK` with the documented JSON shape; endpoint returns the queue-depth read from the work_items table; endpoint returns the `last_error_at` from the errors table; endpoint refuses non-localhost connections (verified by binding to `127.0.0.1` and asserting the listening socket).

## 4. Acceptance Criteria

### 4.1 Schema migration

- [ ] A new migration file under `src/developer_assistant/state/migrations/` adds the three tables. The migration is versioned and idempotent (re-running does nothing).
- [ ] The schema for `errors` exactly matches `OPERATIONAL-STATE-STORE.md` v0.3.0 § 3.5 (columns: `id`, `ts_iso`, `runtime_role`, `kind`, `message`, `stack`, `work_item_id`, `context_json`, plus indices documented there).
- [ ] The schema for `llm_calls` exactly matches `OPERATIONAL-STATE-STORE.md` v0.3.0 § 3.6.
- [ ] The schema for `llm_calls_daily` exactly matches `OPERATIONAL-STATE-STORE.md` v0.3.0 § 3.7.
- [ ] Schema-version tracking in `_schema_meta` is updated to the new version.
- [ ] A test exists that fresh-migrates an empty `operational.db`, then re-runs the migration, and asserts row count of `_schema_meta` is unchanged and the new tables exist with expected columns.

### 4.2 `observability_store.py`

- [ ] `record_error(role, kind, message, work_item_id=None, stack=None, context=None)` writes one row to `errors` with `ts_iso = utcnow_iso()`, `runtime_role = role`, and `context_json = json.dumps(context, sort_keys=True)`. Returns the inserted `id`.
- [ ] `record_llm_call(role, work_item_id, model_id, routing_path, tokens_in, tokens_out, latency_ms, cost_usd, status, error_class=None)` writes one row to `llm_calls`. Status is one of `success`, `fail`. The `routing_path` is one of `omniroute:<provider>` (e.g., `omniroute:fireworks`, `omniroute:openrouter`) per `MODEL-CATALOG.md` § 5 / `ADR-011`.
- [ ] `aggregate_llm_calls_daily(target_date)` aggregates `llm_calls` rows for the previous UTC day into `llm_calls_daily` keyed by `(date, runtime_role, model_id)`, computes summed `tokens_in`, `tokens_out`, `cost_usd`, and `call_count`, and is idempotent (uses `INSERT OR REPLACE` keyed on `(date, runtime_role, model_id)`).
- [ ] `query_errors(since, until=None, runtime_role=None, kind=None)` returns rows ordered by `ts_iso DESC`, applying optional filters.
- [ ] `query_llm_calls(...)` and `query_llm_calls_daily(...)` provide the equivalent typed query interface for those two tables.
- [ ] All write helpers acquire and release SQLite connections in a way that respects the WAL mode declared in `OPERATIONAL-STATE-STORE.md` § 3 (use `journal_mode=WAL` and `synchronous=NORMAL`).
- [ ] Tests cover happy path, NULL fields, JSON encoding of `context_json`, and idempotency of `aggregate_llm_calls_daily`.

### 4.3 OmniRoute middleware

- [ ] `src/developer_assistant/observability/omniroute_middleware.py` exposes a function that registers as middleware in the OmniRoute service at startup.
- [ ] The middleware reads `X-DEVASSIST-Work-Item-Id` and `X-DEVASSIST-Runtime-Role` from the inbound HTTP request headers; absent headers result in `work_item_id = NULL` and `runtime_role = "unknown"` (the middleware logs a one-time warning per missing-header case).
- [ ] On request completion (response status 2xx), the middleware calls `record_llm_call(role, work_item_id, model_id, routing_path, tokens_in, tokens_out, latency_ms, cost_usd, status="success")`. `model_id` is the resolved Fireworks model slug, normalized to its `MODEL-CATALOG.md` § 4.1 catalog id (e.g., the Fireworks slug `accounts/fireworks/models/qwen3p6-plus` is normalized to the catalog id `qwen3.6-plus`).
- [ ] On request failure (response status 4xx/5xx or transport error), the middleware calls `record_llm_call(...)` with `status="fail"` and `error_class` set to a short canonical string (e.g., `rate_limit`, `auth`, `provider_5xx`, `transport`).
- [ ] `cost_usd` is computed from the rate snapshot in `MODEL-CATALOG.md` § 4.1 using the catalog parser from TKT-026: `cost_usd = (tokens_in * input_rate + tokens_out * output_rate) / 1_000_000`.
- [ ] `routing_path` reflects the actual upstream backend used (`omniroute:fireworks` if Fireworks served the request; `omniroute:openrouter` if the OpenRouter backup served it).
- [ ] Tests cover happy-path success and failure, missing headers, rate-snapshot lookup, and routing-path classification.

### 4.4 Per-runtime health endpoints

- [ ] A module `src/developer_assistant/observability/health_endpoint.py` exposes a `start_health_endpoint(role: str, port: int, operational_db_path: str)` function that starts a background server bound to `127.0.0.1:port`.
- [ ] `GET /health` returns `200 OK` with JSON `{"role": <role>, "ok": true, "queue_depth": <int>, "last_error_at": <iso8601 or null>, "current_work_item_id": <str or null>, "uptime_seconds": <int>}` per `OBSERVABILITY-CONTRACT.md` § 11.
- [ ] `queue_depth` is computed from `SELECT COUNT(*) FROM work_items WHERE status IN ('pending','in_progress') AND target_role = <role>`.
- [ ] `last_error_at` is `SELECT MAX(ts_iso) FROM errors WHERE runtime_role = <role>` (NULL if none).
- [ ] `current_work_item_id` is the runtime's currently-claimed work item, read from a small in-memory state set by the work-queue claim path (TKT-022 update; for now the field returns NULL if the integration is not yet wired).
- [ ] `ok` is `false` if any of the following: operational.db is unreachable; `errors` has a row with `runtime_role=<role>` and `ts_iso > now() - 5 minutes`. Otherwise `true`.
- [ ] The endpoint refuses connections from non-localhost (verified in test by attempting a connection from `0.0.0.0` and asserting refusal).
- [ ] The Hermes runtime startup adapter (TKT-028's plugin) calls `start_health_endpoint` once at runtime init; the endpoint shuts down cleanly on SIGTERM.
- [ ] Tests cover happy path JSON shape, ok=false on stale-error, ok=false on operational.db unreachable, queue_depth correctness against synthetic work_items rows.

## 8. Risks

- The OmniRoute middleware extension contract may change between OmniRoute versions; if the contract changes, this ticket's middleware breaks. Mitigated: the `devassist-omniroute.service` pins a specific OmniRoute version via the install script (TKT-020 update). Upgrades are gated by re-verifying the middleware with the new version's harness.
- Rate-snapshot lookup depends on `MODEL-CATALOG.md` parser from TKT-026; ordering matters. TKT-026 must merge first or this ticket's CI fails.
- The health endpoint adds a small per-runtime memory and CPU footprint (~5-15 MB; one stdlib HTTP server thread). Documented in `MULTI-HERMES-CONTRACT.md` § 11 alongside the existing 1.5-3 GB steady-state estimate.
- The errors-retention cron deletes rows older than 90 days; if the Founder ever needs older error history for forensic purposes, it has been deleted. Mitigated: the daily digest is on-disk indefinitely (with weekly archive compression), so a forensic trail of "what mattered" is preserved even when the row-level data is gone.
- The `aggregate_llm_calls_daily` aggregation is keyed on `(date, runtime_role, model_id)`. If a future change splits a single model into multiple variants, the historical aggregation may collapse them; documented as a v0.2+ refinement opportunity.
- The per-runtime health endpoint binds to a fixed port; if a port collision occurs (another service grabs the port), the endpoint fails to start and the runtime's `/health` is unreachable. Mitigated: install script verifies port availability before bringing up units (TKT-020); if a collision occurs after install, the endpoint logs a clear error and the systemd unit fails to come up rather than silently degrading.
