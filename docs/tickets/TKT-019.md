# TKT-019: Add Progress Report Scheduling Persistence Helper

## 1. Scope

Add a tiny, low-risk progress-scheduling helper that determines whether a progress report is due and records that a report was sent, using the existing `scheduled_progress` table in the SQLite operational state store.

This ticket promotes a narrow subset of `TKT-NEW-006-B`: the TKT-006 progress scheduling logic currently stores last-report timestamps and intervals in memory. This helper provides the persistence-facing read/write interface that a future runtime wiring ticket will consume. The helper does not wire into the live Telegram adapter, Hermes runtime, or any external service.

The implementation must add two pure functions in a new module:

- `is_report_due(db, project_key, now_iso)` — returns `True` when `next_report_at` is not null and `now_iso >= next_report_at`, or when no `scheduled_progress` row exists yet (first report is due). Returns `False` otherwise.
- `mark_report_sent(db, project_key, now_iso)` — updates `last_report_at` to `now_iso` and computes `next_report_at` as `now_iso + interval_minutes` using the stored `interval_minutes`. If `interval_minutes` is null, uses a default of 60 minutes per `ARCH-001` Section 7.

Both functions must accept a `sqlite3.Connection` (as provided by `state_store.open_store`) and a sanitized `project_key`. All fixtures and tests must use sanitized keys such as `chat:proj-alpha`.

This ticket exists to serve as the trial vehicle for `TKT-011@0.2.0` iter-3. It must be safe to implement without live Telegram, GitHub, VPS, or external-service access.

- The `is_report_due` semantics for "no row exists" (first report is due) could surprise callers that expect an explicit scheduling setup step first; the function docstring and tests must document this behavior clearly.
- The default interval of 60 minutes is hardcoded; if `ARCH-001` Section 7 later requires a different default, this will need a follow-up change. This is acceptable for a minimal trial vehicle.
- ISO 8601 timestamp comparison uses simple string comparison, which is correct for UTC ISO 8601 timestamps with consistent formatting; tests must verify this assumption with representative timestamps.
- Future agents could confuse this trial vehicle with the `TKT-011@0.2.0` trial itself; the PR and Execution Log must keep that distinction explicit.
