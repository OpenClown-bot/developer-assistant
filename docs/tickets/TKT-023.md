# TKT-023: Escalation Policy Enforcement Plugin (`dev-assist-escalation-policy`) and Work-Queue Plugin (`dev-assist-work-queue`)

## 1. Scope

Implement the two Hermes plugins that operationalize cross-runtime coordination and Founder escalation:

1. **`dev-assist-escalation-policy`** (per `ESCALATION-POLICY.md` v0.1.0 + ADR-008):
   - Hermes plugin loaded into all five specialist runtimes.
   - Registers a `pre_tool_call` hook that walks the `ESCALATION-POLICY.md` § 3 decision tree.
   - Implements the deterministic rule set in `ESCALATION-POLICY.md` § 4 (rules `gov:*`, `git:*`, `state:*`, `secret:*`, `net:*`, `paid:*`, `deploy:*`, `plugin:*`, `scope:*`, `concept:*`).
   - Implements the LLM classifier in `ESCALATION-POLICY.md` § 5 with the hard-coded prompt, the redaction list, and the latency/cost guards.
   - On match, calls `write_escalation()` (provided by TKT-022), then refuses the tool call with a structured "blocked" response that the runtime can read.

2. **`dev-assist-work-queue`** (per `MULTI-HERMES-CONTRACT.md` § 6 + ADR-006):
   - Hermes plugin loaded into all five specialist runtimes.
   - Registers four runtime-callable tools backed by the helpers from TKT-022:
     - `work_queue.claim` (claims highest-priority pending item for this runtime's role).
     - `work_queue.complete` (marks claimed item completed with a result).
     - `work_queue.release` (returns a claimed item to pending; optional `increment_attempts`).
     - `work_queue.write` (creates a new item; tagged with the originating runtime).
   - The four tools' arguments are validated with strict schemas (no free-form `kwargs`).
   - Each tool's behavior is exactly the one specified in `MULTI-HERMES-CONTRACT.md` § 6.2.

The plugin source goes under `/srv/devassist/shared-plugins/<plugin-name>/` on the deployed VPS, but in the repo lives at `src/developer_assistant/hermes_plugins/<plugin-name>/`. The install script (TKT-020) copies them at install time.

## 4. Acceptance Criteria

### 4.1 `dev-assist-escalation-policy`

- [ ] Plugin source under `src/developer_assistant/hermes_plugins/dev_assist_escalation_policy/`.
- [ ] Plugin manifest declares Hermes plugin contract version, exposes a `register(hooks)` entry point that subscribes to `pre_tool_call`.
- [ ] The deterministic rule set is implemented as a sequence of pure-function rules in a module `rules.py`. Each rule is a function `(action_kind, action_args) -> Optional[str]` returning the rule id (e.g., `git:force_push`) on match or `None` on no-match.
- [ ] All rules in `ESCALATION-POLICY.md` § 4 are present, with rule id matching exactly.
- [ ] Rule registry function returns rules in evaluation order.
- [ ] An LLM classifier module `classifier.py` constructs the immutable prompt from `ESCALATION-POLICY.md` § 5.1, calls the configured auxiliary classifier model from `MODEL-CATALOG.md` § 4.2, parses the JSON verdict, and applies the fail-closed default on error/timeout.
- [ ] Classifier has a 10-second hard timeout and a 5-minute per-runtime cache keyed on `(action_kind, action_args_hash)`.
- [ ] Classifier redacts secret-like values per the regex list shared with `secret:*` rules before sending to the LLM.
- [ ] On match (deterministic rule OR `ASK_FOUNDER` verdict), the hook calls `write_escalation` with the structured row from `ESCALATION-POLICY.md` § 6 step 1 and returns a "blocked" Hermes hook response.
- [ ] On `PROCEED` verdict, the hook returns "allow" and the tool call proceeds.
- [ ] Read-only and trivially safe tool calls (read_file, list_files, session_search, MEMORY.md operations on this runtime's own MEMORY.md) bypass both layers per `ESCALATION-POLICY.md` § 3 step A.
- [ ] Within-catalog model picks bypass the LLM classifier per `ESCALATION-POLICY.md` § 3 step C; the rule engine still runs and the `paid:llm_provider_outside_catalog` rule still fires when the model is outside the catalog.
- [ ] Plugin is configured via env vars only: `DEV_ASSIST_ESCALATION_CLASSIFIER_MODEL`, `DEV_ASSIST_ESCALATION_CLASSIFIER_CACHE_TTL_SECONDS`, `DEV_ASSIST_ESCALATION_CLASSIFIER_TIMEOUT_SECONDS`.
- [ ] Plugin tests in `tests/test_escalation_policy_plugin.py` cover: every deterministic rule's match condition, every deterministic rule's no-match condition, classifier called only when no rule matched, fail-closed on timeout, fail-closed on malformed JSON, fail-closed on rule engine exception, redaction of secret values, cache hit avoids second classifier call within TTL, read-only operations bypass both layers, within-catalog model pick bypasses LLM but `paid:*` rule still fires for outside-catalog.
- [ ] Tests use a fake classifier (Python function injected via dependency-injection) that returns canned verdicts; no real LLM API calls.

### 4.2 `dev-assist-work-queue`

- [ ] Plugin source under `src/developer_assistant/hermes_plugins/dev_assist_work_queue/`.
- [ ] Plugin manifest registers four tools: `work_queue.claim`, `work_queue.complete`, `work_queue.release`, `work_queue.write`.
- [ ] Each tool has a strict JSON Schema input definition: `claim` takes `{role, lease_minutes?}`; `complete` takes `{work_item_id, result}`; `release` takes `{work_item_id, increment_attempts?}`; `write` takes `{target_role, kind, payload, priority?, dedup_key?}`.
- [ ] Each tool's implementation calls the corresponding helper from `state_store.py` / `work_queue.py` (TKT-022) and returns a structured response.
- [ ] `work_queue.claim` retrieves a row only if its `target_role` matches the runtime's `HERMES_DEVASSIST_ROLE`; the role argument must equal the env var or the call fails with a structured error (this prevents accidental cross-role claim).
- [ ] `work_queue.write` rejects a `target_role` value not in the five allowed roles.
- [ ] `work_queue.write` honors `dedup_key` per the helper semantics (returns the existing id if a non-terminal duplicate exists).
- [ ] Plugin tests in `tests/test_work_queue_plugin.py` cover: claim on empty queue returns null, claim with role-mismatch fails, complete moves to `completed`, release returns to `pending`, release with increment_attempts increments the attempt count, write inserts and returns id, write with duplicate dedup_key returns existing id.
- [ ] Tests run against a temp SQLite database; no live runtime.

## 8. Risks

- Hermes' `pre_tool_call` hook contract is documented in source rather than a stable API; if Hermes v2026.4.30 changes the hook signature in a patch, this plugin breaks. Mitigated by pinning Hermes (`PRD-001.md` § 7) and by RESEARCH-001 § 3.8 citing the source location.
- The classifier prompt is hard-coded and immutable to MEMORY.md edits — but a runtime could in theory reload its env vars at runtime to change the classifier model. The `MODEL-CATALOG.md` enforcement helper (TKT-026) catches this at startup; runtime mutation is not addressed in v0.1 (low risk in single-Founder threat model).
- Concurrent `claim_work_item` calls are protected by SQL-level atomicity; the plugin layer adds no concurrency primitives. If two runtimes have the same `HERMES_DEVASSIST_ROLE` (configuration error), they would compete; tests in TKT-021 verify role uniqueness.
- The "fail-closed on classifier timeout" default may produce escalation noise during a provider outage. The Founder can /approve a batch in such a window; longer-term mitigation would be a circuit breaker, deferred to v0.2.
