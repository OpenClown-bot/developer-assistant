# TKT-023: Escalation Policy Enforcement Plugin (`dev-assist-escalation-policy`) and Work-Queue Plugin (`dev-assist-work-queue`)

## 1. Scope

Implement the two Hermes plugins that operationalize cross-runtime coordination and Founder escalation:

1. **`dev-assist-escalation-policy`** (per `ESCALATION-POLICY.md` v0.1.1 + ADR-008 v0.1.1 + `PROJECT-CONCEPT.md` v0.1.0):
   - Hermes plugin loaded into all five specialist runtimes.
   - Registers a `pre_tool_call` hook that walks the `ESCALATION-POLICY.md` v0.1.1 § 3 decision tree.
   - Implements the deterministic rule set in `ESCALATION-POLICY.md` v0.1.1 § 4 (rules `gov:*`, `git:*`, `state:*`, `secret:*`, `net:*`, `paid:*`, `deploy:*`, `plugin:*`, `scope:*`, `concept:*`).
   - Implements the deterministic concept-deviation classifier in `ESCALATION-POLICY.md` v0.1.1 § 5 (pure Python predicate engine; no LLM in the decision path; algorithm fully specified in `PROJECT-CONCEPT.md` v0.1.0 § 3).
   - Optionally invokes the runtime's catalog main model for the **advisory narrative only** (NOT in the decision path; bounded by the timeout/cost ceilings in `ESCALATION-POLICY.md` v0.1.1 § 5.5; advisory failure or timeout never blocks an escalation that the deterministic path already produced).
   - On match (deterministic rule OR concept-deviation classifier verdict), calls `write_escalation()` (provided by TKT-022), then refuses the tool call with a structured "blocked" response that the runtime can read.

2. **`dev-assist-work-queue`** (per `MULTI-HERMES-CONTRACT.md` § 6 + ADR-006):
   - Hermes plugin loaded into all five specialist runtimes.
   - Registers four runtime-callable tools backed by the helpers from TKT-022:
     - `work_queue.claim` (claims highest-priority pending item for this runtime's role).
     - `work_queue.complete` (marks claimed item completed with a result).
     - `work_queue.release` (returns a claimed item to pending; optional `increment_attempts`).
     - `work_queue.write` (creates a new item; tagged with the originating runtime).
   - The four tools' arguments are validated with strict schemas (no free-form `kwargs`).
   - Each tool's behavior is exactly the one specified in `MULTI-HERMES-CONTRACT.md` § 6.2.

The plugin source goes under `/srv/devassist/shared-plugins/<plugin-name>/` on the deployed VPS, but in the repo lives at `src/developer_assistant/hermes_plugins/<plugin-name>/`. The install script (TKT-020) copies them at install time.

## 4. Acceptance Criteria

### 4.1 `dev-assist-escalation-policy`

- [ ] Plugin source under `src/developer_assistant/hermes_plugins/dev_assist_escalation_policy/`.
- [ ] Plugin manifest declares Hermes plugin contract version, exposes a `register(hooks)` entry point that subscribes to `pre_tool_call`.
- [ ] The deterministic rule set is implemented as a sequence of pure-function rules in a module `rules.py`. Each rule is a function `(action_kind, action_args) -> Optional[str]` returning the rule id (e.g., `git:force_push`) on match or `None` on no-match.
- [ ] All rules in `ESCALATION-POLICY.md` v0.1.1 § 4 are present, with rule id matching exactly.
- [ ] Rule registry function returns rules in evaluation order.
- [ ] A deterministic concept-deviation classifier module `concept_classifier.py` loads the structured concept anchor from `docs/architecture/PROJECT-CONCEPT.md` v0.1.0 at startup, parses the YAML block per `PROJECT-CONCEPT.md` v0.1.0 § 2, and exposes a pure function `classify_concept_deviation(action_kind, action_args, concept_anchor) -> Optional[str]` per the algorithm in `PROJECT-CONCEPT.md` v0.1.0 § 3. NO LLM is called in the decision path. NO randomness, no I/O, no network. The function is fully deterministic and unit-testable.
- [ ] Concept-deviation classifier reloads `PROJECT-CONCEPT.md` on `SIGHUP` per `ESCALATION-POLICY.md` v0.1.1 § 5.2; on missing or malformed anchor, fail-closed with `concept:anchor_unavailable` (the anchor file is a fatal-startup precondition; the runtime refuses to start with an unparseable anchor).
- [ ] Concept-deviation classifier wall-clock timeout is 100 ms per call (typical cost: <5 ms on real workloads). Per-call USD cost is 0 (no LLM in decision path).
- [ ] Concept-deviation classifier redacts secret-like values from `action_args` (per the constant regex list shared with `secret:*` rules) before producing any debug or audit output. The redaction list is encoded as a constant Python mapping (NOT loaded from disk, NOT env-configurable). A unit test asserts that no secret env-var value appears in any captured classifier output. (RV-SPEC-014 N-003 fix.)
- [ ] On match (deterministic rule OR concept-deviation classifier returning a non-`None` rule id), the hook calls `write_escalation` with the structured row from `ESCALATION-POLICY.md` v0.1.1 § 6 step 1 and returns a "blocked" Hermes hook response. NO LLM is called in this path.
- [ ] On no match (deterministic rules return `None` AND concept classifier returns `None`), the hook returns "allow" and the tool call proceeds.
- [ ] Optional advisory narrative: the plugin MAY (best-effort) call the runtime's catalog main model (`MODEL-CATALOG.md` v0.1.1 § 4.1; e.g., orchestrator runtime calls `accounts/fireworks/models/minimax-m2p7` via OmniRoute) to produce a Russian-language narrative attached to the escalation row. This call is bounded by `ESCALATION-POLICY.md` v0.1.1 § 5.5 (10s timeout, <0.001 USD per call). Advisory failure or timeout NEVER changes the deterministic decision; the escalation is still written with the deterministic rule id and `narrative=NULL` on advisory failure.
- [ ] Read-only and trivially safe tool calls (read_file, list_files, session_search, MEMORY.md operations on this runtime's own MEMORY.md) bypass both layers per `ESCALATION-POLICY.md` v0.1.1 § 3 step A.
- [ ] Within-catalog model picks bypass the concept-deviation classifier per `ESCALATION-POLICY.md` v0.1.1 § 3 step C; the rule engine still runs and the `paid:llm_provider_outside_catalog` rule still fires when the model is outside the catalog.
- [ ] Plugin is configured via env vars only: `DEV_ASSIST_PROJECT_CONCEPT_PATH` (default: `/srv/devassist/repo/docs/architecture/PROJECT-CONCEPT.md`); `DEV_ASSIST_ADVISORY_NARRATIVE_ENABLED` (default: `true`); `DEV_ASSIST_ADVISORY_NARRATIVE_TIMEOUT_SECONDS` (default: `10`). NOTE: the v0.1.0 envs `DEV_ASSIST_ESCALATION_CLASSIFIER_MODEL`, `_CACHE_TTL_SECONDS`, `_TIMEOUT_SECONDS` are removed because there is no LLM in the decision path; an advisory model is the runtime's catalog main and is NOT independently configurable.
- [ ] Plugin tests in `tests/test_escalation_policy_plugin.py` cover: every deterministic rule's match condition, every deterministic rule's no-match condition, concept-deviation classifier called only when no rule matched, concept-deviation classifier returns `None` for in-scope actions and a rule id for out-of-scope actions per `PROJECT-CONCEPT.md` v0.1.0 § 3, fail-closed on missing anchor file, fail-closed on malformed anchor YAML, fail-closed on rule engine exception, redaction of secret values (one assertion per category in the redaction list, demonstrating no secret value appears in classifier audit output), advisory narrative timeout produces escalation with `narrative=NULL` (does NOT block), advisory narrative success produces escalation with non-empty narrative, read-only operations bypass both layers, within-catalog model pick bypasses concept classifier but `paid:*` rule still fires for outside-catalog.
- [ ] Tests use a fake advisory-narrative LLM dispatcher (Python function injected via dependency-injection) that returns canned narratives or raises `TimeoutError`; the deterministic rule and concept paths use no LLM at all. No real LLM API calls.

### 4.2 `dev-assist-work-queue`

- [ ] Plugin source under `src/developer_assistant/hermes_plugins/dev_assist_work_queue/`.
- [ ] Plugin manifest registers four tools: `work_queue.claim`, `work_queue.complete`, `work_queue.release`, `work_queue.write`.
- [ ] Each tool has a strict JSON Schema input definition: `claim` takes `{role, lease_minutes?}`; `complete` takes `{work_item_id, result}`; `release` takes `{work_item_id, increment_attempts?}`; `write` takes `{target_role, kind, payload, priority?, dedup_key?}`.
- [ ] Each tool's implementation calls the corresponding helper from `state_store.py` / `work_queue.py` (TKT-022) and returns a structured response.
- [ ] `work_queue.claim` retrieves a row only if its `target_role` matches the runtime's `HERMES_DEVASSIST_ROLE`; the role argument must equal the env var or the call fails with a structured error (this prevents accidental cross-role claim).
- [ ] `work_queue.write` rejects a `target_role` value not in the five allowed roles.
- [ ] `work_queue.write` honors `dedup_key` per the helper semantics (returns the existing id if a non-terminal duplicate exists).
- [ ] Plugin tests in `tests/test_work_queue_plugin.py` cover: claim on empty queue returns null, claim with role-mismatch fails, complete moves to `completed`, release returns to `pending`, release with increment_attempts increments the attempt count, write inserts and returns id, write with duplicate dedup_key returns existing id.
- [ ] Tests run against a temp SQLite database; no live runtime.

## 8. Risks

- Hermes' `pre_tool_call` hook contract is documented in source rather than a stable API; if Hermes v2026.4.30 changes the hook signature in a patch, this plugin breaks. Mitigated by pinning Hermes (`PRD-001.md` § 7) and by RESEARCH-001 § 3.8 citing the source location.
- The deterministic concept-deviation classifier loads its anchor from `PROJECT-CONCEPT.md` at startup; a malformed or missing anchor is a fatal-startup precondition. This is intentional per RV-SPEC-012 F3: silent fallback to permissive behavior on anchor failure is unacceptable. Tests cover the fail-closed paths.
- The advisory-narrative call (catalog main model via OmniRoute) is bounded by 10 s / <0.001 USD per `ESCALATION-POLICY.md` v0.1.1 § 5.5; provider latency or cost spikes only delay (or omit) narrative attachment, never the deterministic decision.
- Concurrent `claim_work_item` calls are protected by SQL-level atomicity; the plugin layer adds no concurrency primitives. If two runtimes have the same `HERMES_DEVASSIST_ROLE` (configuration error), they would compete; tests in TKT-021 verify role uniqueness.
- The "fail-closed on classifier exception" default may produce escalation noise during a runtime-internal misconfiguration (e.g., missing PROJECT-CONCEPT.md after a bad deploy). Recovery is operator-driven via `RECOVERY-PLAYBOOK.md` (PR-E).
