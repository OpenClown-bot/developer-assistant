# TKT-021: Multi-Hermes Runtime Layout (per-runtime config rendering, skills loadout, role gating)

## 1. Scope

Implement the per-runtime Hermes configuration rendering, skills loadout discipline, and role-gating logic that operationalize `MULTI-HERMES-CONTRACT.md` v0.1.0:

- A Python module `src/developer_assistant/runtime_layout.py` that, given a role id (`orchestrator`, `planner`, `architect`, `executor`, `reviewer`), produces the rendered `config.yaml`, `auth.json` skeleton, and `SOUL.md` skeleton for that runtime.
- A startup-check helper `src/developer_assistant/runtime_check.py` that each runtime invokes before normal operation. It verifies: (a) `HERMES_DEVASSIST_ROLE` is set to one of the five allowed values; (b) the loaded skills match the per-role expected set in `MULTI-HERMES-CONTRACT.md` § 5; (c) the per-runtime config references `/srv/devassist/state/operational.db` via the symlink (the shared operational store, NOT the per-runtime `~/.hermes/state.db` Hermes native sessions database); (d) the schema version of the operational store matches the version this build expects; (e) for the Orchestrator only, that the Telegram bot token env var is non-empty; for non-Orchestrator runtimes, that the Telegram-gateway skill is NOT loaded.
- A small CLI `src/developer_assistant/cli/runtime_layout_cli.py` that the install script (TKT-020) invokes during the install step to render and write per-runtime files.
- The per-runtime config templates as data files under `etc/runtime-templates/<role>/` consumed by the renderer.

## 8. Risks

- Hermes config schema may evolve between v2026.4.30 and a later version; the templates pin the v2026.4.30 shape. If the Founder upgrades Hermes, the templates may need adjustment.
- The runtime-check helper enforces "Telegram-gateway skill MUST NOT be loaded by non-Orchestrator runtimes" by inspecting the loaded set; if a future Hermes version reports loaded skills differently, the check may need updating.
- Per-role config differences must remain easy to audit. The test that asserts pairwise inequality across roles is the primary defense against silent template drift.
- Per-token cost optimization within the catalog is waived for v0.1 (ADDENDUM-001 cost-posture override; `MODEL-CATALOG.md` v0.1.1 § 8). Runtime templates do not encode any cost-cap directive; the only cost-related guard is the deterministic `paid:llm_provider_outside_catalog` rule (`ESCALATION-POLICY.md` v0.1.1 § 4.6).
- Real Fireworks identifiers are reachable only if OmniRoute resolves them at runtime (binding precondition recorded in ADR-011, lands in PR-E). The runtime-layout templates set the OpenAI-compatible base URL to OmniRoute's localhost endpoint; `verify-self.sh` (TKT-020) invokes TKT-026's CLI to confirm the resolution at install time.
