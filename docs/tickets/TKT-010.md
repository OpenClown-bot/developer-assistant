# TKT-010: Define Generated Project VPS Deployment Contract

## 1. Scope

Define the v0.1 contract that generated projects must expose for one-command deployment readiness to a user-owned VPS.

## 2. Non-scope

- Do not deploy this repository to production.
- Do not perform live deployment to a founder VPS.
- Do not store VPS credentials.
- Do not implement app-specific deployment logic for future generated projects.

## 3. Required Context

- `AGENTS.md`
- `CONTRIBUTING.md`
- `docs/prd/PRD-001.md`
- `docs/architecture/ARCH-001.md`
- Relevant ADRs in `docs/architecture/adr/`

## 4. Acceptance Criteria

- [ ] A deployment contract document states the expected one-command entry point, such as `make deploy` or an equivalent script.
- [ ] The contract states required environment variables and secret categories without values.
- [ ] The contract states founder approval is required before live deployment in v0.1.
- [ ] The contract covers logs, rollback expectation, health check expectation, and handoff notes.
- [ ] The contract explains how generated project tickets should adapt deployment details to their application stack.
- [ ] `python scripts/validate_docs.py` passes.

## 5. Allowed Files

- `docs/architecture/`
- `docs/tickets/TKT-010.md` Execution Log only

## 6. Test/Validation Requirements

- Run `python scripts/validate_docs.py`.
- Manually verify consistency with PRD-001 and ARCH-001.

## 7. PR Requirements

- Link this ticket.
- Include validation output.
- Document unresolved deployment assumptions.

## 8. Risks

- A generic deployment contract may be too vague for real generated projects.
- Overly specific deployment rules may not fit different application stacks.
- Founder VPS environments may vary substantially.

## 9. Dependencies

- User-approved baseline: generated projects expose one-command `make deploy` or equivalent; final live deployment requires founder approval.
