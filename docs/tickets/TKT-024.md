# TKT-024: Upstream-Adapter Scaffolding (registry + Telegram adapter binding + identity & session helpers)

## 1. Scope

Implement the v0.1 upstream-adapter abstraction inside the Orchestrator runtime per `UPSTREAM-ADAPTER-CONTRACT.md` v0.1.0 + ADR-007:

- **Adapter registry** (`src/developer_assistant/upstream_adapters/registry.py`): an in-process Python registry that maps `adapter_id` → adapter implementation; v0.1 has one entry, `telegram`. The registry exposes `get(adapter_id)`, `iter_adapters()`, and a thin `register(adapter_id, impl)` API used at startup.
- **Adapter base contract** (`src/developer_assistant/upstream_adapters/base.py`): an abstract base class declaring the five operations from `UPSTREAM-ADAPTER-CONTRACT.md` § 4: `inbound_message`, `outbound_message`, `outbound_approval_prompt`, `bind_founder_identity`, `get_or_create_session`.
- **Telegram adapter binding** (`src/developer_assistant/upstream_adapters/telegram.py`): the v0.1 Telegram adapter implementation. Translates between Hermes' `telegram-gateway` skill events and the abstract base operations. Calls into `state_store.py` helpers (TKT-022) for `founder_identity_bindings` and `upstream_sessions`.
- **Outbound routing helper** (`src/developer_assistant/upstream_adapters/router.py`): given a Founder id and a message intent, looks up the adapter to use and the session/chat id, then calls `outbound_message` or `outbound_approval_prompt` on the resolved adapter.
- **CLI entry point** (`src/developer_assistant/cli/upstream_adapter_cli.py`): used during development for local testing of the adapter contract without a running Hermes runtime; not exposed in production.

## 2. Non-scope

- Do not implement the v0.2 OpenClaw adapter; v0.1 has only `telegram`.
- Do not implement the `dev-assist-classifier`, `dev-assist-progress-report`, or `dev-assist-escalation-surface` skills (those are TKT-025); they consume this scaffolding.
- Do not implement Telegram webhook mode; v0.1 is polling-only per `HERMES-SKILL-ALLOWLIST.md` § 4.1.
- Do not modify Hermes' bundled `telegram-gateway` skill source; only configure it.
- Do not run any Hermes runtime against a real Telegram bot token. Tests use a fake gateway.
- Do not introduce paid third-party dependencies.
- Do not change the operational state store schema (TKT-022 owns that).

## 4. Acceptance Criteria

- [ ] `UpstreamAdapter` abstract base class in `base.py` declares exactly the five operations (`inbound_message`, `outbound_message`, `outbound_approval_prompt`, `bind_founder_identity`, `get_or_create_session`) with the input/output dataclasses defined in `UPSTREAM-ADAPTER-CONTRACT.md` § 4.
- [ ] Each operation's input is a typed dataclass (Python `@dataclass`) and the output is either a dataclass or a primitive return type. No untyped `dict`s in the public signatures.
- [ ] `registry.register(adapter_id, impl)` rejects duplicate `adapter_id` registrations.
- [ ] `registry.get(adapter_id)` raises a structured error when `adapter_id` is not registered.
- [ ] `registry.iter_adapters()` returns adapters in registration order.
- [ ] `TelegramAdapter` (in `telegram.py`) extends `UpstreamAdapter` and implements all five operations.
- [ ] `TelegramAdapter.inbound_message` looks up the Founder via `lookup_founder_by_upstream_identity('telegram', upstream_user_id)`; if not bound, drops the message and writes a `dropped_inbound` log entry.
- [ ] `TelegramAdapter.inbound_message` looks up or creates the upstream session via `get_or_create_session('telegram', founder_id, upstream_chat_id)` and returns a normalized inbound dataclass.
- [ ] `TelegramAdapter.outbound_message` accepts a normalized message dataclass + a session id, retrieves the chat id from the session, and calls Hermes' `telegram-gateway` skill via the runtime tool dispatcher (mocked in tests).
- [ ] `TelegramAdapter.outbound_approval_prompt` accepts an escalation id and prompt text, formats a Russian-localized message that includes the escalation id, the proposed action, the trigger kind, the recommended default, the impact, and the urgency (per `ESCALATION-POLICY.md` § 7), and sends via `outbound_message`. Returns the upstream message id.
- [ ] `TelegramAdapter.bind_founder_identity` calls `bind_founder_identity` (TKT-022); the operation is invoked only as part of the explicit pairing flow, not during ordinary inbound messages.
- [ ] `TelegramAdapter.get_or_create_session` is the same call wrapper.
- [ ] `router.send_outbound_to_founder(founder_id, intent)` resolves the adapter via the bindings table (in v0.1 there is exactly one binding, `telegram`), looks up the session, and dispatches to `outbound_message` or `outbound_approval_prompt`.
- [ ] All adapter operations log to a structured logger; secret values (bot tokens, user-supplied tokens in commands) are never logged.
- [ ] `register('telegram', TelegramAdapter())` is the only call in the v0.1 startup wiring.
- [ ] Tests in `tests/test_upstream_adapter_registry.py`, `tests/test_telegram_adapter.py`, `tests/test_upstream_router.py` cover: registry duplicate rejection, registry lookup, registry iteration; Telegram adapter with bound identity → inbound succeeds; Telegram adapter with unbound identity → inbound dropped; Telegram outbound calls the mocked Hermes tool with the expected payload; Telegram approval prompt formats the Russian-localized text correctly with the seven required fields; router resolves the bound adapter and calls outbound; router fails clearly when no binding exists.
- [ ] No real `TELEGRAM_BOT_TOKEN`, raw Telegram chat IDs, raw Telegram user IDs, or Founder PII appears in tests. Use sanitized labels (`tg-user-1`, `chat:proj-alpha`).
- [ ] `python scripts/validate_docs.py` passes.
- [ ] `python -m unittest discover -s tests -p "test_*.py" -v` passes.

## 8. Risks

- The Russian-localized approval prompt format is defined in code; if the Founder requests wording changes, follow-ups touch this module. Acceptable.
- The `TelegramAdapter.outbound_message` dispatch through Hermes' `telegram-gateway` skill assumes a particular Hermes tool-dispatch interface; if Hermes' skill API changes between v2026.4.30 and a future version, this binding needs a small update. Mitigated by Hermes version pinning.
- The router currently assumes one adapter per Founder; v0.2 may need multi-channel routing per `PRD-001.md` § 10 Q18, which is out of scope here. Documented as a future change.
- Pairing flow (which call paths can invoke `bind_founder_identity`) is gated by an explicit pairing ticket-not-yet-filed; until then, founders are bound via a one-time admin action. This is acceptable for v0.1.
