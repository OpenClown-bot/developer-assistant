# TKT-022: Work-Queue And Escalations Schema (work_items, escalations, founder_identity_bindings, upstream_sessions)

## 1. Scope

Extend the SQLite operational state store (`/srv/devassist/state/operational.db` per `OPERATIONAL-STATE-STORE.md` v0.2.0; explicitly distinct from the per-runtime `~/.hermes/state.db` Hermes native sessions database) with the four tables required by the multi-Hermes IPC layer (`MULTI-HERMES-CONTRACT.md` § 6) and the upstream-adapter abstraction (`UPSTREAM-ADAPTER-CONTRACT.md` § 4-5), plus matching read/write helpers in `state_store.py`.

New tables:

- `work_items` — the canonical inter-runtime IPC primitive (`MULTI-HERMES-CONTRACT.md` § 6.2).
- `escalations` — pending Founder-facing prompts (`MULTI-HERMES-CONTRACT.md` § 6.3).
- `founder_identity_bindings` — adapter-to-Founder identity mapping (`UPSTREAM-ADAPTER-CONTRACT.md` § 4.4).
- `upstream_sessions` — per-adapter session state (`UPSTREAM-ADAPTER-CONTRACT.md` § 4.5).

New helpers in `src/developer_assistant/state_store.py` (or a sibling `work_queue.py` / `escalations.py` module if the existing file is at a size limit):

- For `work_items`: `write_work_item`, `claim_work_item`, `complete_work_item`, `release_work_item`, `read_work_items_by_role`, `reclaim_expired_leases`.
- For `escalations`: `write_escalation`, `read_pending_escalations`, `mark_escalation_surfaced`, `resolve_escalation`, `expire_old_escalations`.
- For `founder_identity_bindings`: `bind_founder_identity`, `lookup_founder_by_upstream_identity`, `revoke_founder_binding`.
- For `upstream_sessions`: `get_or_create_session`, `update_session_last_message`, `set_session_paused`, `set_session_current_project`.

All functions take a `sqlite3.Connection`, follow the existing module conventions, and operate in WAL mode.

A new schema migration file under `db/migrations/` (or whatever convention `OPERATIONAL-STATE-STORE.md` v0.2.0 uses) creates the tables, indexes, and CHECK constraints exactly as specified in the contracts.

## 4. Acceptance Criteria

- [ ] A new migration file (under whatever convention the existing migrations use) creates `work_items`, `escalations`, `founder_identity_bindings`, `upstream_sessions` with the exact columns, types, defaults, CHECK constraints, and indexes described in `OPERATIONAL-STATE-STORE.md` v0.2.0 (authoritative) and cross-referenced from `MULTI-HERMES-CONTRACT.md` § 6.2 / § 6.3 and `UPSTREAM-ADAPTER-CONTRACT.md` § 4.4 / § 4.5.
- [ ] The migration is idempotent: applying it twice on the same database is a no-op after the first.
- [ ] `schema_version` is bumped to reflect the new migration.
- [ ] `write_work_item(db, target_role, kind, payload, priority=50, dedup_key=None) -> int` inserts a `pending` row, returns the new id, and refuses to insert if a row with the same `dedup_key` already exists in `pending|claimed|failed` status (returns the existing id instead).
- [ ] `claim_work_item(db, claiming_role, lease_minutes=30) -> Optional[dict]` atomically claims the highest-priority `pending` row for the given role using a single `UPDATE ... RETURNING *`; returns `None` if no eligible row exists.
- [ ] `complete_work_item(db, work_item_id, result_dict)` sets `status='completed'`, `completed_at=now`, `result_json=...`.
- [ ] `release_work_item(db, work_item_id, increment_attempts=False)` returns a row to `pending` with `claimed_by_runtime=NULL`, `claimed_at=NULL`, `claim_lease_until=NULL`; optionally increments `attempt_count`.
- [ ] `reclaim_expired_leases(db) -> int` returns claimed rows whose `claim_lease_until < now` to `pending`; returns the count.
- [ ] `write_escalation(db, originating_runtime, originating_work_item_id, trigger_kind, context, proposed_action, options, recommended_default, impact, urgency, durable_artifact_target) -> int` inserts a `pending` escalation, returns the id.
- [ ] `read_pending_escalations(db, limit=10) -> list[dict]` returns escalations in `(status, urgency, id)` order where status is `pending` or `surfaced` and `surfaced_at IS NULL OR surfaced_at < now-5min` (so re-surface gets retried after a 5-minute cooldown).
- [ ] `mark_escalation_surfaced(db, escalation_id, telegram_message_id)` sets `status='surfaced'`, `surfaced_at=now`, `telegram_message_id=...`.
- [ ] `resolve_escalation(db, escalation_id, founder_response, status)` accepts `status='approved'` or `status='denied'`, sets `resolved_at=now`, `founder_response=...`.
- [ ] `expire_old_escalations(db, max_age_days=7) -> int` sets `status='expired'` for escalations older than the cutoff still in `pending` or `surfaced`; returns the count.
- [ ] `bind_founder_identity(db, founder_id, adapter_id, upstream_user_id, display_name=None) -> int` inserts a binding; raises if `(adapter_id, upstream_user_id)` already bound and not revoked.
- [ ] `lookup_founder_by_upstream_identity(db, adapter_id, upstream_user_id) -> Optional[str]` returns `founder_id` if a non-revoked binding exists; otherwise `None`.
- [ ] `revoke_founder_binding(db, binding_id)` sets `revoked_at=now`.
- [ ] `get_or_create_session(db, adapter_id, founder_id, upstream_chat_id) -> dict` returns the existing session row (creating it on first call) and updates `last_message_at`.
- [ ] `update_session_last_message`, `set_session_paused(boolean)`, `set_session_current_project(project_id)` all update only the named columns.
- [ ] All helpers operate over a `sqlite3.Connection` in WAL mode; tests verify WAL is set.
- [ ] All helpers use parameterized SQL; no string concatenation of user-supplied values.
- [ ] Tests in `tests/test_work_queue.py`, `tests/test_escalations.py`, `tests/test_founder_identity.py`, `tests/test_upstream_sessions.py` cover: each helper's happy path, idempotency, dedup-key behavior, claim race-safety (two simulated callers; only one wins), lease expiration and reclaim, escalation expiration sweep, identity binding uniqueness, session creation idempotency.
- [ ] No real `TELEGRAM_BOT_TOKEN`, raw Telegram chat IDs, raw Telegram user IDs, GitHub PAT values, or production hostnames appear in tests. Use sanitized labels (e.g., `chat:proj-alpha`, `tg-user-1`).
- [ ] `python scripts/validate_docs.py` passes.
- [ ] `python -m unittest discover -s tests -p "test_*.py" -v` passes.

## 8. Risks

- The dedup-key "reusable after terminal state" semantics may be confusing; the helper's docstring and tests must document it explicitly.
- The `claim_work_item` atomicity depends on SQLite's `RETURNING *` clause behaving as expected on the targeted SQLite version; the migration must enforce the minimum SQLite version per `OPERATIONAL-STATE-STORE.md` v0.2.0.
- The 5-minute re-surface cooldown for escalations is implementation detail; if the Founder finds it noisy, it can be tuned in a follow-up without schema change.
- The 7-day expiration default for escalations is conservative for v0.1; tunable per `ESCALATION-POLICY.md` § 9.
- The schema-version bump must coordinate with TKT-020 (the install script's preflight) and TKT-021 (the runtime-check helper). Documented in those tickets' dependencies.
