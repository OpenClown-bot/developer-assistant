## 1. Context

TKT-041 § 1.4 (3) proposes a 12th `runtime_check` invariant
`smoke_fixture_token_mismatch` to enforce at runtime that:

- When `/srv/devassist/state/smoke-mode.flag` is present, the
  `TELEGRAM_BOT_TOKEN` env var MUST match
  `^smoke-fixture-token-[a-z0-9]{8}$`.
- When the marker is absent, the token MUST NOT match the fixture
  shape (so a development fixture cannot accidentally ship to
  production).

The proposal explicitly defers folding this into
`src/developer_assistant/runtime_check.py` because that file is the
frozen AUDIT-001 surface (TKT-033 v0.3.0 + TKT-021).

## 2. Current Executor coverage

TKT-041 iter-1 enforces the invariant at TWO non-runtime layers:

1. **Install-time** (`scripts/install-self.sh::assert_smoke_fixture_token_shape`):
   refuses `--smoke-mode` when `TELEGRAM_BOT_TOKEN` does not match the
   fixture shape; refuses to combine with `--rotate-secrets`.
2. **Verify-time** (`scripts/verify-self.sh::check_smoke_mode_mutual_exclusion`):
   when the marker file is present, asserts the token has the fixture
   shape (NOT the production shape) in the rendered env file. Only
   registered when the marker is present so production count stays at
   19/19.

What is NOT covered: the **boot-time runtime gate**. A Hermes runtime
process that is started directly (e.g. via `python -m`) bypassing
`install-self.sh` and `verify-self.sh` will not surface this mismatch
until the smoke-inject endpoint refuses the first inbound request. The
TKT-021 + TKT-033 pattern is to add the check to `runtime_check.py`'s
boot-time invariant list so the systemd unit fails to start (`ExitCode=1`
with a structured `RUNTIME_CHECK_FAILED:smoke_fixture_token_mismatch`
journal line).

## 3. Requested SO action

Dispatch a sibling AUDIT-001-successor TKT to:

1. Amend `src/developer_assistant/runtime_check.py` to add a 12th
   `RUNTIME_CHECK_INVARIANTS` enum entry
   `smoke_fixture_token_mismatch`.
2. Add the corresponding check function that:
   - Reads `DEVASSIST_SMOKE_MODE_MARKER_PATH` (default
     `/srv/devassist/state/smoke-mode.flag`).
   - When the marker is present + `TELEGRAM_BOT_TOKEN` does not match
     `^smoke-fixture-token-[a-z0-9]{8}$`: fail with
     `RUNTIME_CHECK_FAILED:<role>:smoke_fixture_token_mismatch`.
   - When the marker is absent + `TELEGRAM_BOT_TOKEN` matches the
     fixture shape: fail with the same diagnostic.
3. Extend the prompt manifest (TKT-033 pattern) to include the new
   invariant in the canonical list.
4. Add unit tests for the new invariant under
   `tests/test_runtime_check.py` (allowed write zone in the sibling TKT).

The Architect must own the spec change (`runtime_check.py` is frozen for
AUDIT-001) and dispatch the Executor TKT against the 11→12 enum
extension.

## 4. Risk if deferred

A `smoke-fixture-token-<8>`-shaped `TELEGRAM_BOT_TOKEN` could:

- Be exported in a developer's shell and never matched against the
  marker absence, causing a production install attempt to silently
  use a fixture token.
- Leak into journald or operational.db payloads (the AC-9 grep matches
  on `^[0-9]+:[A-Za-z0-9_-]{35,}$`, NOT on the fixture shape — by
  design).

The 12th invariant is the only layer that catches this at runtime boot
(before any inbound smoke-inject request).
