## 1. Context

TKT-041 § 3.2 (the per-role expected `loaded_skills` table) lists the
Reviewer runtime's loaded skills as:

> `{cronjob, memory, dev-assist-reviewer-rubric, dev-assist-review-writer, dev-assist-work-queue-poll}`

`MULTI-HERMES-CONTRACT.md` § 5.5 (the authoritative architecture
contract) lists the Reviewer's Hermes built-in skills as:

> `terminal` (Docker backend; read-only mounts of the project repo), `cronjob`, `memory`

So the contract has `terminal` in Reviewer's loadout; the TKT-041 spec
table does NOT.

## 2. Resolution per TKT-041 § 3.2 closing paragraph

TKT-041 § 3.2 closing paragraph explicitly says:

> The implementer MUST NOT hard-code this table in test fixtures;
> instead, the test parses MULTI-HERMES-CONTRACT.md § 5.1–5.5 at test
> time so a future contract amendment automatically rebases the
> assertion.

Following this directive, the offline harness in
`tests/test_behaviour_smoke.py::TestAC2LoadedSkillsSetEquality` does
NOT assert set equality against the § 3.2 table; instead it parses the
contract at test time and asserts each role's parsed set contains the
canonical anchors (`dev-assist-classifier` only on Orchestrator, every
specialist has `dev-assist-work-queue-poll`, etc.). The contract is
the source of truth.

This means the offline test PASSES regardless of which document is
correct — the parser respects the contract. The discrepancy is a
documentation issue, not a code issue.

## 3. Requested SO action

The Architect (Founder-authorized via a sibling Architect amendment
ticket) should reconcile:

(a) If Reviewer truly needs `terminal` (for read-only Docker mounts so
    it can run static checks per § 5.5 narrative): the TKT-041 § 3.2
    table should be amended to include `terminal` in Reviewer's set.
(b) If Reviewer does NOT need `terminal` (because static checks are
    delegated to CI and the Reviewer only writes RV-CODE/RV-SPEC
    artifacts): the `MULTI-HERMES-CONTRACT.md` § 5.5 entry should be
    amended to remove `terminal`.

Either way, the resolution MUST come from the Architect cycle; the
Executor does not amend frozen architecture or frozen ticket text per
TKT-041 § 5 NOT-list.

## 4. Risk if deferred

Low. The offline test passes (it parses the contract, not the table).
The on-VPS smoke run will assert set equality against the deployed
runtime's actual loaded set — which is determined by Hermes runtime
behavior, not the TKT-041 spec table. If the runtime loads `terminal`
(per the contract) and the spec table says it shouldn't, the smoke
will surface a structured `loaded_skills:reviewer:terminal_unexpected`
diagnostic at run time — but this is a documentation question, not a
runtime correctness question.
