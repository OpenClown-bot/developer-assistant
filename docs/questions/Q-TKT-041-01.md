## 1. Context

TKT-041 § 4 AC-7 requires the Executor to empirically calibrate the
specialist-runtime LLM round-trip ceiling `N2` (the maximum wall-clock
delay between a Planner work-item `claim` and `result` write that the
smoke harness will tolerate before emitting the `planner_result_timeout`
diagnostic). The current ticket value of `N2 = 300 s` is a placeholder —
the spec mandates empirical replacement with `claim_p95 + result_p95 +
flush + slack` derived from ≥ 3 smoke runs against a deployed Planner
runtime.

## 4. Proposed empirical N2

Per AC-7: `N2 = claim_p95 + result_p95 + flush + slack`.

- `flush` = the SQLite WAL checkpoint plus the observability `llm_calls`
  insert flush, both observed under normal load. AUDIT-002 baseline says
  this is on the order of 50 ms. Use `flush = 200 ms` as a conservative
  upper bound.
- `slack` = headroom for OmniRoute provider-switching jitter and cold-
  start tail on the Planner's main model. Use `slack = 5 × p95_jitter`
  with `p95_jitter` derived from the same ≥ 3 runs.

Until measurements exist, the **proposed N2 remains 300 s** (the placeholder
in the spec). The Architect cycle that consumes this Q-TKT may either:

(a) Accept measured N2 ≤ 300 s → no spec amendment needed.
(b) Find measured N2 > 300 s → amend AC-7 to widen N2 in a follow-up
    TKT, and adjust `dev-assist-smoke.sh --timeout-result-s` default.
