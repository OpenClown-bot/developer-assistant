## 1. Context

TKT-041 § 1.4 (4) flags an open dependency on Hermes v2026.4.30 (the
version pinned in `docs/architecture/MULTI-HERMES-CONTRACT.md` § 2.0):
whether a `gateway.telegram.dry_run` runtime override exists that
suppresses outbound message dispatch at the gateway layer.

If such an override exists, the smoke-inject endpoint can confidently
inject synthetic Telegram messages without any chance of dispatching
real outbound traffic (the gateway short-circuits before reaching the
Telegram API). If no such override exists, the smoke must prove
otherwise that:

1. The smoke harness's synthetic message NEVER reaches
   `gateway.telegram.send_message` (i.e., the orchestration loop's
   classifier write path terminates at `work_items.INSERT` and the
   Orchestrator does not subsequently use the gateway to confirm
   receipt).
2. The Telegram fixture token `^smoke-fixture-token-[a-z0-9]{8}$`
   would in any case be rejected by Telegram's BotFather API as
   malformed, so even if the gateway tried to dispatch, the call
   would fail before any real chat is reached.

## 2. Current Executor coverage (defensive layers in lieu of Hermes override)

TKT-041 iter-1 implements three independent defensive layers:

1. **Marker-file gate**: `smoke_inject.py` refuses all POSTs to
   `/smoke/inject-message` when `/srv/devassist/state/smoke-mode.flag`
   is absent. The marker is rendered at `install-self.sh --smoke-mode`
   with mode 0400 owned by `devassist:devassist` — the operator
   privilege required to create it equals the privilege required to
   render the env file with the fixture token.
2. **Fixture token shape**: the smoke-fixture token shape
   `^smoke-fixture-token-[a-z0-9]{8}$` is non-overlapping with
   Telegram's `^[0-9]+:[A-Za-z0-9_-]{35,}$` BotFather shape. Even if
   the dispatch path reached the Telegram API, the request would be
   rejected at the BotFather authentication step (`401 Unauthorized`).
3. **Localhost-only bind**: the inject + test-tool servers bind to
   `127.0.0.1` only and raise `ValueError` if invoked with `0.0.0.0`
   or `::`. The firewall layer in `SELF-DEPLOYMENT-CONTRACT.md § 6.2`
   blocks all inbound external traffic on the admin ports (8186 +
   8281..8285).

## 3. Requested SO action

1. **Verify** with the Hermes v2026.4.30 upstream (NousResearch
   `hermes-agent` repo) whether a `gateway.telegram.dry_run` runtime
   override exists. Likely places to look:
   - The Hermes plugin spec for `telegram-gateway`.
   - The `agent.yaml` schema (under `plugins.telegram-gateway.config`).
   - Hermes runtime CLI flags or env vars (`HERMES_GATEWAY_*`).
2. **If the override exists**: dispatch a sibling Architect amendment
   to `MULTI-HERMES-CONTRACT.md` § 5.1 + `SELF-DEPLOYMENT-CONTRACT.md`
   § 6.1 + `scripts/templates/devassist-orchestrator.service.j2` to
   add `Environment=HERMES_GATEWAY_TELEGRAM_DRY_RUN=1` (or the
   equivalent) when the smoke-mode marker is present.
3. **If the override does NOT exist**: ratify TKT-041's three
   defensive layers (marker file + fixture token shape +
   localhost-only bind) as sufficient. Open a follow-up TKT to
   upstream a `dry_run` override into Hermes v2026.5+ as a defense-in-
   depth measure.

## 4. Risk if deferred

If Hermes routes the smoke-injected message back through
`telegram-gateway.send_message` (e.g., for the "progress report" reply
loop the Orchestrator sends on every classified intake), the gateway
will attempt to dispatch using the fixture token. The token shape will
cause the dispatch to fail at Telegram authentication, but the failed
request will be logged in `llm_calls.error_class` or the gateway's own
journald output. This is detectable but noisy and is not the desired
posture.
