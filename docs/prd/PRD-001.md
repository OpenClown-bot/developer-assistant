# PRD-001: developer-assistant v0.1

## 1. Problem Statement

Building real software with LLM agents is difficult because important project state often lives only in chat history, agent roles are blurred, implementation begins before product and architecture scope is clear, and review gates are inconsistent. This creates scope drift, hidden assumptions, unreproducible decisions, and unsafe handling of credentials or repository access.

`developer-assistant` v0.1 should solve this by acting as an autonomous development orchestrator for a non-engineer founder. The founder should be able to say "new project" through a conversational interface, answer product questions, and let the system coordinate planning, architecture, repository creation, ticket execution, PR review, and final delivery readiness.

The first useful product is a project-owning orchestration layer, not merely a coding helper. It should behave like a lightweight AI development team lead that can drive a project through separate specialist agents while preserving enough repository state, documentation, tickets, and review history for a human team to take over later.

## 2. Target Users

- Primary user: a non-engineer founder who wants to initiate and supervise software projects through conversation while the assistant manages the engineering workflow.
- Secondary user: LLM role agents acting as Business Planner, Architect, Executor, Reviewer, and Orchestrator.
- Tertiary user: a human engineer or technical reviewer who may inspect repository artifacts, tickets, pull requests, CI results, and review notes.

## 3. v0.1 Goal

Deliver an autonomous project orchestration workflow that can take a new software project from founder conversation to a GitHub repository containing a complete, reviewed, documented product implementation with a clear one-command deployment path for a user-owned VPS.

v0.1 is successful when the founder can start a project through Telegram or a lightweight web interface, answer product and architecture questions as they arise, receive periodic progress reports, and end with a GitHub repository that is ready for the founder to deploy manually.

## 4. Non-Goals

- Choose Hermes Agent, OpenClaw, a custom orchestration layer, or a hybrid approach during PRD creation.
- Support every possible Git host, deployment target, or CI provider.
- Provide a polished public SaaS product, marketplace, or multi-tenant hosted service.
- Automate secret creation, secret storage, or credential rotation in v0.1.
- Guarantee that generated implementation is correct without CI, review, and explicit user approval.
- Perform live production deployment to the founder's VPS without the founder's final action.
- Support arbitrary messaging platforms beyond Telegram and the selected web interface.
- Replace long-term human ownership of the resulting project after handoff.

## 5. User Journeys

### Journey 1: Founder Starts a Project

1. The founder sends a "new project" request through Telegram or the web interface.
2. The Orchestrator starts a guided project intake pipeline in Russian.
3. The Orchestrator asks the minimum useful founder questions before repository creation.
4. The Orchestrator creates a dedicated GitHub repository under the assistant's GitHub account.
5. The Orchestrator creates skeleton repository files for docs-as-code state, prompts, questions, PRD, architecture, tickets, CI, and review records.
6. Open questions are captured in repository files instead of being lost in chat.

### Journey 2: Business Planner Interviews the Founder

1. The Orchestrator opens or invokes the Business Planner role.
2. The Business Planner develops the product brief and asks product questions.
3. The Orchestrator forwards Business Planner questions to the founder in Russian.
4. The founder answers in natural language.
5. The Orchestrator returns the answers to the Business Planner and records durable decisions.
6. The Business Planner produces a PRD in English.

### Journey 3: Product Scope Becomes Architecture

1. The Architect reads the PRD, session state, questions, and repository rules.
2. The Architect asks clarifying architecture questions when product or technical tradeoffs require founder input.
3. The Orchestrator forwards those questions to the founder and returns the answers to the Architect.
4. The Architect evaluates required platform candidates, including Hermes Agent and OpenClaw, without assuming either is already selected.
5. The Architect writes a complete implementation plan, architecture specification, and ADRs when needed.
6. The architecture is decomposed into implementation tickets.

### Journey 4: Orchestrator Runs the Development Team

1. The Orchestrator prioritizes tickets and selects the next optimal development path.
2. Executor agents implement tickets in isolated branches and pull requests.
3. The Orchestrator monitors CI, PR status, automated review output, and Reviewer LLM verdicts.
4. The Orchestrator asks the founder only when product ambiguity, high-risk tradeoffs, credentials, or external access decisions require input.
5. The Orchestrator keeps project state current so humans can understand or take over the project later.

### Journey 5: Reviewer Validates a PR

1. Automated checks run in GitHub Actions.
2. Automated docs validation, tests, lint/typecheck, and configured static or security checks run where available.
3. `pr-agent` and a separate Reviewer LLM review the PR.
4. The Reviewer writes a verdict and findings against the ticket, architecture, ADRs, CI results, and repository rules.
5. The Orchestrator decides the next engineering action based on review results, including requesting fixes, re-running checks, or merging under the assistant's GitHub account when policy allows.

### Journey 6: Founder Receives Progress Reports

1. The Orchestrator sends concise progress reports in Russian after each completed ticket.
2. If work runs for a long time, the Orchestrator sends time-based updates every 30 to 60 minutes.
3. Each report summarizes completed work, current ticket or phase, blockers, decisions needed, and notable risks.
4. The Orchestrator avoids deep technical explanation unless the founder asks or a decision requires it.

### Journey 7: Founder Receives a Ready Project

1. The Orchestrator completes the planned tickets and review gates.
2. The repository contains product docs, architecture docs, implementation code, tests, CI, review records, and handoff notes.
3. The repository contains a documented one-command deployment path for the target VPS environment.
4. The founder manually runs the deployment command or gives separate explicit deployment approval outside v0.1 scope.
5. The project remains understandable and maintainable by future human engineers.

### Journey 8: Founder Understands Durable State

1. The Orchestrator explains high-level progress and blockers in Russian with enough detail for a non-engineer founder.
2. Durable artifacts remain in English for tool and agent compatibility.
3. The founder can inspect current phase, active tickets, active PRs, blockers, and pending decisions from repository files.

## 6. Functional Requirements

- The system must maintain durable project state in repository files, especially under `docs/orchestration/`, `docs/questions/`, `docs/prd/`, `docs/architecture/`, `docs/tickets/`, and `docs/reviews/`.
- The system must provide a founder-facing conversational interface through Telegram and a lightweight web interface.
- The system must support a "new project" flow that starts guided intake, creates a dedicated GitHub repository, and initializes skeleton project artifacts.
- The system must operate through the assistant's own GitHub account for project repositories.
- The system must support forwarding questions from specialist agents to the founder and returning founder answers to the originating agent.
- The system must allow high autonomy for routine engineering actions, including creating branches, opening PRs, running checks, requesting fixes, and selecting next tickets.
- The system must support role-separated workflows for Orchestrator, Business Planner, Architect, Executor, and Reviewer.
- The system must define and enforce role write zones through repository rules and prompts.
- The system must support a lightweight PRD artifact before architecture and ticket creation.
- The system must support an architecture specification and ADRs before production implementation begins.
- The system must support ticket-based implementation, where each meaningful implementation PR links to one approved ticket.
- The system must support PR-based delivery through GitHub.
- The system must support CI validation for docs, tests when code exists, lint/typecheck when configured, and static/security checks when configured.
- The system must support Reviewer LLM review as a required implementation PR gate.
- The system must support automated or orchestrator-approved merges inside assistant-owned repositories when configured policy allows.
- The system must reserve founder approval for product decisions, major scope changes, sensitive credential decisions, and final deployment action.
- The system must capture unresolved product, architecture, and process questions in `docs/questions/`.
- The system must keep user-facing conversation in Russian by default.
- The system must keep long-lived repository artifacts in English by default.
- The system must provide role prompts that explain required reading, allowed write zones, outputs, and completion criteria.
- The system must make the current project phase, blockers, active tickets, active PRs, and pending decisions visible through durable session state.
- The system must send progress reports after each completed ticket and at least every 30 to 60 minutes during long-running work.
- The system must produce handoff documentation that lets a future human team understand product intent, architecture, implementation status, deployment path, and known risks.
- The system must produce a one-command deployment path for a user-owned VPS, without requiring v0.1 to perform the final live deployment automatically.

The following requirements apply specifically to self-deployment of the assistant itself (Section 12), as distinct from deployment of generated projects:

- The system must provide a one-command install experience that takes a clean Ubuntu VPS from zero to a state where the assistant is installed, configured, and ready to start, after the Founder supplies a single environment file with the required values.
- The system must accept all self-deployment configuration (Telegram bot token, GitHub PAT for project repositories, OmniRoute API key, VPS-local paths) through that single environment file only, never through committed values, interactive prompts, repository artifacts, or command-line arguments.
- The system must provide a one-command health verification that the Founder can run after install and that returns a clear human-readable pass/fail summary together with a non-zero exit code on failure.
- The system must provide a rollback mechanism that restores the assistant to a prior known-good state without losing the operational data needed to resume project bindings, scheduled progress timers, and in-flight run metadata.
- The system must enforce explicit Founder approval before the assistant starts accepting Telegram messages for the first time after install; install alone must not auto-start the runtime.
- The system must enforce explicit Founder approval and a backup of the operational state store before any upgrade of the assistant.
- The system must keep all self-deployment secrets out of repository artifacts, logs, CI output, and PR artifacts; secrets must live only in the Founder's environment file and the runtime environment derived from it.
- The system must make the self-deployment install experience repeatable: re-running install on a previously installed VPS must not corrupt operational state or duplicate runtime processes.

## 7. Non-Functional Requirements

- Simplicity: v0.1 should prefer a lightweight docs-as-code process over complex automation that is hard to inspect.
- Traceability: product decisions, architecture decisions, implementation scope, reviews, and approvals should be linkable through repository artifacts.
- Reproducibility: a new agent session should be able to resume work by reading repository files instead of relying on previous chat context.
- Safety: role prompts and repository rules should reduce unauthorized writes, scope expansion, direct pushes to `main`, and unreviewed merges.
- Autonomy: the assistant should minimize founder interruptions during routine engineering work and ask only when product, security, external access, or high-risk tradeoff decisions require input.
- Explainability: orchestration updates should be concise and understandable to a non-engineer founder, with deeper detail available on request.
- Portability: the initial workflow should target GitHub and a user-owned VPS, while avoiding unnecessary lock-in before architecture evaluation.
- Minimalism: v0.1 should implement only the process and automation needed to prove reliable orchestration.
- Auditability: CI results, review verdicts, ticket status, and handoff notes should be inspectable after the fact.
- Handoff quality: generated repositories should be understandable by future human engineers and should not depend on hidden agent memory.
- Operational continuity: long-running orchestration should survive interruptions through durable state and resume from repository artifacts.

## 8. Security and Secrets Assumptions

- The system may need access to the assistant's GitHub account, created project repositories, LLM API keys, VPS deployment parameters, and service tokens.
- Secrets must never be committed to the repository.
- `.env` files, private keys, GitHub PATs, LLM API keys, VPS SSH keys, private repository credentials, and service tokens are security-sensitive.
- v0.1 should assume the assistant has broad autonomy inside its own GitHub account, but still uses least-privilege tokens where practical.
- v0.1 should prefer explicit user action for provisioning or rotating credentials.
- Repository artifacts may document which secret categories are needed, but must not include secret values.
- CI and automation should avoid printing secrets in logs.
- The architecture phase must decide the concrete secret handling mechanism for Telegram, the web interface, GitHub Actions, LLM providers, and VPS deployment configuration.
- Final deployment to a VPS should require a separate founder action or explicit deployment approval in v0.1.

## 9. Success Criteria

- A founder can start a new project from Telegram or a web interface.
- The Orchestrator can create a dedicated GitHub repository and initialize required project skeleton artifacts.
- The Orchestrator can route Business Planner and Architect questions to the founder and persist the answers as durable decisions.
- The Architect can use the PRD to produce an architecture spec, evaluate platform candidates, and create tickets.
- Executors can implement tickets without needing hidden chat state.
- Reviewers can validate PRs against tickets, architecture, ADRs, CI, and repository rules.
- The Orchestrator can autonomously sequence tickets, monitor checks and reviews, and continue work without asking for routine approvals.
- The founder receives useful progress reports after each ticket or every 30 to 60 minutes during long-running work.
- Every meaningful implementation change can be traced from PRD or architecture intent to ticket, PR, CI result, review verdict, and user merge approval.
- The process prevents or surfaces missing ticket links, missing review gates, out-of-zone writes, and undocumented deviations from the plan.
- The final project repository contains enough documentation, code, tests, CI, review history, and deployment instructions for a human team to take over.
- The final project repository supports a documented one-command deployment path to a user-owned VPS.
- The repository can pass baseline docs validation.

The following success criteria apply specifically to self-deployment of the assistant itself (Section 12):

- A Founder can deploy the assistant on a clean Ubuntu 22.04 LTS VPS with only the environment file and one command, and after explicit start approval the assistant responds to a Telegram message within 15 minutes of starting the install.
- A Founder can verify deployment health with one command and receive a clear pass/fail result; a passing result corresponds to an assistant that can actually accept Telegram input and dispatch a role session, not merely a process that is running.
- A Founder can roll back to the previous version of the assistant without losing the operational state needed to resume project bindings, scheduled progress timers, and in-flight Hermes run metadata.
- A Founder receives an explicit approval prompt before the assistant comes online for the first time after install, and a separate explicit approval prompt referencing a state-store backup before any upgrade.
- No secret value (Telegram bot token, GitHub PAT, OmniRoute API key, VPS credential, or other) appears in committed files, logs, PR artifacts, or review artifacts at any point in the self-deployment lifecycle.
- Re-running the install command on a VPS that already has the assistant installed leaves the assistant in a working state and does not duplicate runtime processes or corrupt the operational state store.

## 10. Open Product Questions

1. What should be the exact Telegram interaction model: bot commands, free-form chat, threaded project conversations, or a hybrid?
2. What should the lightweight web interface include in v0.1 beyond chat and project status?
3. Does the orchestrator need persistent storage beyond repository files for chat history, user identity, project registry, agent runs, and scheduled progress updates?
4. Which static and security tools should be enabled first beyond docs validation?
5. What merge policy should apply inside assistant-owned GitHub repositories after CI and review gates pass?
6. What exact one-command deployment contract should generated projects expose for a user-owned VPS?
7. Should v0.1 support multiple simultaneous project repositories or one active project at a time?
8. What is the expected trust boundary between Telegram/web sessions, local or hosted agent runners, GitHub Actions, LLM providers, and the user-owned VPS?
9. Which actions still require explicit founder approval despite high autonomy: spending money, installing paid services, changing product scope, using external APIs, or exposing public endpoints?

The following open product questions apply specifically to self-deployment of the assistant itself (Section 12):

10. Should the v0.1 install experience support non-Ubuntu targets (Debian, RHEL family, other Linux distributions), or remain Ubuntu-only? Recommended default: Ubuntu 22.04 LTS only in v0.1; expand later only if a concrete Founder need appears.
11. Is 15 minutes an acceptable upper bound for "clean VPS to Telegram-reachable assistant," or should the target be faster (and worth giving up some pre-flight validation depth to achieve)? Recommended default: 15 minutes is acceptable; pre-flight validation depth wins over raw install speed.
12. Should the health-verification command verify LLM dispatch end-to-end (for example, actually send a prompt to an Executor model and read the response back), or remain a connectivity-only check (Telegram reachable, GitHub PAT valid, OmniRoute reachable, state store writable, runtime process alive, project registry queryable)? Recommended default: connectivity-only for the v0.1 health check; a separate, slower full-smoke command may verify LLM dispatch on demand.
13. Should the assistant auto-start on VPS reboot, or always require an explicit manual Founder action to come online again? Recommended default: manual start only in v0.1; auto-start on reboot can be reconsidered after the first stable trial because it weakens the explicit-approval gate.
14. Is rollback a hard v0.1 requirement, or is "clean install from scratch using repository artifacts as the source of truth" an acceptable fallback? Recommended default: rollback is a v0.1 requirement because the operational state store contains data not reproducible from repository artifacts (active project bindings, in-flight run metadata, scheduled progress timers).
15. What happens when the Founder must rotate a secret value (replace the Telegram bot token, replace the GitHub PAT for project repositories, replace the OmniRoute API key)? Should the install/upgrade flow include a documented rotate-and-restart path, or is "edit environment file, restart runtime" sufficient as an out-of-band Founder action? Recommended default: document the rotate-and-restart path explicitly; do not automate rotation in v0.1.

## 11. Handoff Notes for the Architect

- Evaluate Hermes Agent and OpenClaw as required platform candidates, but do not treat either as pre-selected.
- Treat Telegram plus a lightweight web interface as required founder-facing surfaces for v0.1.
- Define whether the runtime should be a service, daemon, worker queue, GitHub workflow layer, local runner, or hybrid.
- Decide whether repository files alone are sufficient durable state or whether additional storage is needed for conversations, project registry, scheduled updates, credentials metadata, and agent run state.
- Define the minimum architecture for GitHub account integration, repository creation, branch/PR operations, CI observation, review orchestration, and autonomous merge policy.
- Define how role prompts, write-zone rules, docs validation, and ticket state transitions are enforced.
- Define the initial security model for Telegram, the web interface, GitHub credentials, LLM API keys, VPS deployment configuration, CI secrets, and local development secrets.
- Define the generated-project contract for handoff quality and one-command VPS deployment readiness.
- Keep the first architecture small enough to prove autonomous project orchestration without overbuilding a general multi-tenant SaaS platform.

The following handoff notes apply specifically to self-deployment of the assistant itself (Section 12):

Sequencing constraint: this PRD revision -> Architect pass -> TKT-011 trial.
- The Architect must produce, at minimum: an architecture section (or new contract document such as `docs/architecture/SELF-DEPLOYMENT-CONTRACT.md`) that specifies the install entry point, environment-variable contract, pre-flight validation, health-check checklist, rollback mechanism, upgrade procedure, and Founder-approval gates; a new ADR recording the deployment-mechanism choice (container vs systemd vs bare-metal install, secrets storage, logging destination, operational state store backend); and at least three implementation tickets covering install, health check, and rollback/upgrade with state-store backup.
- Refer to `docs/backlog/TKT-NEW-self-deployment-architect-pass.md` for the full gap analysis that triggered this PRD revision and for the cross-repo audit (`docs/session-log/2026-05-05-session-1.md`) that produced the directive.
- The Architect must keep the product/implementation boundary intact: this PRD commits only to the Founder experience (one-command install, one-command health check, one-command or clearly documented rollback, three approval gates, env-file-only secrets, repeatable install). ## 12. Self-Deployment Operational Target

This section describes, in product language, the operational experience of deploying `developer-assistant` itself onto the Founder's VPS. It is distinct from the generated-project deployment story addressed by Sections 6 and 9 and by `docs/architecture/GENERATED-PROJECT-DEPLOYMENT-CONTRACT.md`: that story covers products the assistant builds for its users, while this section covers how the assistant itself becomes reachable in the first place. Implementation details (command names, container choice, file paths, mode bits, the exact health-check invariant list) are intentionally left to the Architect; this section commits only to the Founder-visible experience.

### 12.1 What "the assistant is deployed" means

From the Founder's perspective, the assistant is deployed when:

- The Founder can send a Telegram message to the assistant's bot identity and receive a coherent response routed through the Hermes runtime.
- The Founder can use the same Telegram conversation to start a new project (`/new_project` or its equivalent), answer intake questions, and watch the orchestrator pick up the work end-to-end as described in the v0.1 user journeys (Section 5).
- The assistant is registered against the Founder's GitHub identity for project repositories and against the Founder's OmniRoute LLM access, with no further interactive credential setup required during normal operation.
- The assistant's runtime is the kind of thing the Founder can stop and start on the VPS in a controlled manner, not a one-shot script that disappears after install.

The end state is a Telegram-reachable assistant capable of beginning and running real project work, not a partially-installed runtime that still requires manual wiring before it can answer a message.

### 12.2 What the Founder does to reach that end state

The Founder reaches the deployed end state through a one-command install experience on a clean Ubuntu VPS:

1. The Founder provisions a clean Ubuntu VPS, opens an SSH session, and clones the `developer-assistant` repository onto it.
2. The Founder fills in a single environment file (referred to here as `SELF-DEPLOY.env`) with the values the install needs: a Telegram bot token, a GitHub personal access token for project repositories, an OmniRoute API key for LLM access, and the VPS-local paths the install needs (operational state store path, log directory, secrets directory, repo workspace). The environment file is the only place the Founder writes secret values; secrets do not appear anywhere in the repository.
3. The Founder runs one command. From the Founder's view, that command takes the VPS from "no assistant present" to "assistant installed, configured, and ready to start" without further interactive prompts beyond what is explicitly required for security confirmation.

The PRD does not commit to the command name, the script language, or the underlying packaging mechanism (Docker, Docker Compose, systemd, bare-metal install, etc.). The Architect chooses these and records the choice in an ADR.

The install experience must be repeatable: re-running the same command on the same VPS with the same environment file must not corrupt the assistant, duplicate runtime processes, or destroy the operational state store.

### 12.3 What the Founder does to verify it worked

The Founder verifies the install with a single health-verification command. From the Founder's view, that command:

- Returns a clear, human-readable pass/fail summary that names each thing it checked and the outcome of that check.
- Returns a non-zero exit code on failure so simple wrappers can detect "is the assistant healthy" without parsing prose.
- Is safe to re-run any number of times; it does not change runtime state, mutate the operational state store, or alter the Founder's data.

The PRD does not commit to the command name or to a fixed list of checks. The Architect chooses both. The product requirement is that the Founder, after running the verification once, can say "the assistant is healthy" or "the assistant has problem X" and act on the answer without asking an engineer to interpret the output.

### 12.4 What happens when something breaks

If a deploy or upgrade leaves the assistant in a broken state, the Founder can return the assistant to a previously known-good state through a documented rollback experience. From the Founder's view:

- Rollback is initiated by a single command or a clearly documented short sequence; it does not require manual database surgery, hand-edited config, or out-of-band engineer assistance.
- Rollback restores the assistant to the previous deployed version together with the operational state required to resume project bindings, scheduled progress timers, and in-flight Hermes run metadata. Rollback does not lose operational data the assistant needs to resume work.
- Rollback never deletes the Founder's repository artifacts; repository state remains the authoritative record of product, architecture, ticket, review, and decision history.
- If rollback itself fails, the Founder receives a clear error and a recommended next step (typically: stop the runtime, restore from backup, contact a human engineer); the Founder is not left guessing.

The PRD does not commit to a specific rollback mechanism (versioned pins, state-store snapshots, container image swap, symlink retarget, etc.). The Architect chooses. The product requirement is that "I can get back to last week's working state" is true, and that "I can get back" never silently destroys the Founder's projects, decisions, or operational data.

### 12.5 Approval gates

Self-deployment has three product-level approval gates that exist because the assistant has access to the Founder's GitHub identity, Telegram bot identity, and LLM provider; bringing it online or upgrading it has security and trust consequences that must not happen as a side effect of running an install command:

- **Install** runs without explicit Founder approval. Install brings the assistant onto the VPS in a stopped or idle state. Install does NOT auto-start the assistant; the assistant does not begin accepting Telegram messages or dispatching role sessions simply because the install completed.
- **Start** transitions the assistant from installed-and-idle to actively accepting Telegram messages and dispatching work. The first time the assistant comes online after install requires an explicit Founder approval action; it is a deliberate Founder decision, not a side effect of the install command.
- **Upgrade** replaces the running assistant with a new version. Upgrade requires both an explicit Founder approval action AND a backup of the operational state store taken before the upgrade begins. Upgrade without a state-store backup is not allowed.

The same gates apply whether the install/start/upgrade is initiated by the Founder directly or proposed by the assistant on the Founder's behalf through Telegram.

### 12.6 Secrets handling

All secrets needed for self-deployment are provided by the Founder through the environment file described in Section 12.2 and remain on the VPS:

- Secrets are read only from the environment file. The install does not embed them in the repository, in committed config, in image tags, in PR descriptions, in command-line arguments, in shell history, or in any artifact derived from the repository.
- Secrets never appear in logs, in PR artifacts, in CI output, in screenshots, in Reviewer artifacts, or in any output the Founder might paste somewhere shareable.
- The environment file lives outside the repository's git tree and is mode-restricted so that only the Founder's VPS user can read it.
- The repository may document which secret names must exist (so an engineer reading the code knows what is required) but never contains secret values.

The PRD does not commit to a specific file mode, path convention, or secret-handling library; the Architect chooses these and records the choice in an ADR. The product requirement is that no secret value, at any point in the self-deployment lifecycle, leaves the Founder's VPS.
